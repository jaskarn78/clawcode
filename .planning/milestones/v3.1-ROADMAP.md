# Roadmap: ClawCode v3.1 — Multi-LLM Runtime + Subscription-Pool Fallback

**Theme:** Decouple ClawCode from the Anthropic Agent SDK subscription pool by introducing a provider-neutral `LlmRuntimeService` seam. Replicate the Phase 117 advisor-backend pattern (per-agent config-flippable rollback) for the **primary** runtime. Retain full ClawCode functionality (per-agent SQLite memory, Phase 117 advisor, knowledge graph, Discord routing, MCP tool surface, prompt caching) while supporting Anthropic API key, OpenAI Codex, OpenRouter, and (probe-gated) interactive Claude Code CLI as alternate backends.

**Status:** PROPOSED 2026-05-15. Driver: Anthropic's 2026-05-14 announcement splitting Agent SDK programmatic usage off the interactive subscription rate-limit pool effective **2026-06-15**, with a flat $200/month credit cap for Max 20x. At ClawCode's 7-agent footprint, the $200 cap depletes in days under realistic burn rate.

**Constraint:** Zero new npm deps for the seam itself. Per-backend deps are isolated and conditional-load (only the chosen backend's deps initialize at boot).

**Deploy gate inherited from v2.9/v3.0:** `feedback_ramy_active_no_deploy` continues to apply. Multi-LLM cutover deploys are high-risk; gate explicitly.

**Hard deadline track:** Waves 1-3 (seam + AnthropicApiKey backend + credit telemetry + failover) MUST land before **2026-06-15** for ClawCode to keep running affordably on the operator's current subscription.

## Bundles

### 🏗️ Bundle A — Seam + safety valve (hard-deadline track, must ship by 2026-06-15)

Three phases, sequenced. Operator must be able to flip ANY agent to direct API billing before the Agent SDK credit policy enforces.

| Phase | Wave-from-999.62 | Title | Why this order |
|-------|-------|-------|------|
| **130** | Wave 1 | LlmRuntimeService seam + AnthropicAgentSdk backend extraction (no behavior change) | Mirrors Phase 117's "scaffold first, no behavior change" approach. Zero-risk landing for the seam |
| **131** | Wave 2 | AnthropicApiKey backend + per-agent `llmRuntime.backend` selection in clawcode.yaml + ConfigWatcher hot-reload | Cheapest path to the safety valve — operator can flip an agent to direct API billing immediately |
| **132** | Wave 3 | Credit telemetry (`agent-sdk-credit: $X / $200, projected runout YYYY-MM-DD`) + Discord 75/90/99% alerts + auto-failover when credit hits threshold (default 95%) | Operational visibility before deeper engineering. Failover triggered by simulated depletion in integration test |

**Success criteria (Bundle A — minimum viable v3.1 closure):**
1. Seam compiles + existing test suite passes with `anthropic-agent-sdk` as default (zero behavior change).
2. Per-agent `llmRuntime.backend: anthropic-api-key` override hot-reloads via existing ConfigWatcher.
3. `research` agent can be flipped from `anthropic-agent-sdk` → `anthropic-api-key` via config-edit-and-reload, verified by inspecting outbound auth on next turn.
4. `clawcode status` shows `agent-sdk-credit: $123.45 / $200.00 (61.7%, projected runout 2026-06-22)`.
5. Failover triggers at 95% — simulated-depletion integration test green.

## Cross-cutting requirements (apply to all phases)

1. **AdvisorService compatibility** — Phase 117's `AdvisorService` seam (`src/advisor/`) must remain functional for every backend. If the chosen backend lacks an in-request advisor mechanism (e.g., Codex), fall back to the `fork` advisor backend pattern (spawn a separate Opus session for advisor calls).
2. **MCP tool surface compatibility** — the existing `clawcode__*` tools (`spawn_subagent_thread`, `post_to_agent`, `ask_advisor`, `memory_lookup`) must work across all backends. Tool-call translation layer (Phase 135) is the integration point.
3. **Prompt caching equivalent** — Phase 115 prompt cache design is Anthropic-specific (`cache_control` blocks). Each non-Anthropic backend needs a documented caching equivalent OR an explicit note that caching is unavailable on that backend (impacts cost projections).
4. **Cost tracking parity** — `clawcode usage` and dashboard must surface correct cost/token telemetry per-backend. Anthropic API key vs Agent SDK credit vs Codex vs OpenRouter all use different billing units.
5. **Per-agent `llmRuntime.backend` cascade per Phase 117 pattern** — every backend selectable per-agent with `defaults.llmRuntime` baseline.

## Out of scope (explicit)

- **Header spoofing** (Path D from 999.62 research) — PERMANENTLY off the table. Do not propose.
- **Browser automation against claude.ai** (Path C from 999.62 research) — PERMANENTLY off the table. Latency catastrophe + Cloudflare detection + same TOS-gray surface as tmux.
- **Multi-account credit stacking** — TOS-gray, ban risk on operator's primary subscription. Out of scope.
- **Local model backends (Ollama, llama.cpp)** — not on this milestone. Tool-call reliability on local models is the weakest in the ecosystem; revisit when small-model tool-use reaches parity. v3.x dot-release candidate if a strong driver emerges.
- **Voice channel work** — v3.0 owns (999.56). Independent surface.
- **Skill creation autonomy** — v3.0 owns (999.59). Independent.

## Risk register

| Risk | Severity | Mitigation |
|------|----------|------------|
| Concurrent-session cap < 7 makes Phase 134 unviable | HIGH | Phase 133 probe gates Phase 134. If probe fails, Codex/OpenRouter become primary fallback (Bundle C promoted). |
| Tool-use translation drops accuracy on non-Anthropic backends | HIGH | Phase 135 includes per-backend tool-use compatibility test suite. Per-agent backend selection means high-stakes agents (`fin-acquisition`) stay on Anthropic until parity proven. |
| Anthropic enforces volume detection on interactive-pool driving | MEDIUM | Treat Phase 134 as defensible fallback, not foundation. Bundle A (API key + failover) is the structural safety net. Document residual risk per 999.62 BACKLOG §"Residual risk" line. |
| Credit telemetry inaccurate (Anthropic billing API rate-limits or returns stale data) | LOW-MEDIUM | Local approximation from `clawcode usage` token counts × current published rates is the fallback. Discord alerts at 75/90/99% give 25% headroom on any single miss. |
| Per-backend prompt caching divergence causes hidden cost regression | MEDIUM | Phase 135/136 each document caching mechanism (or lack thereof). Cost-tracking parity (cross-cutting #4) catches it in dashboard. |
| Discord rate-limit broker (Phase 999.X) saturates differently per-backend due to different turn latency | LOW | Existing rate-limiter is per-token, not per-backend. Should be backend-agnostic. Verify in Phase 130 acceptance criteria. |
