<objective>
Close the single warning-level verification gap on RETR-03: the `retrievalHalfLifeDays` knob defined in `conversationConfigSchema` (min 1, default 14) is currently INERT at runtime because nothing reads it from the resolved agent config. `searchByScope` always falls through to `DEFAULT_RETRIEVAL_HALF_LIFE_DAYS=14` — the default is correct, but agents that set a custom value in their clawcode.yaml (e.g. `retrievalHalfLifeDays: 7` for aggressive recency) see no effect.

The fix is surgical and threads the value through two seams that already exist:

1. `invokeMemoryLookup` must accept `retrievalHalfLifeDays` on `MemoryLookupParams` and forward it to `searchByScope` as `ScopedSearchOptions.halfLifeDays`. `searchByScope` already reads `options.halfLifeDays` and falls back to `DEFAULT_RETRIEVAL_HALF_LIFE_DAYS` when undefined (see `conversation-search.ts:85-87`) — no changes there.
2. The daemon `memory-lookup` IPC switch case must call `manager.getAgentConfig(agentName)?.memory.conversation?.retrievalHalfLifeDays` and include that value in the params it passes to `invokeMemoryLookup`.
3. The `ResolvedAgentConfig.memory.conversation` type in `src/shared/types.ts` must be extended with `retrievalHalfLifeDays: number` so the compiler enforces the new field throughout the resolver chain. Zod already parses this field via `conversationConfigSchema.retrievalHalfLifeDays` (schema.ts line 79), but the readonly type mirror is missing it today (see types.ts:46-55).

Non-goals (DO NOT do these):
- DO NOT refactor `searchByScope` internals. Its `halfLifeDays` parameter is already wired.
- DO NOT rename `DEFAULT_RETRIEVAL_HALF_LIFE_DAYS`. Leave the constant and fallback path as-is.
- DO NOT touch the MCP tool schema — `retrievalHalfLifeDays` is agent-config-level (resolved once per agent), not a per-request MCP parameter.
- DO NOT change the legacy GraphSearch branch (scope='memories' && page=0). Decay weighting is a conversation-search concern; the legacy branch uses GraphSearch's own scoring and is out of scope for RETR-03.
- DO NOT modify pre-existing Phase 68 tests — only ADD a new test in `daemon-memory-lookup.test.ts`.

Purpose: RETR-03 config knob goes from inert to live. Agents configured with `retrievalHalfLifeDays: 7` will see 10-day-old conversation results decay roughly twice as fast as agents on the default 14-day schedule.

Output:
- Extended `ResolvedAgentConfig.memory.conversation` type with `retrievalHalfLifeDays: number` (readonly)
- `MemoryLookupParams.retrievalHalfLifeDays?: number` + passthrough to `searchByScope` options
- Daemon `memory-lookup` case reading the resolved agent config and threading the value
- 1 new integration test in `daemon-memory-lookup.test.ts` proving the runtime effect with two differing half-life values over identical decayed fixtures
</objective>
