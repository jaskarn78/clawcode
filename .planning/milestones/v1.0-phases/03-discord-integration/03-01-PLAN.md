  <behavior>

    - requestPermit on a fresh limiter returns allowed=true immediately
    - requestPermit 51 times in rapid succession: first 50 allowed, 51st returns allowed=false with retryAfterMs > 0 (per D-10: 50 req/s global)
    - requestPermit on same channel 6 times rapidly: first 5 allowed, 6th returns allowed=false (per D-13: 5 msg/5s per channel)
    - After waiting for refill, previously exhausted bucket allows requests again
    - enqueue adds message to queue; dequeueNext returns oldest message (FIFO)
    - Queue respects maxQueueDepth (default 100 per channel per D-12); overflow drops oldest
    - getStats returns current token counts and queue depths
  </behavior>
  <action>
    1. Add to `src/discord/types.ts` (extend the existing file):
       - `TokenBucketConfig`: `{ readonly capacity: number; readonly refillRate: number }`
       - `RateLimiterConfig`: `{ readonly global: TokenBucketConfig; readonly perChannel: TokenBucketConfig; readonly maxQueueDepth: number }`
       - `RateLimitPermit`: `{ readonly allowed: boolean; readonly retryAfterMs: number }`
       - `QueuedMessage`: `{ readonly channelId: string; readonly content: string; readonly enqueuedAt: number }`
       - `RateLimiterStats`: `{ readonly globalTokens: number; readonly channelTokens: ReadonlyMap<string, number>; readonly queueDepths: ReadonlyMap<string, number> }`
       - `DEFAULT_RATE_LIMITER_CONFIG` constant: global capacity=50 refillRate=50, perChannel capacity=5 refillRate=1, maxQueueDepth=100

    2. Create `src/discord/__tests__/rate-limiter.test.ts` with tests for all behaviors. Use a clock injection pattern (pass `now?: () => number` to functions) so tests can control time without real timers.

    3. Create `src/discord/rate-limiter.ts`:
       - Internal `TokenBucket` type: `{ tokens: number; lastRefillAt: number }`
       - `refillBucket(bucket, config, now)`: pure function returning new bucket with refilled tokens based on elapsed time (sliding window per Claude's discretion)
       - `tryConsume(bucket, config, now)`: pure function returning `{ allowed, bucket, retryAfterMs }`
       - `createRateLimiter(config?: RateLimiterConfig)`: returns a `RateLimiter` object (closure-based, NOT class) with methods:
         - `requestPermit(channelId: string): RateLimitPermit` -- checks global bucket first, then per-channel bucket. Both must allow.
         - `enqueue(channelId: string, content: string): boolean` -- adds to per-channel FIFO queue. Returns false if queue full (drops oldest first per Pitfall 5). Logs warning on overflow.
         - `dequeueNext(channelId: string): QueuedMessage | undefined` -- pops oldest from channel queue
         - `getStats(): RateLimiterStats` -- returns current state for IPC introspection

    Use immutable patterns for bucket updates (return new objects, never mutate). The limiter itself holds mutable state internally (Map of buckets) since it is a stateful service, but individual bucket operations are pure.
  </action>
