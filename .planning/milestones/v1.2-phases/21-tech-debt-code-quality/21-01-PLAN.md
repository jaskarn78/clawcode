  <action>
    Fix all silent catch blocks identified in the tech debt audit. Each catch must either log the error with context or have an explicit documented reason for swallowing.

    **src/discord/bridge.ts:**
    - Line ~309: `try { await messageRef.current.delete(); } catch { /* ignore */ }` — This is a "best effort delete" of a typing indicator message. Replace with: `try { await messageRef.current.delete(); } catch (err) { this.log.debug({ error: (err as Error).message }, "failed to delete typing indicator message"); }` — Use debug level since this is non-critical.
    - Line ~341: `} catch { // Ignore reaction failure }` — Replace with: `} catch (err) { this.log.debug({ error: (err as Error).message }, "failed to add error reaction"); }` — Debug level, non-critical reaction.
    - Line ~372: `} catch { this.log.warn(...) }` — This one already logs, leave as-is.
    - Line ~61: `} catch { // Fall through to env var check }` — This is in token reading, a deliberate fallback. Add: `} catch { // Config file not found or unreadable — fall through to env var }` — Explicit reason documented.

    **src/heartbeat/checks/inbox.ts:**
    - Line ~51: `} catch { failed++; // Don't mark processed on failure }` — Replace with: `} catch (err) { failed++; context.sessionManager.getLog?.().warn?.({ agent: agentName, messageId: msg.id, error: (err as Error).message }, "inbox message delivery failed, will retry next heartbeat"); }` — Actually, the check module doesn't have direct logger access. Instead, since the function already tracks `failed` count and returns a warning status with the count, this is explicitly handled (not silent). The comment explains the behavior. Make the comment more explicit: `} catch { failed++; // Delivery failed — don't mark processed so it retries next heartbeat cycle }`

    **src/heartbeat/checks/thread-idle.ts:**
    - Line ~66: `} catch { // Log but continue }` — Read the actual code at this line. If the catch is truly empty, add logging. If it has a comment explaining the behavior, make it more explicit. The pattern should be: `} catch (err) { /* thread cleanup is best-effort — continue checking other threads */ }`

    **src/manager/daemon.ts:**
    - Line ~298: `await unlink(SOCKET_PATH).catch(() => {});` — These are cleanup-on-shutdown calls. Replace with: `await unlink(SOCKET_PATH).catch((err) => { log.debug({ path: SOCKET_PATH, error: (err as Error).message }, "socket file cleanup failed (may not exist)"); });`
    - Line ~299: `await unlink(PID_PATH).catch(() => {});` — Same pattern: `await unlink(PID_PATH).catch((err) => { log.debug({ path: PID_PATH, error: (err as Error).message }, "pid file cleanup failed (may not exist)"); });`
    - Check if `daemon.ts` already imports a logger. If not, add: `import { logger as log } from "../shared/logger.js";` or use an existing logger reference.

    **Guiding principle:** Use `debug` level for truly non-critical best-effort operations (delete typing indicator, remove socket on shutdown). Use `warn` for operations where failure might indicate a real problem. Never leave a catch completely empty — at minimum, add a comment explaining why the error is intentionally swallowed.
  </action>
