# Quick 260511-pw3 — Schema registry auto-discovery

## Context

Observed 2026-05-11 in Discord exchange between Admin Clawdy and Projects agent. Admin Clawdy tried `delegate_task` with schema `bug.report`. Projects' schema registry rejected it as `unknown_schema`. So no typed task ever landed.

Admin Clawdy flagged: *"need to register `bug.report` (or publish valid schema names) so cross-agent typed delegations actually work."*

The current design treats schemas as a closed allowlist that each agent maintains separately. Senders have no way to discover what's accepted. The natural fallback is `post_to_agent` (fire-and-forget channel post) — which has its own silent-drop bug (see quick 260511-pw2).

So: typed cross-agent delegation is functionally broken without out-of-band schema coordination. Two failure modes compound.

## Design options (pick one based on existing schema-registry architecture)

**Option A — Schema introspection tool:** Add new MCP tool `list_agent_schemas(targetAgent: string)` that returns the recipient's accepted schemas. Sender's LLM can introspect before calling `delegate_task`. Minimal change to existing registry semantics.

**Option B — Auto-register on first use:** When `delegate_task` receives an unknown schema, auto-register it as a generic JSON-typed task in the recipient's queue. Skips schema validation but unblocks the delivery. Aligns with the "post_to_agent is just a channel post" semantic.

**Option C — Schema validation error returns accepted set:** When `delegate_task` rejects `unknown_schema`, the error response includes the recipient's accepted schemas list. Senders can retry or fall back. Minimal API change, fixes the immediate symptom.

**Recommended: A + C combined.** A provides programmatic discovery; C is the helpful-error safety net. Both are small.
