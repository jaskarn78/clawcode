Refactor SdkSessionAdapter and wrapSdkSession to use the query() API with streamInput() for multi-turn conversations. The SessionHandle interface and MockSessionAdapter/MockSessionHandle are UNCHANGED.

**Architecture for multi-turn with query():**

The query() API works differently from the old session model:
- Old: `session.send(msg)` then iterate `session.stream()` — session persists across turns.
- New: `query({ prompt, options })` returns a Query (AsyncGenerator). For multi-turn, pass an `AsyncIterable<SDKUserMessage>` as `prompt`, then call `query.streamInput()` to feed subsequent messages. The generator keeps yielding messages across turns.

**SdkSessionAdapter.createSession() refactor:**

1. Call `loadSdk()` and `transformMcpServersForSdk(config.mcpServers)` as before.
2. Create a prompt channel: an async iterable that the adapter controls. Use a simple pattern with a resolve queue:
   ```typescript
   // Channel for feeding user messages to the query
   type MessageChannel = {
     iterable: AsyncIterable<SdkUserMessage>;
     push: (msg: SdkUserMessage) => void;
     end: () => void;
   };
   function createMessageChannel(): MessageChannel { ... }
   ```
   Implementation: Use an array as buffer + a pending promise resolve pattern (like a simple async queue). When `push()` is called, either resolve a waiting consumer or buffer the message. When `end()` is called, signal completion.

3. Create the initial prompt channel and send the first message (use config.systemPrompt or a minimal init message like "You are starting a new session." — the systemPrompt is passed via options, so the initial prompt can be a brief activation message).

4. Call `sdk.query({ prompt: channel.iterable, options: { cwd: config.workspace, model: config.model, systemPrompt: config.systemPrompt, permissionMode: "bypassPermissions", ...mcpServers ? { mcpServers } : {}, settingSources: ["project"] } })`.

5. Start a background async loop that drains the query generator, processing messages (extracting text, usage). Store this as part of the wrapper state.

6. Return a `wrapSdkQuery(query, channel, usageCallback)` handle.

**wrapSdkQuery() implementation:**

Replace `wrapSdkSession()` with `wrapSdkQuery(query: SdkQuery, sdk: SdkModule, options: SdkQueryOptions, usageCallback?: UsageCallback)`.

Internal state:
- `currentQuery: SdkQuery` — the active query generator
- `sessionId: string` — extracted from first result message's `session_id` field, or generated as `pending-${Date.now()}`
- `errorHandlers: Array<(error: Error) => void>`
- `endHandlers: Array<() => void>`

For `send(message)`:
1. Create a new single-message async iterable yielding `{ type: 'user', content: message }`.
2. Call `currentQuery.streamInput(iterable)`.
3. Drain the generator until the next `result` message, extracting usage.

For `sendAndCollect(message)`:
1. Same as send but collect assistant text blocks and return them.
2. Create new single-message iterable, call `streamInput()`.
3. Iterate generator collecting assistant messages until result. Return collected text.

For `sendAndStream(message, onChunk)`:
1. Same as sendAndCollect but call `onChunk` with accumulated text as each assistant message arrives.

For `close()`:
1. Call `query.close()`.

For `onError(handler)` / `onEnd(handler)`:
1. Store handlers. Invoke error handlers if the generator throws during iteration.
2. Invoke end handlers when the generator completes (returns).

**Key insight for multi-turn iteration:**

The query generator keeps yielding across multiple streamInput() calls. The background drain loop processes ALL messages. The send/sendAndCollect/sendAndStream methods:
1. Feed a message via streamInput()
2. Wait for the current turn to complete (next `result` message from the generator)
3. Return the collected response

Use a turn-tracking mechanism: each send creates a Promise that resolves when the next `result` message arrives from the background drain loop.

**Simpler alternative (recommended):** Instead of a persistent background loop, use a per-turn pattern:
- Each send/sendAndCollect/sendAndStream creates a new query() call (NOT streamInput). This is simpler and matches how the old session.send() + stream() worked.
- For session continuity, pass `resume: sessionId` in options for subsequent calls.
- Extract sessionId from the first result message's `session_id` field.

This per-turn-query approach is simpler and avoids complex async coordination:

```typescript
// Per-turn: each message creates a fresh query with resume
async sendAndCollect(message: string): Promise<string> {
  const opts = sessionId ? { ...baseOptions, resume: sessionId } : baseOptions;
  const q = sdk.query({ prompt: message, options: opts });
  const textParts: string[] = [];
  for await (const msg of q) {
    if (msg.type === "assistant" && typeof msg.content === "string" && msg.content.length > 0) {
      textParts.push(msg.content);
    }
    if (msg.type === "result") {
      if (msg.session_id) sessionId = msg.session_id;
      extractUsage(msg, usageCallback);
      if ("result" in msg && typeof msg.result === "string" && msg.result.length > 0) return msg.result;
      if (msg.subtype !== "success" && msg.is_error) throw new Error(`Agent error: ${msg.subtype}`);
      break;
    }
  }
  return textParts.join("\n");
}
```
