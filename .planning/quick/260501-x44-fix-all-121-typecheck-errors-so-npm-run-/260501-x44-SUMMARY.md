## Schema-vs-resolved-type drift (root cause for several errors)

Two production-side type definitions had drifted from their zod schemas:

1. **`effort`**: `effortSchema` (config/schema.ts) lists 7 values (`low | medium | high | xhigh | max | auto | off`). `ResolvedAgentConfig.effort` and `AgentSessionConfig.effort` were stuck at 4 values. The `narrowEffortForSdkOption` helper in session-adapter.ts already handled the projection — the resolved types just hadn't been widened.

2. **`tiers.centralityPromoteThreshold`**: `tierConfigSchema` (memory/schema.ts) has the field with a default of 5; `DEFAULT_TIER_CONFIG` (memory/tiers.ts) has it; `TierConfig` (memory/tiers.ts) requires it; the schema even has an inline comment explaining why it was added. `ResolvedAgentConfig.memory.tiers` (shared/types.ts) was missing it.

Both drifts were widening fixes (consumer types caught up to producer schemas).

### 1. `src/usage/budget.ts:138` — TS2367 (FLAGGED, NOT FIXED)

**Quoted code (line 121-144):**

```ts
checkAlerts(
  agent: string,
  model: string,
  budgetConfig: AgentBudgetConfig,
): "warning" | "exceeded" | null {
  let highest: "warning" | "exceeded" | null = null;

  for (const periodType of ["daily", "weekly"] as const) {
    const limit = this.getLimit(budgetConfig, periodType, model);
    if (limit === undefined) continue;

    const usage = this.getUsageForPeriod(agent, model, periodType);
    const ratio = usage / limit;

    if (ratio >= 1.0) {
      return "exceeded"; // Can't get higher than exceeded
    }
    if (ratio >= 0.8 && highest !== "exceeded") {
      highest = "warning";
    }
  }

  return highest;
}
```

**TS2367 message (verbatim):**

> `This comparison appears to be unintentional because the types '"warning" | null' and '"exceeded"' have no overlap.`

**Why this looks like a real bug, not a type-fallout:**

- `highest` is declared `"warning" | "exceeded" | null` — three states.
- The loop's only assignment to `highest` (line 139) sets it to `"warning"`, never `"exceeded"`.
- The line 135 `return "exceeded"` short-circuits before any `"exceeded"` could be written to `highest`.
- Therefore at line 138, `highest` is provably `"warning" | null` (TS is correct), and the `highest !== "exceeded"` guard is **always true**.
- That guard is dead code. The question is: **did the original author intend to set `highest = "exceeded"` somewhere and forget?**

**Hypothesis 1 (lost union member):** Earlier code had a path that set `highest = "exceeded"` (e.g., a per-period exceeded-but-not-yet-returning state for aggregation across periods), and a refactor removed it but left the guard. Today the dead guard is harmless — but the function reports "warning" + "exceeded" together as `null` if no period is at warning level — which feels suspicious for a budget alerting function that should be max-of-periods.

**Hypothesis 2 (typo in original variable):** The comparison may have been intended against a different variable (e.g., a per-period `state` local) that was later renamed/inlined. The dead-guard is the residue.

**Recommendation for /ultrareview:** Pull this thread. Write a unit test for `checkAlerts` covering the case where one period is at 0.85 (warning) and another is at 1.05 (exceeded) **while the first period is iterated first**. The current code returns `"exceeded"` (correct via line 135 short-circuit). But re-order the periods, or imagine a 3-period future, and the dead-guard's intent becomes important: was it meant to prevent `"warning"` from clobbering `"exceeded"` in a non-short-circuiting variant?

The plan was correct to quarantine this — it's not a type-upgrade fallout, it's a refactor scar.
