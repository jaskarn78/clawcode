## Audit Findings (Task 3)

Per the plan's audit-first directive:

1. **Fork escalation** — `src/manager/escalation.ts:122-129` calls `forkSession(parent)` → `sendToAgent(forkName)` → `stopAgent(forkName)`. The forked session is ephemeral — created, used for one response, stopped. `stopAgent(parent)` is NEVER called inside `escalate()`. Phase 73 persistent-subprocess on the parent is untouched because the fork gets its OWN SDK query via `startAgent(forkName, buildForkConfig(...))`.

2. **subagent-thread guidance** — `src/manager/session-config.ts:250` injects the guidance based on `config.skills.includes("subagent-thread")`. This is session-scoped (evaluated once per agent at boot), NOT per-turn. The guidance appears in `toolDefinitionsStr` which is fed to the agent's system prompt and reaches BOTH Discord turns and OpenAI-endpoint turns via the same `TurnDispatcher.dispatchStream` path. No divergence; no gap; no fix needed.

Both invariants are STRUCTURAL — the codepaths simply don't expose a lever to break them. Regression tests pin the invariants so a future refactor that accidentally introduces a lever fails fast in CI.
