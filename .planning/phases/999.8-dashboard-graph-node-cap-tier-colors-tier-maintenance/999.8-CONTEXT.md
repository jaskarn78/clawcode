# Phase 999.8: Dashboard graph fixes + heartbeat check discovery restoration — Context

**Gathered:** 2026-04-29 (--auto, recommended defaults)
**Status:** Ready for planning

> **Scope expansion notice (auto-mode finding):** During codebase scout, the third issue (tier-maintenance investigation) revealed that the **entire heartbeat-check discovery mechanism is silently broken in production** due to a tsup bundling vs runtime-discovery mismatch. Tier-maintenance has never run in prod; neither have ~9 other heartbeat checks. Fixing this is now part of the phase scope (Plan 03).

<domain>
## Phase Boundary

Three issues bundled, ordered from smallest fix to largest:

**1. Lift the 500-node hardcap on `memory-graph` IPC** (`src/manager/daemon.ts:5927`):
- Replaces hardcoded `LIMIT 500` with a configurable cap (default: 5000)
- Caller can pass `limit: number` param to override
- Prod data: fin-acquisition has 1,434 memories — only 35% currently visible

**2. Distinct colors per memory tier in dashboard graph** (`src/dashboard/static/graph.html:421-425`):
- Current `nodeClr` only emits 3 colors: grey (orphan), red (hot), purple (warm + cold lumped)
- Add: yellow/orange for warm, blue for cold, keep red for hot, grey for orphan
- Add a legend in the dashboard UI so users can map colors to tiers

**3. Restore heartbeat-check discovery** — the systemic root cause of "tier maintenance never runs":
- `tsup.config.ts` bundles only `src/cli/index.ts` → `dist/cli/index.js` (single file, `splitting: false`)
- Runtime discovery at `src/heartbeat/discovery.ts:36` does `import(join(checksDir, file))` reading from disk
- Resolved path is `/opt/clawcode/dist/heartbeat/checks/*.js` — but `dist/heartbeat/` does NOT exist after bundling
- Result: discovery returns 0 modules → ALL heartbeat checks are silently broken
- Affected checks: `tier-maintenance`, `consolidation`, `attachment-cleanup`, `thread-idle`, `task-retention`, `trace-retention`, `fs-probe`, `mcp-reconnect`, `context-fill`, `inbox` (heartbeat path)
- Verified: `grep "tier-maintenance" /opt/clawcode/dist/cli/index.js` → 0 hits. Confirms the bundle has no tier-maintenance code.
- Auto-linker is NOT affected — Phase 1.6 moved it OUT of the heartbeat path to eager-on-save (verified: production data shows recent auto-linker activity at 20:56 UTC today, while heartbeat-driven checks have never fired)

**Fix approach for #3:** Replace dynamic disk-based discovery with a static import registry. All check modules imported at compile time and registered in an exported array. The `tsup` bundler then includes them in `dist/cli/index.js` automatically. Loses the file-drop plugin convenience, but gains correctness at production runtime.

**Out of scope (deferred):**
- Backfilling tier values for existing memories (one-shot maintenance run will handle this naturally on the first heartbeat tick post-fix)
- Auto-linker rework (it's working — eager-on-save covers what was previously a 6h heartbeat)
- Dashboard pagination / lazy-load for huge graphs (>5000 nodes is a future concern; 5000 is sufficient for current fleet)
- Per-check enable/disable per-agent (existing config supports this via `defaults.heartbeat.checks.<name>.enabled`)

<decisions>
## Implementation Decisions

### 500-node cap (Plan 01)
- **D-CAP-01:** Default cap raised from 500 → 5000. fin-acquisition's 1,434 memories fit easily. Headroom for ~3-5 years of accumulation per agent at current rates.
- **D-CAP-02:** Cap is configurable via optional `limit` param on the IPC method. Dashboard does NOT need to send the param — picks up the new default. Future operators can override per-call.
- **D-CAP-03:** Validation: `limit` clamped to range `[1, 50000]`. Above 50K is a perf risk; below 1 is invalid. Out-of-range returns `400`-style error verbatim.
- **D-CAP-04:** Schema update minor — add optional `limit` to the `memory-graph` IPC handler params. No breaking change.

### Dashboard color palette (Plan 02)
- **D-COLOR-01:** Color mapping (matches OpenClaw's design tokens — purple is the brand color and stays for "main searchable" warm tier):
  - **Hot** (active context): `#e06c75` red — UNCHANGED, already shipping
  - **Warm** (searchable): `#7f6df2` purple — UNCHANGED, this is the brand color and represents "the main body"
  - **Cold** (archived): `#5a8db8` muted blue — NEW, distinct from warm
  - **Orphan** (no links): `#444` grey — UNCHANGED
- **D-COLOR-02:** Add a legend to the dashboard graph view. Position: top-right corner of the canvas, collapsible, ~150px wide. Shows each tier's color swatch + label + current count.
- **D-COLOR-03:** Sliders for tier filtering NOT in scope this phase (could add a "show only hot/warm/cold/orphan" filter, but that's polish — defer until users request it).

### Heartbeat discovery restoration (Plan 03)
- **D-HB-01:** Replace `src/heartbeat/discovery.ts`'s `readdir + dynamic import` with a static import registry. New file `src/heartbeat/check-registry.ts` exports an array of CheckModule. The 10 existing check files (`attachment-cleanup`, `auto-linker`, `consolidation`, `context-fill`, `fs-probe`, `inbox`, `mcp-reconnect`, `task-retention`, `thread-idle`, `tier-maintenance`, `trace-retention`) import statically.
- **D-HB-02:** `discovery.ts` keeps the same exported function signature `discoverChecks(checksDir): Promise<CheckModule[]>` for back-compat — but the body returns the static registry, ignoring `checksDir`. Tests update accordingly.
- **D-HB-03:** When a future check is added, the developer adds an import + array entry to `check-registry.ts`. One-time pain for permanent correctness.
- **D-HB-04:** Validate fix on prod post-deploy by:
  - Checking `journalctl -u clawcode | grep -E "tier-maintenance|consolidation|context-fill"` shows tick logs after the next heartbeat interval
  - Querying the heartbeat-status IPC returns populated zones
  - Verifying memory tier distribution improves over the next 6-24 hours (some memories transition warm → hot or warm → cold based on access patterns)
- **D-HB-05:** Add a daemon-startup log line announcing how many heartbeat checks were registered (e.g., `"heartbeat: 11 checks registered"`). Catches future regressions if registry gets out of sync.
- **D-HB-06:** Keep `auto-linker` in the registry for completeness even though Phase 1.6 moved its hot path to eager-on-save — the heartbeat version still runs as a safety net for any memories that slipped through.

### Test infrastructure
- **D-TST-01:** Plan 01 tests pin: default `limit` returns 5000 ceiling; explicit `limit: 100` is honored; `limit: -1` returns error; large `memories.length` produces correct paged output.
- **D-TST-02:** Plan 02 tests pin: each tier has a distinct color; legend renders all 4 tier categories.
- **D-TST-03:** Plan 03 tests pin: `discoverChecks()` returns all 11 modules statically; static registry in `check-registry.ts` is in lockstep with the on-disk `src/heartbeat/checks/*.ts` files (a regression test that fails when a new check file is added without registering it — uses readdir to compare).

### Plan structure
- **D-PLN-01:** 3 plans, sequential waves:
  - **Plan 01 (Wave 1):** Lift 500-node cap. Small, isolated. Cover CAP-01..CAP-04.
  - **Plan 02 (Wave 2):** Distinct tier colors + legend. Dashboard-only changes. Cover COLOR-01..COLOR-02.
  - **Plan 03 (Wave 3, depends on Plan 02 for shared dashboard testing):** Heartbeat discovery restoration. Static registry + lockstep regression test. Cover HB-01..HB-06.
- **D-PLN-02:** Plans 01 and 02 could be parallelized (different files), but I lock them sequential for atomic commits and easy review. ~3-4h total work.
- **D-PLN-03:** NO production deploy step. Operator authorizes deploy explicitly.
