# Phase 999.8: Dashboard graph fixes + heartbeat discovery restoration ‚Äî Research

**Researched:** 2026-04-29
**Domain:** TypeScript daemon IPC, dashboard frontend (D3/HTML), Node.js bundler-vs-runtime mismatch
**Confidence:** HIGH (all 3 plans verified end-to-end against the source tree + production journalctl evidence)

## Summary

Three plans, ordered smallest to largest. Plan 01 (lift hardcoded `LIMIT 500`) and Plan 02 (tier color palette + legend) are mechanical, well-bounded edits. Plan 03 (replace dynamic `discoverChecks` with a static registry) is the systemic fix ‚Äî and the production logs CONFIRM the bug: every restart since the last bundling change writes `"checkCount":0,"msg":"heartbeat checks discovered"` to journalctl. Zero checks are running in production. None of the 10 modules has fired since the bug landed.

The fix is small in code (one new file, one rewrite of `discovery.ts`) but high-leverage: tier-maintenance, mcp-reconnect, fs-probe, attachment-cleanup, task-retention, trace-retention, thread-idle, context-fill, inbox-reconciler, and the auto-linker safety net all start running again on the next deploy.

**Primary recommendation:** Implement plans in 3 sequential waves. For Plan 03, write `src/heartbeat/check-registry.ts` exporting a `const CHECK_REGISTRY: readonly CheckModule[]` populated by 11 static imports; rewrite `discoverChecks` to return the registry verbatim (signature unchanged); add a lockstep regression test that uses `readdirSync` at test time to compare against the static registry; add a daemon boot log that surfaces the registered count.

### Pattern 1: Static Registry Module (Plan 03)
**What:** Replace runtime disk scans with compile-time imports the bundler can see.
**When to use:** Any time a tsup `splitting: false` bundle needs to expose a "plugin"-style API.
**Example:**

**What:** Match the existing project style ‚Äî no validator helper, just typeof guard + clamp.

### Anti-Patterns to Avoid
- **Plan 03 ‚Äî keeping the `try/catch {}` swallow:** the current `discovery.ts` silently skips imports that fail. The static-registry approach must NOT swallow ‚Äî if any one of the 11 imports fails to resolve, the daemon should fail fast at boot, not silently lose a check. Static imports already do this (compile-time error), but mention it in the plan.
- **Plan 02 ‚Äî putting the legend inside the SVG:** D3's `<g>` zoom transform would scale the legend with the graph. Use HTML overlay positioned absolute on `.graph-area`.
- **Plan 01 ‚Äî using `prepare` with template-string interpolation for the limit:** ALWAYS use `?` parameter binding. Don't backslide just because limit is a number.

## Common Pitfalls

### Pitfall 1: Plan 03 ‚Äî `inbox.ts` has a NAMED export beyond default
**What goes wrong:** Daemon.ts at line 2120 does `const { setInboxSourceActive } = await import("../heartbeat/checks/inbox.js");`. If a refactor accidentally removes the named export from inbox.ts, the daemon crashes at boot when InboxSource registers.
**Why it happens:** The static registry only consumes `default`; one might assume named exports are unused.
**How to avoid:** Plan 03 MUST NOT modify any check file. The registry imports `default` only, but inbox.ts retains its full export surface (default + `setInboxSourceActive`).
**Warning signs:** Test failures or daemon-boot import errors after Plan 03 changes.

### Pitfall 2: Plan 03 ‚Äî Eager imports may execute module-level side effects
**What goes wrong:** Module-level `new Set<string>()` (auto-linker.ts, tier-maintenance.ts) and module-level `let inboxSourceActive = false` (inbox.ts) run once at first import ‚Äî same as before, but now at daemon boot rather than first heartbeat tick. `mcp-reconnect.ts` does `import pino from "pino"` then likely calls `pino()` at module scope (verify in plan).
**Why it happens:** ESM static imports execute immediately.
**How to avoid:** Audit each of the 11 check modules' top-level statements. Document any side effect that's now happening earlier (boot vs first tick). None observed are problematic ‚Äî they're all initialization (Set, flag, logger) that's inert until `execute()` is called.
**Warning signs:** Boot-time stack traces; missing env-var reads; imports of files that don't exist at boot time (e.g., DB files lazily-created).

### Pitfall 3: Plan 03 ‚Äî Failure to fail-fast on registry/disk drift
**What goes wrong:** A new check file is added in `src/heartbeat/checks/` but not registered. With dynamic discovery, it would silently appear; with static registry, it silently disappears.
**Why it happens:** Easy to forget the registry update during code review.
**How to avoid:** Implement the lockstep regression test (D-TST-03). Read `src/heartbeat/checks` at test time, filter to non-test files (`.ts` excluding `.test.ts` and `__tests__`), strip `.ts`, compare against `CHECK_REGISTRY.map(c => c.name)` (which is the runtime name, NOT filename). Use `name.replace(/-/g, "")` to derive expected filename ‚Äî OR keep an explicit `expectedFiles` list in the test for clarity. Recommendation: compare `filenames-without-ext` set against a hand-maintained `EXPECTED_FILENAMES` set in the test, and ALSO assert each filename's default-export name matches the registry entry.
**Warning signs:** Test passes locally but registry has missing entries; would catch this only if test runs CI on every PR.

### Pitfall 4: Plan 02 ‚Äî Legend pointer-events shadowing
**What goes wrong:** The legend overlay sits on top of `.graph-area` SVG. Without `pointer-events: none` (or careful sizing), it can intercept zoom/drag near the corner.
**Why it happens:** Default `pointer-events: auto` on positioned elements.
**How to avoid:** Set the legend to `pointer-events: auto` (so the `<details>` toggle works) but constrain it to a small, fixed-width box that's clearly out of the way. The 150px wide top-right placement matches CONTEXT D-COLOR-02 and leaves 95% of the canvas unobstructed. Test by scrolling/dragging in the corner area ‚Äî if legend captures the event, narrow its width or disable pointer events on the container while keeping them on the toggle.
**Warning signs:** Users report "can't drag in upper-right area".

### Pitfall 5: Plan 01 ‚Äî `LIMIT N` with N > total rows
**What goes wrong:** None ‚Äî SQLite returns all rows, no error. But a naive test that checks `.toHaveLength(limit)` will fail for small DBs.
**Why it happens:** Misunderstanding test expectations.
**How to avoid:** Test asserts `length <= limit AND length === expectedRowCount`, not `length === limit`. The handler is correct; the TEST must be correct.
**Warning signs:** Test failures with "expected 5000 received 47".

### Pitfall 6: Plan 03 ‚Äî Production deploy without rebuild
**What goes wrong:** Operator deploys updated source files but skips `tsup` rebuild. Production still runs the old bundle with the old broken discovery.
**Why it happens:** Source-only deploys that rely on tsx-style live execution (the daemon does NOT ‚Äî it runs the bundled `dist/cli/index.js`).
**How to avoid:** Plan 03 deploy step MUST include the bundle build. Verification: `grep -c "tier-maintenance" /opt/clawcode/dist/cli/index.js` should return ‚â• 1 after the build (today: 0).
**Warning signs:** Post-deploy journalctl still shows `checkCount:0`.

### Pitfall 7: Plan 03 ‚Äî `try/catch{}` removal must be intentional
**What goes wrong:** Current `discovery.ts:46` swallows ALL import errors. Replacement code must NOT preserve that behavior; static imports throw at module load (boot-time crash), which is the desired behavior. If a check is broken, the daemon should refuse to start so the operator notices, rather than silently running with 10 of 11 checks.
**Why it happens:** Defensive-coding instinct to wrap things in try/catch.
**How to avoid:** No `try/catch` in `check-registry.ts`. If any check module is malformed, the bundler/runtime will surface the error.
**Warning signs:** Plan author proposes adding `try{ import }catch` ‚Äî reject in plan-check.

### Pitfall 8: Plan 03 ‚Äî Test file may need `import.meta.dirname` adjustment
**What goes wrong:** The lockstep test uses `readdirSync` against `src/heartbeat/checks` ‚Äî but at test runtime, the working directory may not be the repo root.
**Why it happens:** Vitest runs from project root by default, but the test file may need to resolve `__dirname`-relative paths.
**How to avoid:** Use `fileURLToPath(import.meta.url)` + `dirname` to anchor the path relative to the test file location. Then `join(__dirname, "../checks")` resolves regardless of cwd.
**Warning signs:** Test passes when run via `npm test` from root but fails in IDE-driven runs.

import { CHECK_REGISTRY } from "../check-registry.js";

const __dirname = dirname(fileURLToPath(import.meta.url));
const checksDir = join(__dirname, "../checks");

/** Map of registry NAME ‚Üí expected filename slug. Maintained by hand
 *  alongside check-registry.ts so a missing entry fails this test. */
const EXPECTED_FILENAMES: ReadonlyMap<string, string> = new Map([
  ["attachment-cleanup", "attachment-cleanup.ts"],
  ["auto-linker", "auto-linker.ts"],
  ["consolidation", "consolidation.ts"],
  ["context-fill", "context-fill.ts"],
  ["fs-probe", "fs-probe.ts"],
  ["inbox", "inbox.ts"],
  ["mcp-reconnect", "mcp-reconnect.ts"],
  ["task-retention", "task-retention.ts"],
  ["thread-idle", "thread-idle.ts"],
  ["tier-maintenance", "tier-maintenance.ts"],
  ["trace-retention", "trace-retention.ts"],
]);

describe("CHECK_REGISTRY ‚Üî disk lockstep", () => {
  it("registers all 11 known checks", () => {
    expect(CHECK_REGISTRY).toHaveLength(11);
  });

  it("every CHECK_REGISTRY entry has a unique name", () => {
    const names = CHECK_REGISTRY.map(c => c.name);
    expect(new Set(names).size).toBe(names.length);
  });

  it("every CHECK_REGISTRY entry corresponds to a file on disk", () => {
    const onDisk = new Set(
      readdirSync(checksDir).filter(
        f => f.endsWith(".ts") && !f.endsWith(".test.ts") && f !== "__tests__",
      ),
    );
    for (const check of CHECK_REGISTRY) {
      const expectedFile = EXPECTED_FILENAMES.get(check.name);
      expect(expectedFile, `unmapped check name '${check.name}'`).toBeDefined();
      expect(onDisk.has(expectedFile!), `missing file ${expectedFile}`).toBe(true);
    }
  });

  it("every check file on disk is registered (drift detector)", () => {
    const onDisk = readdirSync(checksDir).filter(
      f => f.endsWith(".ts") && !f.endsWith(".test.ts") && f !== "__tests__",
    );
    const registered = new Set(
      [...EXPECTED_FILENAMES.values()],
    );
    for (const f of onDisk) {
      expect(
        registered.has(f),
        `file ${f} is on disk but not in CHECK_REGISTRY ‚Äî register it in src/heartbeat/check-registry.ts`,
      ).toBe(true);
    }
  });

  it("each registry entry exposes the CheckModule contract", () => {
    for (const check of CHECK_REGISTRY) {
      expect(typeof check.name).toBe("string");
      expect(check.name.length).toBeGreaterThan(0);
      expect(typeof check.execute).toBe("function");
    }
  });
});
