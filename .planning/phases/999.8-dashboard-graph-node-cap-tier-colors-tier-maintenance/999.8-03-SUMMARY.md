## Final Shape — `discovery.ts`

The body is now a one-liner:

```typescript
export async function discoverChecks(
  _checksDir: string,
): Promise<readonly CheckModule[]> {
  return CHECK_REGISTRY;
}
```

The async signature, return type, and parameter shape are preserved for back-compat with `runner.ts:90`'s `await discoverChecks(this.checksDir)` and any other callers. The `_checksDir` arg is intentionally unused (underscore convention) and the JSDoc explains the bundler-vs-runtime mismatch that motivated the rewrite. Zero `readdirSync` or dynamic `import(...)` references remain.

## Final Shape — `runner.ts initialize()` Boot Log

```typescript
async initialize(): Promise<void> {
  this.checks = await discoverChecks(this.checksDir);
  this.log.info(
    {
      checkCount: this.checks.length,
      checks: this.checks.map((c) => c.name),
    },
    "heartbeat checks registered",
  );
}
```

Two changes from the prior shape: (1) the log object now includes `checks: [...names]` so operators see exactly which 11 checks are alive in the bundle; (2) the message string changed from `"heartbeat checks discovered"` → `"heartbeat checks registered"` to give post-deploy `journalctl | grep` a clean marker against the prior (broken) deployment's log lines.

## Decisions Made

- **Static registry over preserving the readdir scan.** tsup `splitting:false` makes runtime disk discovery structurally broken in the bundled output — there's no `dist/heartbeat/checks/` for `readdirSync` to find. Static imports give the bundler the dependency graph it needs.
- **No try/catch wrapping the imports.** Pitfall 7 was explicit about this — the prior `discovery.ts` had a `try/catch{}` swallow at line 46 that silently dropped failed imports. The static registry inherits ESM's fail-fast semantics: if any check module is broken, the daemon refuses to start, which is the correct behaviour. Defensive try/catch would re-introduce the silent-skip bug class.
- **Async signature preserved on `discoverChecks` despite a synchronous body.** runner.ts:90 already calls `await discoverChecks(...)` and there may be other callers — keeping the signature stable means zero downstream churn for a body change. The unused `_checksDir` arg is underscored to satisfy `noUnusedParameters` and to telegraph to readers that it's a back-compat residue.
- **Hand-maintained `EXPECTED_FILENAMES` map in the test.** Could have auto-derived filenames from registry names + `.ts`, but explicit-by-design forces the developer adding a future check to update three places (import, array entry, EXPECTED_FILENAMES entry). That's the prevention surface for the kind of drift that just bit production for ~10 weeks.
- **Boot log message string changed (`discovered` → `registered`).** Gives operators a clean grep marker. Old (broken) deployments wrote `"heartbeat checks discovered"`; healthy ones now write `"heartbeat checks registered"`. The two strings cleanly co-exist in journalctl history without ambiguity.
- **Born-green inbox-named-export test.** The pre-existing daemon.ts:2120 dynamic import of `setInboxSourceActive` is a HB-03 landmine — easy to break in a future refactor that "consolidates" inbox.ts to a single default export. Pinning the contract today (when both exports exist) means any removal trips the suite immediately.
