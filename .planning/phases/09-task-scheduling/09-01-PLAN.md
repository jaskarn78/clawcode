    Create `src/scheduler/scheduler.ts` implementing `TaskScheduler` class (per D-04, D-05, D-06, D-07, D-08, D-09, D-10):

    ```
    class TaskScheduler {
      private readonly sessionManager: SessionManager;
      private readonly log: Logger;
      private readonly jobs: Map<string, Cron[]>;  // agentName -> cron jobs
      private readonly statuses: Map<string, ScheduleStatus[]>;  // agentName -> statuses
      private readonly locks: Map<string, boolean>;  // agentName -> isRunning (per D-05)

      constructor(options: TaskSchedulerOptions)

      addAgent(agentName: string, schedules: readonly ScheduleEntry[]): void
        - For each enabled schedule, create a Cron job using croner
        - On trigger: acquire per-agent lock, call sendToAgent(agentName, schedule.prompt), update status, release lock
        - If lock held (another task running), skip this execution (per D-05)
        - On success: update lastRun, lastStatus="success", compute nextRun from cron
        - On error: update lastRun, lastStatus="error", lastError=message (per D-07)
        - Store initial status as "pending" with nextRun computed from Cron.nextRun()

      removeAgent(agentName: string): void
        - Stop all Cron jobs for agent, remove from maps

      getStatuses(): readonly ScheduleStatus[]
        - Return flattened array of all statuses across all agents

      getAgentStatuses(agentName: string): readonly ScheduleStatus[]
        - Return statuses for a specific agent

      stop(): void
        - Stop all Cron jobs for all agents
    ```

    Use `import { Cron } from "croner"` for scheduling. Each Cron instance manages its own timer internally -- no setInterval needed.

    For the per-agent sequential lock (per D-05): use a simple boolean flag per agent. When a cron triggers, check if lock is held. If yes, skip. If no, acquire lock, execute, release. This prevents overlapping scheduled tasks within the same agent.

    For nextRun computation: use `cronJob.nextRun()?.getTime() ?? null` after job creation and after each execution.

    Tests should use croner's `{ paused: true }` option to create jobs without auto-starting, then call the trigger callback manually to test execution logic. Alternatively, mock the Cron constructor.
