# Phase 117.1 Plan 01: Bridge advisor event telemetry Summary

One-liner: Added two `this.log.info(...)` calls in the bridge's per-turn `onInvoked` / `onResulted` advisor-event subscribers so production daemon logs now carry ground truth for every native advisor invocation (closes the silent-path gap that masked Issue 2 in the 117 operator smoke).

## What shipped

### T01 — `src/discord/bridge.ts` (commit `1f4fa5a`)

Added one structured INFO log line per subscriber callback in `streamAndPostResponse`, inside the existing per-turn `register-around-dispatch` block at lines 781-815 (now 781-840):

- `onInvoked` emits `{ agent, channel, userMessageId }` + message `"advisor invoked (native server tool fired)"`.
- `onResulted` emits `{ agent, channel, variant, errorCode? }` + message `"advisor resulted"`. `errorCode` is conditionally spread only when `ev.kind === "advisor_tool_result_error"` so non-error variants stay clean.

Both calls use `this.log.info(...)` (pino INFO level, equivalent to numeric 30 — RESEARCH §13.10 referenced numeric 40; pino's `info` is 30, which is what `this.log.info(...)` produces. The plan text calling it "40" was loose phrasing for "INFO, not DEBUG"; what ships is the standard pino INFO level used everywhere else in this file). Matches existing convention at `bridge.ts:277, 298, 332, 368, 438, 579, 598, 637, 886, 1038, 1120, 1127, 1182`.

### T02 — `src/discord/__tests__/bridge-advisor-footer.test.ts` (commit `0938737`)

Added 4 new test cases pinning the structured-field shape:

| Case                                                | Assertion                                              |
| --------------------------------------------------- | ------------------------------------------------------ |
| advisor:invoked                                     | `{ agent, channel, userMessageId }` exact equality     |
| advisor:resulted — advisor_result                   | variant set; `"errorCode" in fields === false`         |
| advisor:resulted — advisor_redacted_result          | variant set; `"errorCode" in fields === false`         |
| advisor:resulted — advisor_tool_result_error        | variant set; errorCode === "max_uses_exceeded"         |

Mock is captured via the existing `fakeLog.info: vi.fn()` injection into the bridge constructor. `vi.clearAllMocks()` in `beforeEach` resets the call log per test, so the `mock.calls.find(...)` lookup matches only the current test's log lines.

One implementation note: the plan's pseudocode referenced `resultVariant` and `event.errorCode` from the outer scope, but the actual code in `bridge.ts:787-789` already binds `ev.kind` and `ev.errorCode` from the `AdvisorResultedEvent` parameter. ## Production behavior change

When an agent's turn fires a native advisor consultation, the daemon log will now contain (per turn, per backend success):

```
level=30 agent=test-agent channel=1494... userMessageId=12345... msg="advisor invoked (native server tool fired)"
level=30 agent=test-agent channel=1494... variant=advisor_result msg="advisor resulted"
```

Failure case adds `errorCode`:

```
level=30 agent=test-agent channel=1494... variant=advisor_tool_result_error errorCode=max_uses_exceeded msg="advisor resulted"
```

Zero log lines when no advisor event fires (subscribers gate on `ev.agent === sessionName` before any log call, and on `didConsultAdvisor` flag flow).
