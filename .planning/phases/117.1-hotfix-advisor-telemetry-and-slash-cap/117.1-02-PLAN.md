# Plan 117.1-02 — Slash command guild-cap remediation

**Phase:** 117.1
**Status:** ready
**Dependencies:** none
**Estimated tasks:** 4

## Goal

Reduce the per-guild slash command count from 193 to ≤90 so Discord registration succeeds and `/clawcode-verbose` (plus every other CONTROL_COMMAND added since the last successful registration) becomes visible in the slash picker.

## Why

Phase 117 shipped `/clawcode-verbose` via `CONTROL_COMMANDS` (Plan 117-11 T03). Production daemon log at `23:54:52`:

```
level=ERROR  commandCount=193  limit=90
msg="too many commands for guild — refusing to register (CMD-07)"
```

The Phase 87 pre-flight assertion at `src/discord/slash-commands.ts:1186` correctly refused the batch to avoid partial-state, but the side effect is that NOTHING registered for that guild. Every recent slash command addition is silently absent.

## Files touched

- **MODIFIED** `/etc/clawcode/clawcode.yaml` on clawdy (production) — prune per-agent `slashCommands` blocks.
- **AUDIT** local `clawcode.example.yaml` — mirror the production pruning so the example stays representative.
- **POSSIBLY MODIFIED** `src/discord/slash-commands.ts:1186` cap value (90 → 95) if pruning alone is insufficient.

## Tasks

### T01 — Audit the production slash command surface

SSH to clawdy and dump the registration payload:

```bash
ssh clawdy 'cat /etc/clawcode/clawcode.yaml' | \
  python3 -c "
import sys, yaml
y = yaml.safe_load(sys.stdin)
total = 0
for agent in y.get('agents', []):
    cmds = agent.get('slashCommands', []) or []
    if cmds:
        names = [c['name'] for c in cmds]
        print(f'{agent[\"name\"]}: {len(names)} — {names}')
        total += len(cmds)
print(f'TOTAL per-agent slashCommands: {total}')
"
```

Combined with 51 (DEFAULT_SLASH_COMMANDS) + 15 (CONTROL_COMMANDS) = 66 baseline. The per-agent total bridges to 193 (so ~127 per-agent commands).

Produce `SLASH-AUDIT.md` in the phase directory with:
- Per-agent table: agent name → count → command list
- Redundancy analysis: which entries duplicate CONTROL_COMMANDS, which are unused (no recent invocations in 30 days of logs if possible)
- Recommended prune list with rationale

## Out of scope

- Per-guild sharding (split commands across multiple guilds) — would require multi-guild bot architecture changes.
- Lazy command registration (only register on first invocation) — Discord doesn't support this pattern; commands must be pre-registered.
- Re-architecting `CONTROL_COMMANDS` to use sub-commands like Phase 999.32 did for GSD — possible follow-up if cap pressure returns.
