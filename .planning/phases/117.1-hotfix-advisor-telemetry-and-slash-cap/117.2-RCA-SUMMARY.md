# Phase 117.2 — Native Advisor Production Activation RCA + Resolution

**Date:** 2026-05-13
**Outcome:** ✅ Phase 117 native advisor verified working end-to-end in production.

## TL;DR

The native advisor path through `sdk.query()` was blocked by **a config-side ghost**, not a code bug. Production yaml had `defaults.advisor.backend: fork` set — a leftover from an interrupted yaml-patcher attempt earlier in the 117.1 session. This made `shouldEnableAdvisor` short-circuit to `false`, the spread-conditional in `buildSessionConfig` never fired, and `Options.advisorModel` + `Options.extraArgs` never reached the SDK. The entire native code path — written in Phase 117, instrumented in 117.1, patched with `extraArgs` injection in 117.1 — was correct the whole time.

## The four silent-path bifurcations of this work (operator-flagged pattern)

This investigation hit the `feedback_silent_path_bifurcation` failure mode FOUR times in one session. Documenting for future prevention:

1. **Phase 117-04 spike (T01):** Verified TypeScript type `Options.advisorModel?: string` exists in SDK 0.2.132. Concluded "outcome B" (SDK exposes the option). Did NOT verify end-to-end that the option produces a CLI arg on spawn. The type existed; the runtime wiring did not. → Mitigation: type-existence checks are necessary but NOT sufficient.

2. **117.1 SDK bump 0.2.132 → 0.2.140:** Assumed newer version would wire `advisorModel`. Confirmed `assistant.mjs` has advisor wiring but didn't confirm `sdk.mjs` (the entry point ClawCode uses) does too. → Mitigation: confirm the *specific entry point* the consumer code uses, not the package as a whole.

3. **117.1 `extraArgs` injection:** Added `extraArgs: { advisor: <model> }` to Options based on `sdk.d.ts:1268` docstring. Deployed; no `--advisor` in spawn args. Initially concluded extraArgs was also unwired — but Agent 1's deep trace showed the iteration loop in minified `u7.initialize()` does emit args from extraArgs. The actual reason it didn't appear was Issue #4 (below). → Mitigation: when a fix appears to silently fail, distinguish "feature not wired" from "feature not reached" before re-patching.

4. **The ghost yaml block (the actual RCA):** A `defaults.advisor.backend: fork` block had been written to production yaml by an earlier yaml-patcher attempt that the operator interrupted. The interrupt landed AFTER the sed insert completed. The block lingered through all subsequent restarts, defeating every fix attempt. → Mitigation: on operator interrupt of a mutation operation, verify whether the mutation completed before assuming it was rolled back.

## The diagnostic that exposed the ghost

Step 1 of the approved plan: enable `debug.dumpBaseOptionsOnSpawn: true` for Admin Clawdy. The dump at `/home/clawcode/.clawcode/agents/Admin_Clawdy/diagnostics/baseopts-create-*.json` showed `baseOptions` had **none** of `advisorModel`, `extraArgs`, or `settings`. The spread-conditional in session-adapter never fired — proof that `shouldEnableAdvisor` returned false upstream.

Then a temporary `phase117.2-advisor-gate` console.info inside `shouldEnableAdvisor` (Phase 117.2-DIAGNOSTIC commit, since reverted) logged:

```
{"agent":"Admin Clawdy","backend":"fork","budgetSet":true,"canCall":true,
 "defaultsShape":"{\"advisor\":{\"backend\":\"fork\",\"model\":\"opus\",\"maxUsesPerRequest\":3}}",
 "decision":"DISABLE"}
```

Two lines of telemetry pinpointed the ghost in two minutes. The plan's "diagnose first, don't pick a fix without seeing the dump" gate is the explicit countermeasure to this session's repeated silent-path failures.

## Resolution

1. Removed `defaults.advisor` block from production `/etc/clawcode/clawcode.yaml`. Snapshot kept at `clawcode.yaml.pre-117.2-remove-fork-default`.
2. Restored `orphanClaudeReaper: { mode: alert }` block that got nicked in an over-eager sed (also restored from snapshot before re-doing surgically).
3. Restarted daemon. Gate evaluation flipped to `decision: "ENABLE"`.
4. Operator re-ran canonical advisor-trigger prompt (zero-downtime schema migration). All four signals fired.
5. Removed the temporary diagnostic log line from `src/manager/session-config.ts:shouldEnableAdvisor`.
6. Rebuilt + redeployed clean.

## Known minor issue (follow-up, not blocking)

The `advisor:resulted` event subscriber in `bridge.ts:799` (registered alongside `:invoked` subscriber) didn't fire on this test — no `advisor resulted` log line appeared. The `:invoked` subscriber + budget counter + footer all worked, so functionally the system is fine. But the `:resulted` emit site in `session-adapter.ts` content-block scanner may have a gap where `advisor_tool_result` blocks aren't being detected. Worth a 117.3 mini-investigation; non-urgent.

## Lessons for future SDK integrations

1. **TypeScript-type ≠ runtime-wired.** Pre-1.0 SDKs ship typed-but-unimplemented forward-looking fields routinely. Always probe with `grep` against the implementation file (`.mjs`), not just the `.d.ts`. Even better: verify end-to-end via live invocation that the option produces the expected effect at the lowest observable layer (process args, env, network call).

2. **Use the existing diagnostic infrastructure first.** ClawCode already has `dumpBaseOptionsOnSpawn` (Phase 115 sub-scope 14) that captures the exact SDK Options object. Two-minute diagnostic. Should have been Step 1 of every fix attempt this session.

3. **Interrupted mutations may have completed.** When the operator interrupts a tool call, the local state may already be different from before the call. Re-verify before assuming rollback.
