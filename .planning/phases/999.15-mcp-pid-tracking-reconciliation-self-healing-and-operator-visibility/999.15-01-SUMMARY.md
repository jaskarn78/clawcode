## Decisions Made

1. **isPidAlive validates pid <= 0 and non-finite input WITHOUT calling process.kill.** Guards against stale tracker values rather than throwing. Returning false for bad input is safer than NaN propagation through the kill path.
2. **EPERM treated as alive in isPidAlive.** A live proc owned by a different user is still "running" — for our purposes (reconciliation, "should we re-discover claudePid?"), only ESRCH proves death.
3. **Immutable mutation pattern over copy-on-write Maps.** Each updateAgent/replaceMcpPids creates a new RegisteredAgent object but reuses the same internal Map. Callers holding prior entry references see pre-mutation state. Cheaper than cloning the Map; sufficient for the reconciler's diff-based logging in Plan 02.
4. **Sync entry write + async cmdline enrichment for register.** Tracker state is observable to subsequent sync code immediately (matches the orchestrator's invariant per CONTEXT). Cmdline cache enrichment runs after the entry write — failures during enrichment don't roll back state.
5. **getRegisteredAgents returns the live Map cast as ReadonlyMap (no copy).** TypeScript's structural subtyping makes Map → ReadonlyMap a zero-cost cast. Callers cannot mutate via the readonly type. Reference identity of values changes per update — pinned by PT-3, PT-4.
6. **pruneDeadPids returns `{ pruned, alive }` partition.** Idiomatic for caller logging (reconciler logs `oldMcpCount, newMcpCount`). Idempotent: second call with same /proc state is a no-op.
7. **list/listForAgent/unregister keep their existing mutable `number[]` return shape.** Avoids breaking existing call patterns like `.sort()` and array spread; fresh-copy semantics already in place from 999.14.
8. **One-line surgical update to session-manager.ts:751 only.** The polled-discovery rewrite that replaces the surrounding fire-and-forget settle block ships in Plan 02. Here we just bump the existing 2-arg call to 3-arg using the local claudePid var already in scope.

### Auto-fixed Issues

**1. [Rule 1 - Bug] Fixed broken node:fs/promises mock wrapper in proc-scan.test.ts**
- **Found during:** Task 1 verification (PS-6, PS-7 returned null instead of expected pid).
- **Issue:** The Wave 0 wrapper called the spy as a side effect but discarded its return value, then returned `actual.readFile(...)` unconditionally. As a result `readFileSpy.mockImplementation(...)` could not drive synthetic /proc returns — PS-6 / PS-7 were structurally untestable.
- **Fix:** Wrapper now uses the spy's return value when defined and falls through to `actual.readFile(...)` only when the spy returns undefined. Preserves Test 9 (security — checks paths in mock.calls) and Linux integration tests (which don't configure mockImplementation).
- **Files modified:** src/mcp/__tests__/proc-scan.test.ts (mock wrapper at lines 24-43).
- **Verification:** All 24 proc-scan tests pass after the fix.
- **Commit:** ba873b3.

**2. [Rule 1 - Bug] Rewrote PT-6 to spy on process.kill instead of brittle vi.importMock pattern**
- **Found during:** Task 2 verification (PT-6 expected pruned=[102] but got pruned=[101,102,103]).
- **Issue:** Wave 0 PT-6 attempted to override `proc-scan.isPidAlive` via `vi.importMock(...)` then `vi.spyOn(procScan, "isPidAlive").mockImplementation(...)`. Because the file-top `vi.mock("../proc-scan.js", ...)` factory uses `...actual` to spread, `isPidAlive` is the real function — and ES module live-binding semantics mean the spyOn override doesn't propagate to the binding in process-tracker.ts. The probe-and-fallback branch (`if typeof isPidAlive !== "function"`) silently failed to mutate the read-only module namespace.
- **Fix:** Replaced the proc-scan mock-module gymnastics with `vi.spyOn(process, "kill").mockImplementation((pid) => { if (pid === 102) throw ESRCH; return true; })`. Drives `isPidAlive`'s behavior at the syscall layer — robust, deterministic, and matches existing test patterns (Test 2/3/4 of killGroup use the same spy-on-process.kill idiom).
- **Files modified:** src/mcp/__tests__/process-tracker.test.ts (PT-6 body, lines ~451-490).
- **Verification:** PT-6 GREEN; all other PT cases unaffected.
- **Commit:** d642de3.
