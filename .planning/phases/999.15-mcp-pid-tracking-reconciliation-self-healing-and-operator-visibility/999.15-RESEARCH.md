**TRACK-01 reconciliation:**
- MUST be idempotent — running twice with no /proc changes produces zero log emissions and zero tracker mutations.
- MUST handle the "agent process completely gone" case (SDK gave up, agent stopped) — drop the agent's tracker entry entirely. NEW reason value: `"agent-gone"`.
- MUST NOT block the orphan reaper. Reconciler runs sequentially AFTER the orphan sweep; if reconciliation throws, log and continue (best-effort).

**TRACK-02 polling discovery:**
- Total wait budget: 30s max (6 attempts × 5s).
- Abort if the agent handle is disposed during wait.
- `minAge: 5s` filter passed to `discoverClaudeSubprocessPid` — REUSE the existing filter param, don't rewrite.

**TRACK-03 tracker API:**
- All mutations are immutable internally — each `update*`/`replace*` creates a new object reference for the agent's entry.
- `getRegisteredAgents()` returns `ReadonlyMap` — callers cannot mutate.
- `isPidAlive(pid)` uses `process.kill(pid, 0)` and catches `ESRCH` (returns false). Catches `EPERM` (returns true — process exists, we just can't signal it).

**TRACK-04 logging:**
- Diff-based — only log when state actually changed. Use deep-equal on `{ claudePid, mcpPids[].sort() }`.
- Single log line per cycle PER AGENT. 3 agents reconciled in one tick → 3 separate log lines.

**TRACK-05 CLI:**
- Read-only — does not mutate tracker state.
- Returns non-zero exit code if any agent has 0/N MCP procs alive (operator-actionable signal).

**TRACK-06 graceful kill:**
- Reconcile FIRST, THEN kill. If reconciliation throws, fall back to killing the recorded PIDs (safety net).
- Don't reconcile inside `killAll()` (whole-daemon-shutdown — /proc walks may hit timing issues). Use the recorded PIDs from a recent reconcile cycle.

## Summary

Phase 999.15 is a **pure additive extension** of the 999.14 substrate. All required wiring seams already exist: `onTickAfter` was added in 999.14 specifically as an extension hook (Wave 0 Test 8); `discoverClaudeSubprocessPid` and `discoverAgentMcpPids` are stable; `McpProcessTracker` already exposes `register` / `unregister` / `killAgentGroup` / `killAll` / `patterns`. The phase adds five small APIs to the tracker (`updateAgent`, `replaceMcpPids`, `getRegisteredAgents`, `pruneDeadPids`, `isPidAlive`), one new module (`src/mcp/reconciler.ts`), one new CLI (under a name that does not collide with the existing `mcp-status`), one new IPC method (`mcp-tracker-snapshot`), and replaces a 4-line fire-and-forget settle block in `session-manager.ts` with a polled-wait loop.

**The single biggest risk** is the `mcp-status` naming collision — Phase 85 owns that command name for per-server readiness reporting, and rebinding it to mean "tracker PID inspection" would break operator muscle memory and a working production command. **Recommend** the new command be named `clawcode mcp-tracker` (or `mcp-pids`); the planner should lock the name in the first plan.

**The second risk** is concurrency between the polled discovery in `agent.start` (TRACK-02) and the reconciler tick (TRACK-01) — both call `tracker.register/updateAgent/replaceMcpPids` for the same agent. Node.js single-threaded event-loop ordering serializes them naturally because **all tracker methods are sync** (no `await` inside `register`'s critical section — only the outer cmdline-cache enrichment is async, and it's best-effort). This invariant must be documented in code comments AND pinned by a test.

**Primary recommendation:** Build a single `src/mcp/reconciler.ts` module exporting `reconcileAllAgents(args)` (the per-tick loop) and `reconcileAgent(name, args)` (the single-agent variant called by `killAgentGroup`). Wire the loop into `onTickAfter` at the existing daemon-boot site (1576-1631). Extend `McpProcessTracker` with the five new methods. Replace the settle block in `session-manager.ts:721-766` with a polled loop using `vi.useFakeTimers`-friendly `setTimeout` calls. Add `clawcode mcp-tracker` CLI in a new `src/cli/commands/mcp-tracker.ts`. Add IPC method in the routeMethod switch in `daemon.ts:5285+`.

**Estimated surface:** ~280 lines added across 7 files, 5 new test files, zero npm deps. Well within Phase 105 "surgical diffs" guidance.

### Pattern 2: Polled Discovery (TRACK-02)

**What:** Replace the 1s settle + single discovery attempt with up to 6 attempts of 5s each, using `minAge: 5s` to skip freshly-spawned claude processes that may be respawn casualties.

**Current code at `session-manager.ts:721-766`:**

```typescript
// CURRENT — fire-and-forget, single attempt with 1s settle
if (this.mcpTracker) {
  void (async () => {
    try {
      await new Promise((r) => setTimeout(r, MCP_SPAWN_SETTLE_MS));  // 1000ms
      const claudePid = await discoverClaudeSubprocessPid(process.pid);
      if (claudePid == null) {
        this.log.warn({ agent: name }, "could not locate claude subprocess PID...");
        return;
      }
      // ... register
    } catch (err) { /* ... */ }
  })();
}
```

**Replacement (TRACK-02):**

```typescript
// NEW — polled, 6 attempts × 5s, minAge=5s filter
if (this.mcpTracker) {
  void (async () => {
    let claudePid: number | null = null;
    for (let attempt = 0; attempt < 6; attempt++) {
      await sleep(5000);
      // Abort if the agent handle was disposed during the wait
      if (!this.sessions.has(name)) return;
      claudePid = await discoverClaudeSubprocessPid(process.pid, { minAge: 5 });
      if (claudePid !== null) break;
    }
    if (claudePid === null) {
      this.log.warn(
        { agent: name },
        "no claude proc settled within 30s — relying on reaper reconciliation",
      );
      return;
    }
    const tracker = this.mcpTracker;
    if (!tracker) return;
    let patterns: RegExp;
    try { patterns = tracker.patterns; } catch { return; }
    const mcpPids = await discoverAgentMcpPids(claudePid, patterns);
    await tracker.register(name, claudePid, mcpPids);  // NEW SIGNATURE — claudePid is now first-class
    this.log.info({ agent: name, claudePid, mcpPidCount: mcpPids.length },
      "registered agent (Phase 999.15 TRACK-02)");
  })();
}
```

**Note:** `tracker.register` signature MUST change to accept `claudePid` so the tracker can store it for reconciliation. Current 999.14 signature is `register(agentName, pids)`. New: `register(agentName, claudePid, mcpPids)`. This is a breaking change to the tracker API but only one caller (session-manager.ts) — safe.

```typescript
// Source: man 2 kill — signal 0 checks existence + permission only
export function isPidAlive(pid: number): boolean {
  if (pid <= 0) return false;  // 0 = caller's process group; <0 = invalid
  try {
    process.kill(pid, 0);
    return true;  // process exists AND we have permission
  } catch (err: unknown) {
    const e = err as NodeJS.ErrnoException;
    if (e.code === "ESRCH") return false;  // no such process
    if (e.code === "EPERM") return true;   // exists but we can't signal — still alive
    throw err;  // unexpected — propagate
  }
}
```

**Pin-test invariants:**
- `isPidAlive(0)` returns false (validated, no syscall)
- `isPidAlive(-1)` returns false
- `isPidAlive(process.pid)` returns true
- `isPidAlive(99999999)` returns false (ESRCH)

### Pattern 4: Diff-Based Logging (TRACK-04)

**What:** Reconciler captures snapshots before and after each agent's reconciliation. Only emits a log line when the snapshot diff is non-empty.

```typescript
// Source: derived from CONTEXT.md <specifics> reconciler pseudocode
type ReconcileReason = "stale-claude" | "missing-mcps" | "agent-restart" | "agent-gone";

function diffSnapshots(
  before: { claudePid: number; mcpPids: readonly number[] },
  after:  { claudePid: number; mcpPids: readonly number[] },
): boolean {
  if (before.claudePid !== after.claudePid) return true;
  if (before.mcpPids.length !== after.mcpPids.length) return true;
  const a = [...before.mcpPids].sort();
  const b = [...after.mcpPids].sort();
  for (let i = 0; i < a.length; i++) if (a[i] !== b[i]) return true;
  return false;
}
```

**Canonical log envelope (pin in tests):**
```json
{
  "level": 40,
  "component": "mcp-tracker",
  "action": "reconcile",
  "agent": "fin-acquisition",
  "oldClaudePid": 4067770,
  "newClaudePid": 4068098,
  "oldMcpCount": 0,
  "newMcpCount": 8,
  "reason": "stale-claude",
  "msg": "tracker state reconciled"
}
```

**No log on no-op cycles** — required for journal cleanliness on busy hosts where 14 agents tick every 60s.

### Pattern 5: Reconcile-Before-Kill (TRACK-06)

**What:** `killAgentGroup` first calls `reconcileAgent(name)` to sync /proc state before the existing SIGTERM/grace/SIGKILL flow.

```typescript
// Source: extends process-tracker.ts:153-160 killAgentGroup
async killAgentGroup(agentName: string, graceMs: number = 5000): Promise<void> {
  // NEW 999.15 TRACK-06: reconcile before kill (best-effort safety net)
  try {
    await reconcileAgent(agentName, {
      tracker: this,
      daemonPid: process.pid,
      log: this.log,
    });
  } catch (err) {
    this.log.warn({ agent: agentName, err: String(err) },
      "reconcileAgent before killAgentGroup failed — using stale recorded PIDs");
  }
  // EXISTING flow unchanged
  const pids = this.listForAgent(agentName);
  if (pids.length === 0) return;
  await this.killPids(pids, graceMs, "agent-disconnect");
  this.entries.delete(agentName);
  for (const pid of pids) this.cmdlines.delete(pid);
}
```

**Cycle hazard:** `reconcileAgent` MUST NOT call `killAgentGroup` (would recurse). Reconciler only mutates tracker state via `updateAgent` / `replaceMcpPids` / `unregister`. Pin in test.

### Anti-Patterns to Avoid

- **Reconciler that mutates tracker state during iteration of `getRegisteredAgents()`:** copy the keys first or build a mutation list and apply after iteration. The reconciler may call `unregister` for `agent-gone`; iterating the live map then deleting from it is undefined behavior in JS Maps when entries are deleted mid-iteration.
- **Calling `tracker.register` twice during agent.start race:** TRACK-02's polling resolves at most once, but if the polled wait succeeds AND the reconciler tick fires concurrently for an agent already registered, both could call `register`. Solution: reconciler uses `updateAgent`/`replaceMcpPids` (not `register`) — only TRACK-02's initial bootstrap calls `register`. Verify in code review.
- **Renaming `mcp-status` to mean tracker inspection:** breaks existing operator workflows AND breaks the Phase 85 → Phase 94 cross-renderer parity tests (`probe-renderer.ts` consumers). Use a new command name.
- **Reading `/proc/{pid}/environ` to enrich the snapshot output:** would leak resolved op:// secrets. The substrate already enforces this (`proc-scan.ts:18` security comment). Reconciler MUST follow the same allowlist.
- **Logging on every reconcile cycle:** journal blow-up on busy hosts. Diff-only logging is locked.
- **Per-agent reconciler intervals:** explicitly deferred. Use the existing 60s tick.
- **Pulling cmdlines via `getRegisteredAgents()` + IPC every CLI invocation:** the snapshot read is O(M) /proc reads where M is total MCP PIDs across all agents (~10-30 fleet-wide). Acceptable. Don't cache; the data is staleness-sensitive.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Liveness check | `pidof` shell-out, `/proc/{pid}` exists check | `process.kill(pid, 0)` | Single syscall, atomic, idiomatic. The /proc-stat read is for richer info (cmdline, ppid); a bare liveness check needs one syscall, not three. |
| Snapshot diff | Deep-equal library | Inline 4-line array compare | The shape is fixed: `{ claudePid, mcpPids[] }`. A general deep-equal pulls in a dep for one comparison. |
| Polled wait with cancel | `p-retry` / `async-retry` npm | `for` loop with `setTimeout` | 6 attempts, fixed delay, no backoff curve, no jitter. A loop is shorter than the import. |
| CLI subcommand wiring | Custom argv parser | Existing `commander` 12.x | Every other `clawcode` subcommand uses commander. Pattern locked. |
| IPC method registration | New socket route | Extend `routeMethod` switch in `daemon.ts:5285+` | Single-source dispatch matches every other IPC method (40+ existing cases). |
| ASCII table formatting | `cli-table` / `cli-table3` | Manual `padEnd` / `padStart` | Existing `mcp-status.ts` and `threads.ts` both do this manually. Match the local pattern. |

**Key insight:** Phase 999.15 has zero "should I use a library" questions. Every primitive is already in the tree from 999.14 substrate. The work is wiring + tests, not deciding what to build with.

## Common Pitfalls

### Pitfall 1: `mcp-status` Naming Collision
**What goes wrong:** Implementer reads CONTEXT.md ("`clawcode mcp-status [-a <agent>]`"), opens `src/cli/commands/mcp-status.ts`, sees an existing 280-line implementation, panics — either rewrites it (breaks Phase 85 + Phase 94 cross-renderer tests) OR adds a `--pids` flag that returns a totally different shape (operator confusion + test surface explosion).
**Why it happens:** CONTEXT.md was drafted without checking the existing CLI directory. Phase 85 shipped `mcp-status` for per-server *readiness probes*; Phase 94 extended it with capability-probe columns. Renaming or repurposing is high-cost.
**How to avoid:** Use a NEW command name. Recommended: `clawcode mcp-tracker [-a <agent>]` or `clawcode mcp-pids [-a <agent>]`. Document the rationale in the plan. Researcher recommends `mcp-tracker` because it parallels `mcp-status` (same `mcp-` namespace) and the verb "tracker" matches the underlying class name (`McpProcessTracker`).
**Warning signs:** Any plan that says "extend `src/cli/commands/mcp-status.ts`" — STOP, re-read this pitfall.

### Pitfall 2: Tracker `register` Signature Drift
**What goes wrong:** TRACK-02's polled discovery now produces a `claudePid` it must hand to the tracker. If `register(agentName, mcpPids)` (current signature) is left alone, the claudePid is lost; the reconciler can't tell "stale-claude" from "missing-mcps".
**Why it happens:** Current 999.14 tracker only stores MCP PIDs, not the claude PID — discovery happened upstream and was thrown away after registration.
**How to avoid:** Change `register` signature to `register(agentName, claudePid, mcpPids)`. Update the ONE caller in `session-manager.ts:750`. Update tests in `process-tracker.test.ts`.
**Warning signs:** Reconciler logs always show `oldClaudePid: 0` or `oldClaudePid: undefined`; reason field is always `missing-mcps`, never `stale-claude`.

### Pitfall 3: Reconciler-vs-Polling Race for the Same Agent
**What goes wrong:** Agent A is mid-startAgent; TRACK-02 polled wait is on attempt 3 of 6. Concurrently, the 60s reaper tick fires; reconciler walks `getRegisteredAgents()` — agent A is NOT registered yet (TRACK-02 hasn't returned). Reconciler skips A. Then on next tick, reconciler sees stale state.
**Why it happens:** Polled discovery is fire-and-forget per `void (async () => { ... })()` pattern. There's no "registration in progress" state. The reconciler can only see registered agents, not pending ones.
**How to avoid:** This is **acceptable** behavior — the reconciler catches drift on the NEXT tick after registration completes. Total worst-case staleness: 60s (one reconciler tick) + 30s (polling timeout) = 90s. Document in code comments.
**Warning signs:** Operators see "tracker shows 0 MCPs for agent X" in `mcp-tracker` CLI within 60s of agent.start — expected; resolves on next tick.

### Pitfall 4: Reconciler Mutates Map During Iteration
**What goes wrong:** Reconciler iterates `tracker.getRegisteredAgents()`, finds an agent whose claude PID is dead AND new claude PID is null (agent fully gone), calls `tracker.unregister(name)` mid-iteration. JS Map iterator behavior: deleted entries during iteration are skipped (spec-compliant) but NEW entries inserted are visited. With concurrent TRACK-02 registration, this could miss agents.
**Why it happens:** Maps are mutable; the iterator returned by `entries()` reflects the live state, not a snapshot.
**How to avoid:** Snapshot the keys first: `const names = Array.from(tracker.getRegisteredAgents().keys())`. Iterate the snapshot. Reconcile each. Now mutations during iteration are safe.
**Warning signs:** Reconciler logs occasionally skip agents when many start/stop in a tight window.

### Pitfall 5: `isPidAlive` Treats EPERM As Dead
**What goes wrong:** Implementer writes `try { process.kill(pid, 0); return true; } catch { return false; }` — silently treats EPERM (process exists, we can't signal) as dead. On Linux, EPERM is rare for same-UID processes BUT can occur for setuid binaries (e.g., a /usr/bin/sudo child). Treating as dead causes the reconciler to flag a live agent as `agent-gone` and unregister it.
**Why it happens:** `process.kill` throws on both ESRCH and EPERM; naive catch-all loses the distinction.
**How to avoid:** Distinguish via `e.code`: ESRCH → false (truly gone), EPERM → true (alive, just can't signal — sufficient for our purposes). Code is in Pattern 3 above.
**Warning signs:** Reconciler logs `reason: "agent-gone"` for clearly-running agents after a daemon-uid-change event (rare in prod, common in dev under sudo).

### Pitfall 6: Polled Discovery Doesn't Abort on Disposal
**What goes wrong:** TRACK-02 starts polling for agent A. User runs `clawcode stop A` 10s later. Polling continues, succeeds at 25s, registers A's old PIDs in the tracker — but `stopAgent` already called `unregister` 15s ago. The tracker now has zombie state for a stopped agent.
**Why it happens:** `void (async () => { ... })()` has no cancellation; the loop runs to completion.
**How to avoid:** Check `this.sessions.has(name)` before each `setTimeout` resolves. If the handle is gone, abort. Code in Pattern 2.
**Warning signs:** `mcp-tracker` CLI shows registered agents that are no longer in `clawcode status`.

### Pitfall 7: `discoverClaudeSubprocessPid` minAge Filter Doesn't Exist Yet
**What goes wrong:** Implementer writes `discoverClaudeSubprocessPid(daemonPid, { minAge: 5 })`, build fails — current signature is `(daemonPid: number) => Promise<number | null>` (no options object).
**Why it happens:** 999.14 didn't need a minAge filter for this function (the orphan reaper has its own); 999.15 does, for the polled-wait race-safety.
**How to avoid:** Extend the function signature. Pattern: optional second arg `(daemonPid: number, opts?: { minAge?: number }) => Promise<number | null>`. Inside, if `opts.minAge` is set, compare `procAgeSeconds(...)` against the threshold and skip younger procs. Reuse the same helpers as `scanForOrphanMcps`. ~5 lines added.
**Warning signs:** TS compile error at the polled-wait callsite.

```typescript
export function isPidAlive(pid: number): boolean {
  if (pid <= 0) return false;
  try {
    process.kill(pid, 0);
    return true;
  } catch (err: unknown) {
    const e = err as NodeJS.ErrnoException;
    if (e.code === "ESRCH") return false;
    if (e.code === "EPERM") return true;
    throw err;
  }
}
```

### Reconciler core (TRACK-01 + TRACK-04)
```typescript
// Source: derived from CONTEXT.md <specifics> + diff-logging from Pattern 4
// Add to: src/mcp/reconciler.ts
import type { Logger } from "pino";
import {
  discoverClaudeSubprocessPid,
  discoverAgentMcpPids,
  isPidAlive,
} from "./proc-scan.js";
import type { McpProcessTracker } from "./process-tracker.js";

type ReconcileReason = "stale-claude" | "missing-mcps" | "agent-restart" | "agent-gone";

export interface ReconcileArgs {
  readonly tracker: McpProcessTracker;
  readonly daemonPid: number;
  readonly log: Logger;
}

export async function reconcileAllAgents(args: ReconcileArgs): Promise<void> {
  // Snapshot keys to avoid mutation-during-iteration (Pitfall 4)
  const names = Array.from(args.tracker.getRegisteredAgents().keys());
  for (const name of names) {
    try {
      await reconcileAgent(name, args);
    } catch (err) {
      args.log.error(
        { component: "mcp-tracker", agent: name, err: String(err) },
        "reconcileAgent failed — continuing",
      );
    }
  }
}

export async function reconcileAgent(name: string, args: ReconcileArgs): Promise<void> {
  const entry = args.tracker.getRegisteredAgents().get(name);
  if (!entry) return;
  const before = { claudePid: entry.claudePid, mcpPids: [...entry.mcpPids] };
  let newClaudePid = entry.claudePid;
  let reason: ReconcileReason | null = null;

  if (!isPidAlive(entry.claudePid)) {
    const discovered = await discoverClaudeSubprocessPid(args.daemonPid, { minAge: 10 });
    if (discovered === null) {
      args.tracker.unregister(name);
      args.log.warn(
        {
          component: "mcp-tracker", action: "reconcile", agent: name,
          oldClaudePid: before.claudePid, oldMcpCount: before.mcpPids.length,
          reason: "agent-gone" satisfies ReconcileReason,
        },
        "tracker entry dropped — claude proc gone",
      );
      return;
    }
    args.tracker.updateAgent(name, discovered);
    newClaudePid = discovered;
    reason = "stale-claude";
  }

  if (reason === "stale-claude" || before.mcpPids.length === 0) {
    const mcpPids = await discoverAgentMcpPids(newClaudePid, args.tracker.patterns);
    args.tracker.replaceMcpPids(name, mcpPids);
    if (reason === null) reason = "missing-mcps";
    const after = { claudePid: newClaudePid, mcpPids: [...mcpPids] };
    if (snapshotsDiffer(before, after)) {
      args.log.warn(
        {
          component: "mcp-tracker", action: "reconcile", agent: name,
          oldClaudePid: before.claudePid, newClaudePid: after.claudePid,
          oldMcpCount: before.mcpPids.length, newMcpCount: after.mcpPids.length,
          reason,
        },
        "tracker state reconciled",
      );
    }
  }
}

function snapshotsDiffer(
  a: { claudePid: number; mcpPids: readonly number[] },
  b: { claudePid: number; mcpPids: readonly number[] },
): boolean {
  if (a.claudePid !== b.claudePid) return true;
  if (a.mcpPids.length !== b.mcpPids.length) return true;
  const x = [...a.mcpPids].sort();
  const y = [...b.mcpPids].sort();
  for (let i = 0; i < x.length; i++) if (x[i] !== y[i]) return true;
  return false;
}
```

## Open Questions

1. **Where is `bootTimeUnix` available for the polled-wait `minAge` calculation?**
   - What we know: `bootTimeUnix` is read once at daemon boot (`daemon.ts:1593`) and passed into the tracker constructor (`McpProcessTrackerDeps`). The orphan reaper consumes it via `args.bootTimeUnix`.
   - What's unclear: `discoverClaudeSubprocessPid` doesn't currently take `bootTimeUnix` — it doesn't filter by age. Adding `minAge` requires either (a) passing `bootTimeUnix` + `clockTicksPerSec` as additional args, or (b) reading them on demand inside the function (extra syscall per call).
   - Recommendation: Option (a) — add to the opts object: `{ minAge?: number; bootTimeUnix?: number; clockTicksPerSec?: number }`. Callers in `daemon.ts` and `session-manager.ts` already have these in scope (or can store them on the SessionManager). Don't read /proc/stat per call.

2. **Should `pruneDeadPids` log a per-PID line for each prune?**
   - What we know: TRACK-04 says diff-based, one log line per agent per cycle. `pruneDeadPids` is called by the reconciler (or potentially by `killAgentGroup`) — its individual prune events are absorbed into the agent-level reconcile log.
   - What's unclear: If an operator runs `clawcode mcp-tracker -a fin-acquisition` and sees 3/8 MCPs alive, they may want to know WHICH 5 are dead.
   - Recommendation: `mcp-tracker-snapshot` IPC returns alive/total counts; the verbose mode (`-a` flag) should also surface dead PIDs in a separate column. Don't surface in logs (would defeat diff-based logging); surface in the read-only CLI.

3. **`bootTimeUnix` staleness on a multi-day-running daemon?**
   - What we know: `/proc/stat` btime is the kernel boot time. It's invariant for the life of the kernel (unchanged across daemon restarts, only changes on machine reboot).
   - What's unclear: Reading it once at daemon boot and caching is correct. Confirmed.
   - Recommendation: No action — read once, cache, reuse. The 999.14 substrate already does this.
