# Phase 999.12: Cross-agent IPC channel delivery + heartbeat inbox timeout — Research

**Researched:** 2026-04-30
**Domain:** Daemon dispatch + heartbeat runtime — TypeScript / Node 22 / vitest
**Confidence:** HIGH

## Summary

The CONTEXT.md framing is partially out-of-date. **Pillar A's mirror flag (`mirror_to_target_channel`) already exists end-to-end** and ships through the `ask-agent` MCP tool → daemon IPC handler → `handleAskAgentIpc` (`src/manager/daemon-ask-agent-ipc.ts`). It works as **embed-formatted webhook posts** of both the prompt and the response, NOT as plain-text-via-`triggerDeliveryFn`. Two real gaps remain: (1) the response mirror is silently skipped when the target agent has **no webhook configured** (no bot-direct fallback), and (2) the production trace at 09:14:57 shows Admin Clawdy did NOT pass `mirror_to_target_channel: true` — so the existing mirror path simply was not triggered. The phase needs to decide whether to add a bot-direct fallback (matches CONTEXT.md's "same as triggerDeliveryFn" instruction) or change a default.

**Pillar B is straightforward.** The heartbeat inbox check uses `config.checkTimeoutSeconds` (default 10s) because it does not set a per-check `timeout` field. Add `defaults.heartbeatInboxTimeoutMs` and read it as a per-check override; add HB-02 by calling `sessionManager.hasActiveTurn(agentName)` (already exposed publicly via SerialTurnQueue.hasInFlight) at the top of the check and short-circuiting to a `skipped` healthy result.

**Primary recommendation:** Extend `handleAskAgentIpc` to add a bot-direct fallback for the response mirror when the target has no webhook — matching the `triggerDeliveryFn` shape at `daemon.ts:2200`. Bump `defaults.checkTimeoutSeconds` is the wrong knob (it affects every check); instead add a per-check `timeout` on `inboxCheck` driven by a new `defaults.heartbeatInboxTimeoutMs` field. Wire HB-02 directly inside `inbox.ts` using the already-public `sessionManager.hasActiveTurn(agentName)`.

### Pattern 1: triggerDeliveryFn closure (the canonical reference shape)
**What:** A `(targetAgent, response) => Promise<void>` closure that posts plain text to the agent's bound Discord channel via webhook with bot-direct fallback. Captures mutable refs (`webhookManagerRef`, `botDirectSenderRef`) so the closure can be constructed BEFORE its dependencies exist.
**When to use:** Any time output needs to land on the agent's bound channel from outside the discord-bridge live-stream path (cron output, async replies, mirrored responses).
**Example (trimmed; full source `src/manager/daemon.ts:2200-2248`):**

```typescript
// Source: src/manager/daemon.ts:2200
const triggerDeliveryFn: TriggerDeliveryFn = async (targetAgent, response) => {
  const cfg = manager.getAgentConfig(targetAgent);
  if (!cfg) { log.warn(...); return; }
  const channelId = cfg.channels?.[0];
  if (!channelId) { log.warn(...); return; }

  const MAX = 2000;
  const truncated = response.length > MAX
    ? response.slice(0, MAX - 3) + "..."
    : response;

  const wm = webhookManagerRef.current;
  if (wm && wm.hasWebhook(targetAgent)) {
    await wm.send(targetAgent, truncated);
    return;
  }
  const bot = botDirectSenderRef.current;
  if (bot) {
    await bot.sendText(channelId, truncated);
    return;
  }
  log.warn({ targetAgent, channelId }, "no sender available — skipping");
};
```

### Pattern 2: Heartbeat per-check timeout override
**What:** A `CheckModule` may set `readonly timeout?: number` (seconds). The runner uses `(check.timeout ?? config.checkTimeoutSeconds) * 1000` (`src/heartbeat/runner.ts:223`).
**When to use:** When a single check has different timing characteristics from the rest of the fleet — exactly the inbox-check case.
**Example:**
```typescript
// Source: src/heartbeat/runner.ts:223 (already in place)
const timeoutMs = (check.timeout ?? this.config.checkTimeoutSeconds) * 1000;
const result = await this.executeWithTimeout(check, context, timeoutMs);
```

To wire HB-01 the inbox check needs to read the config — but `CheckModule` is a static export. Two options:
- **Option A:** Add `readonly timeout` as a static value on `inboxCheck` (e.g. `timeout: 60`). Loses per-deployment override.
- **Option B (recommended):** Read the override at execute time from `context.config` (which is `HeartbeatConfig`). Extend `HeartbeatConfig` with optional `inboxTimeoutMs`. Inside `execute`, return early via Promise.race fast-fail wrapper, OR re-architect runner so per-check timeout can come from config. **Simpler:** extend `HeartbeatConfig.checkTimeoutSeconds` lookup to include a per-check override map.

**Recommendation:** Add a new optional field on `HeartbeatConfig`: `readonly inboxTimeoutMs?: number`. In `runner.ts` change the timeout-resolution line from:

```typescript
const timeoutMs = (check.timeout ?? this.config.checkTimeoutSeconds) * 1000;
```

to:

```typescript
const timeoutMs = check.name === "inbox" && this.config.inboxTimeoutMs
  ? this.config.inboxTimeoutMs
  : (check.timeout ?? this.config.checkTimeoutSeconds) * 1000;
```

Or — cleaner — generalize via a `perCheckTimeouts: Record<string, number>` map. But for ONE check the special-case is fine and easier to revert.

### Anti-Patterns to Avoid
- **Adding a deliveryFn parameter to `dispatchTurn`:** breaks the single-responsibility shape (RPC vs delivery). The IPC handler is the right layer because it already owns the mirror policy. CONTEXT.md says "extend `options` with `mirror_to_target_channel`" — but the `mirror_to_target_channel` lookup ALREADY happens at the IPC edge before `dispatchTurn` is called. No SessionManager change needed.
- **Bumping fleet-wide `checkTimeoutSeconds`:** turns a per-check fix into a fleet-wide regression — every other check gets sluggish failure detection.
- **Logging at info level for HB-02 skips:** every cross-agent turn would log a skip line per heartbeat tick. Use `log.debug` (level 20) so it's gated behind `LOG_LEVEL=debug`.
- **Failing the ask on mirror failure:** the existing handler already wraps mirror in best-effort try/catch (good); maintain that contract for the new bot-direct fallback.

## Common Pitfalls

### Pitfall 1: Stale snapshot of CONTEXT.md vs running code
**What goes wrong:** Plan a "new" mirror flag → discover it already exists → wasted plan iteration.
**Why it happens:** CONTEXT.md was drafted from an older daemon state; Phase 999.2 Plan 03 already shipped `mirror_to_target_channel` end-to-end on 2026-04 (commit history shows `daemon-ask-agent-ipc.ts` exists with full DI handler).
**How to avoid:** Read the existing `handleAskAgentIpc` BEFORE planning. The phase is now an ENHANCEMENT (add bot-direct fallback) not a NEW FEATURE.
**Warning signs:** Plan tasks that say "create askAgentSchema" or "wire mirror_to_target_channel into MCP tool" — both are already done.

### Pitfall 2: Embed vs plain-text inconsistency
**What goes wrong:** Mirror via webhook posts an embed (rich, with author); fallback via bot-direct posts plain text. Visual inconsistency in the channel.
**Why it happens:** `WebhookManager.sendAsAgent` requires an `EmbedBuilder`; `BotDirectSender.sendText` takes a string.
**How to avoid:** Acceptable inconsistency — operators see the embed when webhook present (the common case), plain text only when webhook absent (rare; only some agent configs lack a webhook). Document the behavior. Alternatively, use `WebhookManager.send(targetAgent, plainText)` (which exists per `triggerDeliveryFn` line 2236) for ALL paths and skip embeds entirely.
**Warning signs:** Tests assert different output formats for webhook-present vs webhook-absent cases — confirm the assertion matches the chosen contract.
**Decision needed by planner:** Option A — preserve existing embed contract for webhook path (asymmetric output); Option B — switch the response mirror to plain-text everywhere (matches `triggerDeliveryFn`, matches CONTEXT.md non-negotiable verbatim, but visually changes existing webhook output).

### Pitfall 3: Heartbeat tick race with turn lifecycle
**What goes wrong:** `hasActiveTurn` returns false at check entry, then the agent starts a new turn between the check and `dispatchTurn` call inside the inbox processing loop.
**Why it happens:** Heartbeat tick is async; turns can begin at any moment.
**How to avoid:** The race is benign — even if the turn starts mid-check, the dispatchTurn call will queue (SerialTurnQueue's queued slot) or throw QUEUE_FULL. The current behavior already retries on next tick (failed messages aren't markProcessed). The HB-02 skip is a heuristic, not a hard guarantee.
**Warning signs:** Tests that assert HB-02 NEVER allows a dispatch during turn — drop them; assert "best-effort skip when in-flight" instead.

### Pitfall 4: HB-01 timeout regression for legacy mode
**What goes wrong:** Bumping inbox timeout to 60s means legacy-mode (when InboxSource is NOT active) takes 60s to detect a stuck inbox.
**Why it happens:** Same check is used in primary AND reconciler modes; one timeout serves both.
**How to avoid:** InboxSource is the production primary path (Phase 61 TRIG-04); reconciler mode uses 120s staleness anyway. 60s timeout in either mode is fine. If concerned, parameterize with two configs — but YAGNI.
**Warning signs:** Operators report sluggish inbox-check critical alerts after deploy.

### Pitfall 5: Forgetting test fixtures for `hasActiveTurn`
**What goes wrong:** Existing inbox-check tests don't pass a sessionManager mock that supports `hasActiveTurn` → tests start failing with `TypeError: sessionManager.hasActiveTurn is not a function`.
**Why it happens:** Test fixtures were built before HB-02.
**How to avoid:** Audit `src/heartbeat/__tests__/` for inbox-check fixtures; add `hasActiveTurn: vi.fn().mockReturnValue(false)` to all sessionManager mocks.
**Warning signs:** Test suite goes red on changes to inbox.ts even though logic is sound.

### Pitfall 6: Webhook ordering — prompt embed posts before dispatch
**What goes wrong:** Existing handler posts the prompt embed BEFORE awaiting `dispatchTurn`. If `dispatchTurn` fails, the prompt embed is in the channel with no response — confusing.
**Why it happens:** Pre-dispatch mirror at line 172 is by design for "prompt visibility" but couples to dispatch success.
**How to avoid:** Out of scope for this phase (existing behavior). Don't change it.
**Warning signs:** Tests assert "no prompt embed if dispatch fails" — those would break the existing contract.

### Pitfall 7: BotDirectSender available but not yet constructed
**What goes wrong:** `botDirectSenderRef.current` is null at boot before the discord bridge connects → mirror fallback no-ops with a warn line.
**Why it happens:** Mutable ref pattern means the ref exists before the value.
**How to avoid:** Pass `botDirectSenderRef` (the ref, not the value) into `handleAskAgentIpc` deps. Read `.current` at call time, not at deps construction. Test with both states.
**Warning signs:** First-boot mirrors silently disappear from the channel; subsequent ones work.

## Open Questions

1. **Should the bot-direct fallback post plain text or a Discord embed?**
   - What we know: Webhook path uses `buildAgentMessageEmbed` (rich, author-attributed). `BotDirectSender.sendText` takes a string. Bot can post embeds via `bot.sendEmbed(channelId, embed)` (need to verify the surface).
   - What's unclear: Whether visual inconsistency between the two paths is acceptable.
   - Recommendation: **Plain text via `BotDirectSender.sendText`**. Matches CONTEXT.md non-negotiable verbatim ("SAME webhook→bot fallback as triggerDeliveryFn"). The webhook embed remains the rich/common path; bot-direct is the rare fallback.

2. **Should HB-02 emit a debug log line per skip?**
   - What we know: CONTEXT.md says "level=20 debug log line for traceability".
   - What's unclear: pino's `level=20` is debug; the daemon's default log level is info (30). Skipping operators won't see the line in production unless they raise the level.
   - Recommendation: Yes — emit `log.debug` with `{ agent, reason: "active-turn" }`. Operators with `LOG_LEVEL=debug` get the trail; default deployments stay quiet.

3. **Should `inboxTimeoutMs` apply only when InboxSource is active (reconciler mode) or always?**
   - What we know: Production has InboxSource active (Phase 61 TRIG-04). Reconciler mode only processes >120s-stale messages.
   - What's unclear: Does "primary mode" still ship in the field anywhere?
