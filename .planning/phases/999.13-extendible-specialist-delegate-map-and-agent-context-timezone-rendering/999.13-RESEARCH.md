## Summary

Phase 999.13 bundles two surgical "agent context hygiene" changes that touch the same daemon-edge code path:

1. **Pillar A (DELEG-01..04)**: A new `delegates: z.record(string,string)` schema field on `agentSchema` with a `superRefine` cross-field validator at `configSchema`, plus a session-boot prompt injection that renders a `## Specialist Delegation` block alphabetized by specialty key when non-empty (omitted when empty for byte-stable prompt-cache).

2. **Pillar B (TZ-01..05)**: A new `renderAgentVisibleTimestamp(date, tz)` helper using `Intl.DateTimeFormat` that converts UTC ISO strings to canonical `YYYY-MM-DD HH:mm:ss ZZZ` at the agent-visible serialization boundary, plus a `defaults.timezone` config knob with a 3-step resolution chain. Internal storage stays UTC.

Both pillars are surgical (~50 lines each), zero-dependency (Intl is built-in to Node 22), additive-optional (back-compat with existing 15-agent fleet), and follow established codebase conventions (Phase 94 D-10 directive rail, Phase 62 superRefine, Phase 78 CONF-01 precedence).

**Primary recommendation:** For Pillar A, factor the canonical directive text into a single exported `DELEGATES_DIRECTIVE_HEADER` + `DELEGATES_DIRECTIVE_FOOTER` constant pair in `src/config/schema.ts` next to `DEFAULT_SYSTEM_PROMPT_DIRECTIVES`, render via a new pure helper `renderDelegatesBlock(delegates)` in `src/config/loader.ts` next to `renderSystemPromptDirectiveBlock`, thread through `ContextSources.delegatesBlock?: string`, and append AFTER `toolDefinitions` in `assembleContext()`. For Pillar B, put `renderAgentVisibleTimestamp` and `resolveAgentTimezone` in `src/shared/agent-visible-time.ts` (no upstream deps so it can be imported anywhere), and convert each of the three confirmed agent-visible `ts` sites surgically (see Sites table).

### Pattern 1: Additive-Optional Schema Field with Resolver Fallback
**What:** New `agentSchema` field is `.optional()`; resolver in loader.ts threads through with explicit `undefined` semantics.
**When to use:** Adding any per-agent config field that must not break existing yaml.
**Example:**
```ts
// src/config/schema.ts (agentSchema, append next to systemPromptDirectives)
delegates: z.record(z.string().min(1), z.string().min(1)).optional(),

// src/config/schema.ts (configSchema.superRefine, after the existing soul/soulFile mutex check)
const agentNames = new Set(cfg.agents.map((a) => a.name));
for (const agent of cfg.agents) {
  if (!agent.delegates) continue;
  for (const [specialty, target] of Object.entries(agent.delegates)) {
    if (!agentNames.has(target)) {
      ctx.addIssue({
        code: z.ZodIssueCode.custom,
        path: ["agents"],
        message: `agent "${agent.name}": delegates["${specialty}"] points to unknown agent "${target}". Configure that agent or remove this delegate entry.`,
      });
    }
  }
}
```

### Pattern 2: Pure Renderer Co-Located with Resolver
**What:** Schema → resolver (loader.ts) → renderer (loader.ts) → injection point (session-config.ts → context-assembler.ts).
**When to use:** Per-agent prompt content that must be deterministic (alphabetical sort for prompt-cache stability).
**Example:**
```ts
// src/config/loader.ts (next to renderSystemPromptDirectiveBlock)
export const DELEGATES_DIRECTIVE_HEADER =
  "## Specialist Delegation\nFor tasks matching a specialty below, delegate via the spawn-subagent-thread skill:";
export const DELEGATES_DIRECTIVE_FOOTER =
  "Verify the target is at opus/high before delegating; if mismatch, surface to operator and stop. The subthread posts its summary back to your channel when done.";

export function renderDelegatesBlock(
  delegates: Readonly<Record<string, string>> | undefined,
): string {
  if (!delegates) return "";
  const keys = Object.keys(delegates);
  if (keys.length === 0) return "";
  // Alphabetical sort — required for prompt-cache hash stability (REG-DETERMINISTIC).
  keys.sort();
  const lines = keys.map((k) => `- ${k} → ${delegates[k]}`);
  return [
    DELEGATES_DIRECTIVE_HEADER,
    ...lines,
    DELEGATES_DIRECTIVE_FOOTER,
  ].join("\n");
}
```

### Pattern 3: Pure TZ Helper with Resolution Chain
**What:** Single helper accepts both `Date` and `ISO string`; resolves TZ from optional config → env → host once.
**Example:**
```ts
// src/shared/agent-visible-time.ts
const HOST_TZ = (() => {
  try {
    return Intl.DateTimeFormat().resolvedOptions().timeZone;
  } catch {
    return "UTC";
  }
})();

export function resolveAgentTimezone(
  configTz: string | undefined,
): string {
  return configTz ?? process.env.TZ ?? HOST_TZ;
}

export function renderAgentVisibleTimestamp(
  date: Date | string,
  tz?: string,
): string {
  const d = typeof date === "string" ? new Date(date) : date;
  if (Number.isNaN(d.getTime())) return "invalid date";
  const timeZone = tz ?? HOST_TZ;
  let parts: Intl.DateTimeFormatPart[];
  try {
    parts = new Intl.DateTimeFormat("en-US", {
      year: "numeric",
      month: "2-digit",
      day: "2-digit",
      hour: "2-digit",
      minute: "2-digit",
      second: "2-digit",
      hour12: false,
      timeZone,
      timeZoneName: "short",
    }).formatToParts(d);
  } catch {
    // Bad IANA TZ — fall back to UTC.
    return renderAgentVisibleTimestamp(d, "UTC");
  }
  const m: Record<string, string> = {};
  for (const p of parts) m[p.type] = p.value;
  // Node 22 emits "24" for midnight in some locales; normalize to "00".
  const hh = m.hour === "24" ? "00" : m.hour;
  return `${m.year}-${m.month}-${m.day} ${hh}:${m.minute}:${m.second} ${m.timeZoneName}`;
}
```

### Anti-Patterns to Avoid
- **Inlining the directive text at the call site.** The canonical text MUST be a single exported constant pinned by static-grep tests. If two agents render slightly different text the tests catch it; if it's inlined, drift is invisible.
- **Empty-but-present block.** When `delegates` is `{}` or unset, render exactly `""` — NOT a header + zero bullets. Tests pin byte-stable hash continuity for prompt-cache. (Same pattern as the empty `systemPromptDirectives` short-circuit at `loader.ts:675`.)
- **Inlining `new Intl.DateTimeFormat(...)` at every TS site.** Use the helper. The helper centralizes the format constant, the host-TZ resolution, the bad-input fallback, and the test surface. Inlining means 4 different formatters with 4 different bugs.
- **Mutating `process.env.TZ` in tests.** That mutates global state and bleeds across tests. Use the helper's `tz` parameter to inject TZ explicitly in tests.
- **Validating `delegates` targets at session-config / runtime instead of `configSchema.superRefine`.** Catches the error too late — operator might already have a half-booted fleet by then. Schema-time fail-fast is the codebase convention (pattern from soul/soulFile mutex).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| TZ-aware timestamp formatting | A custom UTC offset table or DST detection algorithm | `Intl.DateTimeFormat` with `timeZoneName: "short"` | Node 22 ICU handles DST + TZ abbreviation correctly. Custom code will get spring-forward/fall-back wrong (and CONTEXT.md's own DST fixture table demonstrates that — see Pitfall 1). |
| TZ abbreviation lookup (PDT/PST/EST) | A hardcoded `{ "America/Los_Angeles": ["PST","PDT"] }` map | `formatToParts` `timeZoneName` part | The Intl ICU bundle returns the locale-correct abbreviation for the given instant — no fall-back-day ambiguity. |
| Cross-agent name validation | A separate config-load pass | `configSchema.superRefine` | Already the project's pattern (see soul/soulFile mutex at schema.ts:1738). One pass, zod-native error UX. |
| Alphabetical-deterministic rendering | An ad-hoc loop over a Map | `Object.keys(record).sort()` | Plain object `Object.keys` has no defined order in ECMAScript spec for non-integer keys (engines do insertion-order in practice, but rely on explicit sort). Pattern from `resolveSystemPromptDirectives` (loader.ts:653). |

**Key insight:** Both pillars hit problems where Node's built-ins (`Intl`, `Object.keys + sort`) and the project's existing patterns (zod superRefine, additive-optional schema, ContextSources field threading) cover the entire surface. Custom code in either pillar is a smell.

## Common Pitfalls

### Pitfall 1: CONTEXT.md DST fixture row #5 is wrong (US 2026 fall-back is Nov 1, not Nov 2)
**What goes wrong:** The CONTEXT.md test fixture table claims `2026-11-02T08:30:00.000Z` → `2026-11-02 01:30:00 PDT`. That's wrong. US DST 2026 fall-back is on **November 1** (first Sunday), not November 2. By 2026-11-02T08:30:00Z, Los Angeles has already been on PST for 30 hours — so the actual local rendering is `2026-11-02 00:30:00 PST`, NOT `2026-11-02 01:30:00 PDT`.
**Why it happens:** The CONTEXT.md author appears to have used 2025 DST dates (where fall-back was Nov 2). In 2026, the first Sunday of November is the 1st.
**How to avoid:** Plan must replace the CONTEXT.md fixture row #5 with a corrected fixture. Options:
- **Option A (preferred):** Use 2025 dates throughout the test fixtures — then the CONTEXT.md table is correct as written. Spring-forward 2025 = Mar 9. Fall-back 2025 = Nov 2.
- **Option B:** Update the fixture to 2026's actual DST boundaries: spring-forward at Mar 8 02:00 PST → 03:00 PDT; fall-back at Nov 1 02:00 PDT → 01:00 PST. Replace row 3-4 with `2026-03-08T09:30:00Z` (01:30 PST, before transition) and `2026-03-08T10:30:00Z` (03:30 PDT, after transition). Replace row 5-6 with `2026-11-01T08:30:00Z` (01:30 PDT, before fall-back) and `2026-11-01T09:30:00Z` (01:30 PST, after fall-back).
- **Option C:** Keep CONTEXT.md fixtures as-is for rows 1-4 (which all happen to round-trip correctly), drop row 5, keep row 6 (which is correct because by Nov 2 LA is firmly on PST regardless of 2025/2026 difference). Add new row covering Nov 1 fall-back ambiguity.

Verified empirically via `node -e` script — see "Code Examples" below.

**Warning signs:** A test fails with `Expected '2026-11-02 01:30:00 PDT' but got '2026-11-02 00:30:00 PST'` — the runtime is right, the fixture is wrong.

### Pitfall 2: Prompt-cache hash invalidation from non-deterministic delegate render
**What goes wrong:** Object key iteration order isn't guaranteed by spec for non-integer keys; if `delegates` block isn't sorted, two daemon boots can produce different stable-prefix bytes and invalidate the SDK prompt cache turn-to-turn. Cost: every turn re-pays the full system-prompt token cost.
**Why it happens:** `for (const k of Object.keys(delegates))` without `.sort()` works in Node 22 (insertion-order in practice) but the spec doesn't require it; future engines or yaml parsers may differ.
**How to avoid:** Always `Object.keys(delegates).sort()` before render. Pattern already used in `resolveSystemPromptDirectives` at `loader.ts:653`. Test asserts byte-identical render across 100 random key orderings.
**Warning signs:** SDK reports cache-miss on every turn; `context_assemble` span shows different stable-prefix-hash values across boots.

### Pitfall 3: Threading agent TZ to bridge.ts at message-format time
**What goes wrong:** `formatDiscordMessage` is a pure function called at incoming-message-time inside `src/discord/bridge.ts`. It doesn't know the recipient agent's TZ — Discord's bridge dispatches one message to N agents bound to that channel.
**Why it happens:** Per-agent TZ resolution lives in session-config / per-agent context. The bridge formats the message ONCE before dispatch; if two agents have different TZs, the bridge can't render both.
**How to avoid:** Per CONTEXT.md "single-host single-TZ for now", all agents on this clawdy host share the host TZ. Resolve TZ ONCE at daemon startup (or once per format call from `Intl.DateTimeFormat().resolvedOptions().timeZone`) and pass into `formatDiscordMessage` as a constructor-time or call-time arg. If the future per-user TZ deferred item lands, this will need a refactor — flag in code comments. Pure function pattern works fine for single-TZ.
**Warning signs:** Tests that mock multiple agents with different TZs and expect different bridge outputs fail. (Out of scope per CONTEXT.md.)

### Pitfall 4: `Intl.DateTimeFormat` constructor cost in tight loops
**What goes wrong:** `new Intl.DateTimeFormat(...)` is not free — locale + ICU initialization. Calling it per-message at high message rates (Discord burst) adds up.
**Why it happens:** The naive helper constructs a new formatter every call.
**How to avoid:** Cache formatters by `timeZone` key in module scope. The format options are constant, so a `Map<string, Intl.DateTimeFormat>` keyed by IANA TZ name covers it. The cache is unbounded in theory but bounded in practice (one entry per configured TZ — usually 1 per host).
**Warning signs:** Heartbeat reports elevated CPU on `formatDiscordMessage`. (Optimization — consider during plan if benchmarks show it; not required for correctness.)

### Pitfall 5: Operator types unknown specialty key — schema accepts any string
**What goes wrong:** `delegates: { reseach: research }` (typo) parses fine — schema is `z.record(string, string)` with no key allowlist. The block renders `- reseach → research` and the LLM never matches it.
**Why it happens:** Per CONTEXT.md "free-form keys for extendibility, no enum lock-in" — typo-tolerant by design.
**How to avoid:** Document in plan that schema is intentionally permissive. The cost of locking down is "operator can't add `coding:` later without code change" — too high. Accept the tradeoff. (Optional mitigation: warn-log unrecognized specialty keys against an operator-curated lint list — out of scope for this phase.)
**Warning signs:** Operator runs `clawcode validate-config` and the parser accepts `delegates: { reseach: research }` silently. By design.

### Pitfall 6: `Intl.DateTimeFormat` with a bad IANA TZ string throws RangeError at construction
**What goes wrong:** `new Intl.DateTimeFormat("en-US", { timeZone: "Pacific/LosAngeles" })` (typo) throws `RangeError: Invalid time zone specified: Pacific/LosAngeles`. Crashes session boot if not caught.
**Why it happens:** Operator typos `defaults.timezone: "Pacific/LosAngeles"` (should be `America/Los_Angeles`).
**How to avoid:** Two layers:
- Schema layer (optional but nice): pre-validate the IANA TZ at parse time via try/catch around `new Intl.DateTimeFormat(undefined, { timeZone: tz })`. Emit a zod issue if it throws.
- Runtime layer (mandatory): `renderAgentVisibleTimestamp` wraps the format call in try/catch and falls back to UTC with a warn-log. This is the pattern in `agent-visible-time.ts` example above.
**Warning signs:** Daemon boot fails with RangeError stack trace from `formatDiscordMessage`. Any thrown error in the prompt path crashes the agent.

### Pitfall 7: `process.env.TZ` precedence vs. host TZ
**What goes wrong:** Node respects `process.env.TZ` when set — it changes what `new Date()` does internally and what `Intl.DateTimeFormat()` resolves to as host TZ. So `Intl.DateTimeFormat().resolvedOptions().timeZone` ALREADY honors `process.env.TZ`. The CONTEXT.md resolution chain "configTz → process.env.TZ → host" implies these are distinct, but the third already includes the second.
**Why it happens:** Node's TZ resolution is layered.
**How to avoid:** The chain in the helper as written (`configTz ?? process.env.TZ ?? HOST_TZ`) is fine because `HOST_TZ` is captured ONCE at module load time — if env.TZ is set then, both `process.env.TZ` and `HOST_TZ` end up pointing at the same string. If env.TZ is unset, `process.env.TZ` is `undefined` and we fall through to `HOST_TZ` (which uses ICU's host detection, e.g., `/etc/localtime`). Behavior matches CONTEXT.md intent.
**Warning signs:** Test that sets `process.env.TZ` AFTER module load and expects the helper to pick it up — fails. Document that the resolution captures host TZ at module-load time. Operators wanting to change TZ live should use `defaults.timezone`.

### Render canonical TZ-aware timestamp via `formatToParts`
```ts
// Source: MDN Intl.DateTimeFormat — formatToParts
// https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Intl/DateTimeFormat/formatToParts
function renderAgentVisibleTimestamp(date: Date | string, tz: string): string {
  const d = typeof date === "string" ? new Date(date) : date;
  const parts = new Intl.DateTimeFormat("en-US", {
    year: "numeric", month: "2-digit", day: "2-digit",
    hour: "2-digit", minute: "2-digit", second: "2-digit",
    hour12: false, timeZone: tz, timeZoneName: "short",
  }).formatToParts(d);
  const m: Record<string, string> = {};
  for (const p of parts) m[p.type] = p.value;
  const hh = m.hour === "24" ? "00" : m.hour;
  return `${m.year}-${m.month}-${m.day} ${hh}:${m.minute}:${m.second} ${m.timeZoneName}`;
}

// Verified empirical outputs (Node 22 LTS, ICU bundled):
// '2026-01-15T08:32:51.000Z' → '2026-01-15 00:32:51 PST'  // winter, standard
// '2026-04-30T18:32:51.000Z' → '2026-04-30 11:32:51 PDT'  // spring/summer, daylight
// '2026-03-09T09:30:00.000Z' → '2026-03-09 02:30:00 PDT'  // post-spring-forward
```
