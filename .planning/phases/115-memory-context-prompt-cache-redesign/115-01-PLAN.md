# Plan 115-01: Quick wins — excludeDynamicSections + memoryRetrievalTokenBudget wire + tag-filter at hybrid-RRF

## Goal

Land three independent <100-line changes that drop fin-acquisition's stable prefix by single-digit-percent immediately and prevent the pollution-feedback-loop in `<memory-context>`. Each change is low-risk, surgical, and behind a config knob (or default-on with explicit override). Ships before structural changes (115-03) so post-fix benchmarks isolate quick-win contributions.

## Sub-scopes covered

- Sub-scope 2 (roadmap line 822): set `systemPrompt.excludeDynamicSections: true` on the SDK options.
- Sub-scope 3 (roadmap line 824): wire dead `memoryRetrievalTokenBudget` config knob through `SessionManager.getMemoryRetrieverForAgent`.
- Sub-scope 4 (roadmap line 826): exclude memories tagged `session-summary`, `mid-session`, `raw-fallback` from per-turn `<memory-context>` injection by default.

## must_haves

- `grep -n "excludeDynamicSections" src/manager/session-adapter.ts` returns ≥1 match (the flag is now set).
- `grep -n "memoryRetrievalTokenBudget" src/manager/session-manager.ts` returns ≥1 match in the retriever closure (no longer dead).
- `grep -n "excludeTags\|excludeSourceTags" src/memory/memory-retrieval.ts` returns ≥1 match (tag-filter parameter exists).
- Default-applied tag exclusion list contains exactly: `session-summary`, `mid-session`, `raw-fallback` (verifiable by grep).
- After this plan ships, no agent's `<memory-context>` contains rows tagged `session-summary` / `mid-session` / `raw-fallback` (acceptance criterion line 905 of ROADMAP).
- All three changes are independently revertable behind config flags.

## Threat Model

| Threat | Vector | Severity | Mitigation |
|---|---|---|---|
| Setting `excludeDynamicSections: true` breaks the SDK shape Phase 52 locked | New SDK option may conflict with cached `claude_code` preset | LOW | Type signature already allows the field per `sdk-types.ts:73,90`; SDK ignores unknown SDKs return undefined; default-on with explicit revert path via config flag |
| Tag-filter accidentally drops legitimate user-curated `session-summary`-tagged manual entries | Operator manually tags a memory with `session-summary` | LOW | Per-agent `memoryRetrievalExcludeTags: []` override exposes opt-out; filter is defensive against pollution-loop, not strict |
| `getMemoryMetadata` N+1 query degrades retrieval latency | Per-row SQL lookup over RRF top-K | LOW | top-K is bounded ≤50; SQL is indexed on PK; sub-1ms per call; fan-out at most 50 lookups (≤50ms total) |
| Reducing `memoryRetrievalTokenBudget` 2000 → 1500 silently loses recall | First-chunk-always-passes guard hides this | MEDIUM | Bench label `pre-115-baseline` already captured pre-change; 115-09 closeout explicitly compares memory-recall scenario p50 across the budget change; per-agent override available |

## Notes

- All three changes are guarded behind config flags with explicit defaults. Reverting any single change is a config edit.
- The tag-filter list is the locked default per CONTEXT.md sub-scope 4. Operators can extend (e.g., add `archived-pending`) by setting `memoryRetrievalExcludeTags: ["session-summary", "mid-session", "raw-fallback", "archived-pending"]` in defaults or per-agent.
- Plan 115-08 measurement will pin whether the 1500-token cap is too tight by sampling memory-recall scenarios; if recall drops materially, 115-08 raises the default to 1750 or 2000.
