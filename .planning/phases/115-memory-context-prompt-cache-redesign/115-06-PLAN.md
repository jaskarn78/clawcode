# Plan 115-06: Embedding upgrade (bge-small-en-v1.5 + int8) + dual-write migration machinery

## Goal

Ship the **machinery** for migrating from `Xenova/all-MiniLM-L6-v2` (2019, MTEB ~56) to `BAAI/bge-small-en-v1.5` (Apache 2.0, MTEB ~64, same 384-dim, ONNX) with int8 quantization in sqlite-vec. Migration approach is dual-write + background batch re-embed (NOT one-shot, NOT lazy on-access) per D-08 lock. The wall-clock T+0 → T+7d → T+14d migration timeline runs in production AFTER deploy; this plan ships the code only.

The Phase 107 VEC-CLEAN-* cascade-delete invariant is preserved: deletes from `memories` cascade-delete from BOTH `vec_memories` AND `vec_memories_v2` atomically. Per-agent isolation per Phase 90 lock is unchanged — every per-agent migration step is its own DB transaction.

    ```ts
    /**
     * Phase 115 D-07 — int8 scalar quantization for sqlite-vec.
     *
     * Quantization scheme: per-vector min/max scalar quantization.
     * For each float32 vector v of length DIM:
     *   min_v = min(v), max_v = max(v)
     *   scale = (max_v - min_v) / 255
     *   quantized[i] = round((v[i] - min_v) / scale) - 128       // mapped to [-128, 127]
     *   stored: int8[DIM]; the dequant min/max recovered from the per-vector header bytes if needed.
     *
     * Recall loss < 2% per perf-caching-retrieval research §3 at 384-dim.
     * Storage: 4x reduction (4 bytes/dim → 1 byte/dim).
     */

    export function quantizeInt8(v: Float32Array): Int8Array {
      const n = v.length;
      let min = Infinity, max = -Infinity;
      for (let i = 0; i < n; i++) {
        if (v[i] < min) min = v[i];
        if (v[i] > max) max = v[i];
      }
      const scale = (max - min) / 255 || 1;
      const out = new Int8Array(n);
      for (let i = 0; i < n; i++) {
        const q = Math.round((v[i] - min) / scale) - 128;
        out[i] = q < -128 ? -128 : q > 127 ? 127 : q;
      }
      return out;
    }

    /**
     * For verification/diagnostics; queries inside sqlite-vec dequantize natively.
     */
    export function dequantizeInt8(q: Int8Array, scale: number, offset: number): Float32Array {
      const n = q.length;
      const out = new Float32Array(n);
      for (let i = 0; i < n; i++) {
        out[i] = (q[i] + 128) * scale + offset;
      }
      return out;
    }
    ```

    ```ts
    /**
     * Phase 115 D-08 — embedding v2 migration state machine.
     *
     * Phases:
     *  1. idle               — no migration in progress; reads/writes use v1 only.
     *  2. dual-write         — T+0 → T+7d. New writes embed BOTH v1 (MiniLM) and v2 (bge-small-int8).
     *                          Reads still use v1.
     *  3. re-embedding       — T+7d → T+14d. Background batch re-embed historical memories at 5% CPU.
     *                          Resumable across daemon restarts via last_cursor.
     *  4. re-embed-complete  — T+14d. All historical embeddings have v2 vectors. Ready for cutover.
     *  5. cutover            — Reads switch to v2; v1 column kept for 24h soak.
     *  6. v1-dropped         — v1 column dropped (post-24h-soak); migration complete.
     *  7. rolled-back        — emergency rollback to phase 1; v2 reads disabled, v2 column kept for re-attempt.
     */

    export type EmbeddingMigrationPhase =
      | "idle"
      | "dual-write"
      | "re-embedding"
      | "re-embed-complete"
      | "cutover"
      | "v1-dropped"
      | "rolled-back";

    export interface EmbeddingMigrationState {
      agent: string;
      phase: EmbeddingMigrationPhase;
      progressProcessed: number;
      progressTotal: number;
      lastCursor: string | null;
      startedAt?: string;
      completedAt?: string;
    }

      constructor(private db: Database, private agent: string) {}

      getState(): EmbeddingMigrationState { /* SELECT FROM migrations WHERE key=embeddingV2.<agent> */ }
      transition(toPhase: EmbeddingMigrationPhase): void { /* UPDATE phase, transactional */ }
      saveCursor(memoryId: string, processed: number, total: number): void { /* UPDATE last_cursor */ }
      currentReadVersion(): "v1" | "v2" {
        const phase = this.getState().phase;
        return phase === "cutover" || phase === "v1-dropped" ? "v2" : "v1";
      }
      currentWriteVersions(): Array<"v1" | "v2"> {
        const phase = this.getState().phase;
        switch (phase) {
          case "idle": case "rolled-back": return ["v1"];
          case "dual-write": case "re-embedding": case "re-embed-complete": return ["v1", "v2"];
          case "cutover": case "v1-dropped": return ["v2"];
        }
      }
    }
    ```

    Step 3 — Create `src/memory/migrations/embedding-v2-runner.ts`:
    ```ts
    /**
     * Resumable batch re-embed runner. Reads memories without v2 embedding, embeds, writes.
     * Yields between batches to respect cpu budget.
     */

    export interface RunnerConfig {
      cpuBudgetPct: number;           // default 5
      batchSize: number;              // default 50
      yieldBetweenBatchesMs: number;  // computed from cpuBudgetPct
    }

    export async function runReEmbedBatch(
      migrator: EmbeddingV2Migrator,
      store: MemoryStore,
      embedder: EmbeddingService,
      cfg: RunnerConfig,
      isAgentActive: () => boolean,    // priority: skip when agent active
      log: Logger,
    ): Promise<{ processed: number; remaining: number; phase: EmbeddingMigrationPhase }> {
      const state = migrator.getState();
      if (state.phase !== "re-embedding" && state.phase !== "dual-write") {
        return { processed: 0, remaining: 0, phase: state.phase };
      }

      // Skip if agent active.
      if (isAgentActive()) return { processed: 0, remaining: state.progressTotal - state.progressProcessed, phase: state.phase };

      // Find memories without v2 embedding, batched.
      const candidates = store.listMemoriesMissingV2Embedding(cfg.batchSize, state.lastCursor);
      if (candidates.length === 0) {
        migrator.transition("re-embed-complete");
        return { processed: 0, remaining: 0, phase: "re-embed-complete" };
      }

      let processed = 0;
      for (const mem of candidates) {
        const embV2Float = await embedder.embedV2Float32(mem.content);
        const embV2Int8 = quantizeInt8(embV2Float);
        store.insertV2Embedding(mem.id, embV2Int8);
        processed++;
        migrator.saveCursor(mem.id, state.progressProcessed + processed, state.progressTotal);

        // Yield for cpu budget.
        await sleep(cfg.yieldBetweenBatchesMs / cfg.batchSize);
      }

      log.info({ agent: migrator.agent, processed, remaining: state.progressTotal - state.progressProcessed - processed, action: "v2-reembed-batch" }, "[diag] embedding-v2 batch re-embed");
      return { processed, remaining: state.progressTotal - state.progressProcessed - processed, phase: state.phase };
    }
    ```

## Threat Model

| Threat | Vector | Severity | Mitigation |
|---|---|---|---|
| Migration corrupts production memory store mid-batch | Daemon crashes during re-embed; cursor lost | HIGH | Transactional write per-row; `last_cursor` saved every batch; resumable on restart; explicit test for resume mid-batch |
| Cutover happens before re-embed completes → silent recall regression | Operator runs `force-cutover` without checking progress | HIGH | `force-cutover` requires interactive confirmation; status command shows `progressProcessed / progressTotal`; daemon-side log line on cutover transition |
| v2 reads return wrong vectors due to int8 quantization mismatch | Round-trip error in dequant; sqlite-vec native handles differently | MEDIUM | T02 vitest verifies quantize→dequant round-trip <1% relative error; T03 + T04 use sqlite-vec native int8 path (no manual dequant in query path) |
| Migration ignores Phase 107 VEC-CLEAN-* cascade — v2 orphans accumulate | New `vec_memories_v2` writes without cascade-delete | HIGH | T03 explicitly extends `MemoryStore.delete` to include v2 cascade; explicit test fixture for orphan cleanup on v2 |
| Migration writes leak across agents | Per-agent isolation breach via shared connection | HIGH | Phase 90 lock — each agent has its own SQLite file; migrator instantiated per-agent; no shared writes; tests assert per-agent isolation |
| Rollback loses the v2 work | Operator rolls back; v2 column dropped → re-migration must start from scratch | MEDIUM | Rolled-back state KEEPS `vec_memories_v2` column populated; only sets read flag; re-attempt picks up where left off |
| Background runner exhausts disk during dual-write | Both v1 and v2 written, doubling embedding storage | MEDIUM | int8 v2 is 4x smaller per-vector than float32 v1; total dual-write footprint is 1.25x v1; daemon-side `[diag]` log warns if memory_chunks file >1GB |

## Notes

### Migration timeline (runs POST-DEPLOY in production — NOT a task in this plan)

This plan ships the *machinery*. The wall-clock timeline executes in production after operator-confirmed deploy:

- **T+0**: Operator runs `clawcode memory migrate-embeddings start <agent>` for each agent. Phase transitions to `dual-write`. New writes embed both v1 (MiniLM) and v2 (bge-small-int8). Reads still use v1.
- **T+0 → T+7d**: Operator monitors via `clawcode memory migrate-embeddings status <agent>`. Counts increase via natural agent activity (new memories written by agent + scanner indexing).
- **T+7d**: Operator transitions phase to `re-embedding`. Background heartbeat-driven batch runs at 5% CPU when daemon idle (Discord-active agents have priority — runner skips when `isAgentActive` returns true). Per-agent: 11 agents × ~30K vectors avg = ~330K total at 50ms each = 4.6 wall-hours of CPU; at 5% budget = 92 wall-clock hours = ~4 days.
- **T+7d → T+14d**: Re-embed runs; resumable across daemon restarts; pausable per-agent.
- **T+14d**: Phase auto-transitions to `re-embed-complete` when no more memories missing v2. Operator runs `clawcode memory migrate-embeddings status` to verify, then transitions to `cutover` (reads switch to v2). Soak for 24h.
- **T+14d + 24h**: Operator transitions to `v1-dropped` — v1 column dropped; migration complete.
- **Rollback path**: At any phase before `v1-dropped`, operator can run `clawcode memory migrate-embeddings rollback <agent>` to revert reads to v1. v2 column data is preserved for re-attempt.

### Phase 107 cascade invariant preserved

Both `vec_memories` AND `vec_memories_v2` deletes happen inside ONE `db.transaction(...)` per Phase 107 VEC-CLEAN-* lock. This holds during the dual-write window AND post-cutover. Test `embedding-v2-cascade-delete.test.ts` regression-pins this invariant.

### Per-agent isolation

Per Phase 90 lock — every per-agent migration step is its own `db.transaction()` per-agent SQLite file. No shared state. Migrator instances are per-agent. The `migrations` table is in each agent's per-agent DB (not in a shared manager DB), making rollback / pause / status all per-agent operations.

### Cost discipline

Per D-09: ~50ms per embedding × ~330K total embeddings = 4.6 hours of CPU time. At 5% budget = 92 wall-clock hours = ~4 days. Fits the T+7d → T+14d window with margin. Discord-active agents have priority — runner yields immediately when `isAgentActive()` returns true.

### Why bge-small over alternatives

D-06 lock — synthesis §2.1 + perf-caching-retrieval §3 both confirm: 5-7 MTEB points over MiniLM, same 384-dim (no MRL truncation), Apache 2.0, ~33MB ONNX, drop-in replacement via `@huggingface/transformers`. Operator's "memory does not need to be human-readable" releases the int8 quantization constraint per D-07.
