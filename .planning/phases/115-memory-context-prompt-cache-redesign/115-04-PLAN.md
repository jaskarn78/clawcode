# Plan 115-04: Cache-breakpoint placement — static-then-dynamic stable-prefix reorder

## Goal

Restructure the stable prefix so static identity (SOUL fingerprint + IDENTITY.md head + capability manifest skeleton + skill catalogue + tool definitions) lands BEFORE the prompt-cache breakpoint and dynamic memory + recent-reflections snippet lands AFTER. Mirrors Hermes pattern. Achieves the operator-priority goal: stable prefix changes only on config change, not on memory write — recovering cache reuse on every turn that doesn't mutate config.

This plan is INDEPENDENT of memory-tier work (115-03) and migration work (115-06), so it ships in parallel within Wave 2.

## Sub-scopes covered

- Sub-scope 5 (roadmap line 828): cache-breakpoint placement — restructure stable prefix; static before, dynamic after.

## must_haves

- The stable prefix order in `assembleContextInternal` reorders so static-only sections (systemPromptDirectives, identitySoulFingerprint, identityFile, identityCapabilityManifest, skillsHeader, toolDefinitions) come BEFORE dynamic-changing sections (identityMemoryAutoload, hotMemories, dreamLogRecent).
- A `BREAKPOINT_MARKER` constant (or sentinel string in the joined prefix) marks the boundary; `session-adapter.ts` is updated to embed the marker so the SDK or our Phase-52 stable-prefix tracking can identify the boundary.
- A new field `staticPrefixHash` is computed (sha256 of the static section ONLY) and tracked alongside `prefixHash` in `traces.db`. When `staticPrefixHash` is unchanged but `prefixHash` is different, we know the dynamic suffix changed but the static portion is cache-eligible — operator can verify cache reuse vs cache write.
- `prompt_cache_hit_rate ≥ 70%` is *measurable* via traces.db (the column exists; this plan does not assert hit rate — that comes from 115-09 closeout running benchmarks).
- The locked SDK shape `{ type: "preset", preset: "claude_code", append: stablePrefix }` is preserved; this plan only changes the *content* of `stablePrefix`, not the SDK call shape.
- Cache-breakpoint behavior is gated by a config flag `defaults.cacheBreakpointPlacement: "static-first" | "legacy"` with default `"static-first"`; legacy revert path preserved.

## Threat Model

| Threat | Vector | Severity | Mitigation |
|---|---|---|---|
| Reordering breaks SDK preset+append cache scaffolding | If the marker text confuses the SDK or downstream parsers | LOW | Marker is HTML comment in markdown; SDK treats `append` as opaque string per Phase 52 lock; legacy mode revert path preserved |
| `staticPrefixHash` divergence indicates cache-bust at wrong granularity | Operators interpret static-bust as expected when it's actually a regression | LOW | `[diag] static-prefix-cache-bust` log emitted ONLY when static portion changes (which is the high-cost event); dynamic-only changes are silent |
| Breakpoint placement misclassifies a section's true volatility | E.g., `identityFile` classified static but operator edits IDENTITY.md daily | LOW | SECTION_PLACEMENT is a single source of truth; misclassification is a config knob — operator can flip per-section in legacy mode pending future phase |
| Legacy mode silently selected and Phase 115 quick wins inactive | Config drift sets `cacheBreakpointPlacement: "legacy"` accidentally | MEDIUM | Default `"static-first"`; `[diag]` log on first request when legacy mode is active surfaces the override |
