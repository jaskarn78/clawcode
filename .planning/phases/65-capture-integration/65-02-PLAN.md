    After line 625 (`try { turn?.end("success"); } catch { /* non-fatal */ }`) and BEFORE the `} catch (error) {` block (line 626), insert:

    ```typescript
    // Phase 65: fire-and-forget conversation capture (SEC-02 instruction detection runs inside)
    try {
      const convStore = this.sessionManager.getConversationStore(sessionName);
      const activeSessionId = this.sessionManager.getActiveConversationSessionId(sessionName);
      if (convStore && activeSessionId) {
        captureDiscordExchange({
          convStore,
          sessionId: activeSessionId,
          userContent: formattedMessage,
          assistantContent: response,
          channelId,
          discordUserId: message.author.id,
          discordMessageId: message.id,
          log: this.log,
        });
      }
    } catch (err) {
      this.log.warn(
        { agent: sessionName, error: (err as Error).message },
        "conversation capture failed (non-fatal)",
      );
    }
    ```

    **Key constraints:**
    - The capture block is INSIDE the existing try block, after turn.end("success") and after the response has been posted to Discord
    - It uses its own nested try/catch so a capture failure does NOT propagate to the outer catch (which would add an error reaction)
    - `sessionName` already handles both channel sessions (agent name like "agent-a") and thread sessions (like "agent-a-thread-thread-100") because both have their own ConversationStore and active session ID from SessionManager.startAgent
    - `formattedMessage` is the raw user content including XML tags from formatDiscordMessage -- this is intentional (store what was actually sent to the agent)
    - `response` is the agent's response text
    - `channelId`, `message.author.id`, `message.id` are Discord.js Message properties already in scope
    - Only capture on SUCCESS path (not in the catch block). On error, there is no `response` to record.
