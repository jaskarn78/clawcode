# Phase 999.3: Specialist subagent routing via delegateTo — Context

**Gathered:** 2026-04-29 (--auto mode, recommended defaults)
**Status:** Ready for planning

<domain>
## Phase Boundary

Extend the existing `spawn_subagent_thread` MCP tool with a new optional `delegateTo: <agent_name>` parameter. When set, the spawned subagent inherits the **target agent's** config (model, soul, identity, skills, subagentModel, mcpServers) instead of the **caller's** config — but the Discord thread is still created in the **caller's** channel, with the caller's webhook URL surfaced under the delegate's display name.

Result: admin-clawdy can spawn `research`-as-a-subagent in admin-clawdy's channel; fin-acquisition can spawn `fin-research`-as-a-subagent in fin-acquisition's channel. The subagent does the work using the delegate's full identity/model/skills, posts in the thread, and (via existing autoRelay) summarizes back to the caller's main channel.

Out of scope (would be scope creep — captured for follow-up):
- Per-agent `specialists: { research: <name>, coding: <name> }` config in clawcode.yaml (Phase 2 extension noted in ROADMAP)
- A `consult_specialist(role)` convenience tool that resolves role → standing agent name automatically (Phase 2 extension)
- Memory continuity across spawns (subagent gets a fresh session — no `forkSession` from delegate's running state)
- Cross-channel delegation (caller specifies a different channel for the thread) — thread always lands in caller's channel

</domain>

<decisions>
## Implementation Decisions

### Architecture
- **D-ARC-01:** Pure additive parameter — `delegateTo: string | undefined` on `spawn_subagent_thread` MCP tool, on the `spawn-subagent-thread` IPC handler, and on `SubagentThreadConfig`. No new infrastructure; all existing scaffolding (thread creation, autoRelay, autoArchive, recursion-guard, webhook identity, registry binding) reused unchanged.
- **D-ARC-02:** Validation at IPC boundary: if `delegateTo` is set, daemon resolves the target name against `resolvedAgents`. If not found, throw `ManagerError("Delegate agent 'X' not found in config")` with verbatim error surfaced to caller per Phase 85 TOOL-04 verbatim-error pattern. The delegate need NOT be currently running — we spawn a fresh subagent from the delegate's static config.

### Config inheritance precedence (caller vs. delegate)
- **D-INH-01:** When `delegateTo` is set, **delegate's config provides:** `model`, `soul`, `identity` (if a separate field exists in ResolvedAgentConfig), `skills`, `subagentModel`, `mcpServers`, `effortLevel` (if set).
- **D-INH-02:** When `delegateTo` is set, **caller's config provides:** `channels` (thread location is caller's first channel), `threads.maxThreadSessions` (the limit applies to the CALLER, not the delegate — caller's quota is what gets consumed), and `webhook.webhookUrl` (Discord's webhook URL is per-channel — caller's channel needs caller's webhook).
- **D-INH-03:** Webhook composition for thread posts: `displayName = delegate.webhook?.displayName ?? delegate.name`, `avatarUrl = delegate.webhook?.avatarUrl`, `webhookUrl = parent.webhook?.webhookUrl` (caller's, since caller owns the channel). This lets the thread show "Acting as fin-research" with fin-research's avatar, posting via admin-clawdy's webhook.
- **D-INH-04:** When `delegateTo` is NOT set, behavior is identical to today — full backwards compatibility. The `delegateTo` branch is purely opt-in.

### Session naming
- **D-NAM-01:** When `delegateTo` is set, session name = `${parentAgent}-via-${delegateTo}-${shortId}` (nanoid 6 chars). Example: `admin-clawdy-via-research-aB3cD9`.
- **D-NAM-02:** When `delegateTo` is NOT set, session name = `${parentAgent}-sub-${shortId}` (current pattern unchanged).
- **D-NAM-03:** The session-name `-via-` infix is greppable in logs (`grep "-via-"` filters all delegated subagent activity) and visible in the thread registry.

### threadContext (subagent system-prompt suffix)
- **D-TCX-01:** When `delegateTo` is set, threadContext extends the standard 6 lines with 4 additional lines:
  ```
  ## Delegation Context
  You are acting on behalf of `${parentAgent}` who delegated this work to you.
  Use your full identity, skills, and standards. Treat the operator's request
  as one fulfilled THROUGH you, not BY you.
  ```
- **D-TCX-02:** The "delegation context" suffix complements (does not replace) the FRESH/DERIV/TRUST/TABLE directives shipped in Phase 104 — those still fire for delegated subagents because they're in the stable prefix above the soul.
- **D-TCX-03:** Recursion guard (Phase 99-N) STILL applies to delegated subagents — `disallowedTools: ["mcp__clawcode__spawn_subagent_thread"]` is set unconditionally, whether `delegateTo` is set or not. Prevents 5-deep recursion regardless of delegation.

### Tool surface
- **D-TOL-01:** Update the MCP tool's `spawn_subagent_thread` Zod schema to accept `delegateTo: z.string().optional().describe("...")`.
- **D-TOL-02:** Tool description for `delegateTo`: *"Optional. When set to a target agent name (e.g., 'fin-research', 'research', 'code-clawdy'), the spawned subagent uses the target's config (model, soul, skills) instead of yours. The thread still spawns in your channel, autoRelay still summarizes back to your main channel — but the work is done with the target agent's identity. Use this to delegate elevated-thinking work (research, coding) to a dedicated specialist standing agent."*
- **D-TOL-03:** Update the parent tool description to reference the new param: *"Use `delegateTo` to have a specialist agent (e.g., research-clawdy, fin-research) do the work using their config — useful when you want elevated thinking, opus-level reasoning, or specialist skills you don't have."*

### Hot-reload + state lifecycle
- **D-HOT-01:** No hot-reload concerns — `delegateTo` is a per-call parameter, not a config field. Daemon picks up the new schema on restart (one-time deploy).
- **D-HOT-02:** Thread binding records the CALLER as `agentName`, not the delegate. Routing for replies in the thread goes to the spawned subagent session (not to the delegate's standing session). The standing delegate is never affected by the spawn — it keeps running its own work.

### Edge cases
- **D-EDG-01:** Delegate has no skills configured: subagent runs without skills (matches today's no-skills agent behavior). Not an error.
- **D-EDG-02:** Delegate has `autoStart: false` (e.g., `research` is dormant): allowed. We spawn fresh from static config; delegate's runtime state doesn't matter.
- **D-EDG-03:** Caller passes `delegateTo: "<own-name>"` (delegate to self): allowed but functionally equivalent to no delegation. Test pins this is a no-op behaviorally (subagent inherits from caller as before, session name uses `-via-` infix to flag the operator's intent).
- **D-EDG-04:** Caller passes `delegateTo: ""` (empty string): treated same as not set — Zod's `.optional()` parses empty as undefined; runtime falls through to standard branch.
- **D-EDG-05:** Caller passes `delegateTo: "non-existent-agent"`: ManagerError thrown at IPC boundary (D-ARC-02), surfaced verbatim to caller's tool result. Subagent NOT spawned.

Phase 2 (deferred): per-caller config maps `research` role → `fin-research` for fin-* agents, `research` for non-fin. A `consult_specialist(role)` tool handles the resolution. Out of scope for this iteration; agents can name the delegate explicitly.

<deferred>
## Deferred Ideas

- **Per-agent `specialists: { research: <name>, coding: <name> }` config** — explicit per-caller specialist mapping in clawcode.yaml. Deferred to Phase 2 of this work; explicit `delegateTo` covers the use case.
- **`consult_specialist(role)` convenience tool** — wraps `spawn_subagent_thread` with role-to-name resolution from per-agent specialists config. Deferred with the per-agent specialists config above.
- **Cross-channel delegation** — caller specifies a different channel for the thread (e.g., delegate to fin-research with the thread spawning in fin-research's channel instead of caller's). Rejected for v1: would require channel-permission validation across agents and complicates the autoRelay routing back. Caller's channel is the standard.
- **Memory continuity via forkSession** — subagent forks from the standing delegate's running session to preserve memory continuity. Rejected for v1: the standing agent's session may not be running, the SDK's forkSession-from-running semantics may be unstable, and a fresh subagent from the delegate's static config covers most use cases. Revisit in a later phase if memory continuity becomes a real pain point.
- **Specialist auto-archive policy** — delegated subagents could default `autoArchive: true` to encourage fire-and-forget consultation. Rejected: caller explicitly sets autoArchive based on use case; auto-overriding to true breaks interactive consult flows.
