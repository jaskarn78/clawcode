## Summary

This phase is a **purely additive parameter extension** of the existing `spawn_subagent_thread` rail. CONTEXT.md locks every architectural decision (D-ARC, D-INH, D-NAM, D-TCX, D-TOL, D-EDG, D-PLN). No new infrastructure, no new files (likely), no new MCP tools. The work is to fan a new optional `delegateTo: string` field through 4 surfaces: MCP tool schema → IPC handler → `SubagentThreadConfig` → `SubagentThreadSpawner.spawnInThread`.

The interesting bits are in the **composition contract** (D-INH-01..04) — when `delegateTo` is set, the spawner must compose a `ResolvedAgentConfig` that takes most fields from the **delegate** agent (model/soul/identity/skills/subagentModel/mcpServers), but keeps `channels[0]`/`threads.maxThreadSessions`/`webhook.webhookUrl` from the **caller**. Webhook composition is non-trivial: caller's `webhookUrl` (channel-bound) but delegate's `displayName + avatarUrl` (per-message overrides via discord.js webhook client). Verified valid — `webhook-manager.ts:71-75` already passes `username` and `avatarURL` per `client.send()` call, decoupled from the webhookUrl's channel binding.

The recursion guard (Phase 99-N) MUST remain unconditional — `disallowedTools: ["mcp__clawcode__spawn_subagent_thread"]` is set on every spawned subagent regardless of delegation. The Phase 99-N test suite (`__tests__/subagent-recursion-guard.test.ts`) is the regression contract; new tests must NOT introduce a code path that bypasses it.

**Primary recommendation:** Mirror existing test patterns from `subagent-thread-spawner.test.ts` (extend the file, don't create a new one). Add `delegateTo?: string` to `SubagentThreadConfig` first (TDD RED), then thread it through the spawner branch in `spawnInThread`, then through the IPC handler at `daemon.ts:5042`, then through the MCP Zod schema at `mcp/server.ts:340-378`. Order matters — types first lets each subsequent layer compile-check correctly.

### Pattern 1: Composition with object spread (immutable)

**What:** The existing `subagentConfig` assembly already uses `{ ...parentConfig, ...overrides }`. The delegate branch follows the same shape — it just chooses which object to spread first.

**When to use:** All config assembly in this codebase. Mutation is forbidden (CLAUDE.md coding-style.md).

**Example (proposed shape — Task 2 GREEN):**
```typescript
// Source: synthesis of subagent-thread-spawner.ts:396-406 + D-INH-01..03

const delegateConfig = config.delegateTo
  ? this.sessionManager.getAgentConfig(config.delegateTo)
  : undefined;

// IPC layer (daemon.ts) already validated delegate exists; defensive re-check.
if (config.delegateTo && !delegateConfig) {
  throw new ManagerError(`Delegate agent '${config.delegateTo}' not found`);
}

// Choose the "config-source" agent for inherited fields.
const sourceConfig = delegateConfig ?? parentConfig;

// 5. Build session name (D-NAM-01 / D-NAM-02)
const shortId = nanoid(6);
const sessionName = config.delegateTo
  ? `${config.parentAgentName}-via-${config.delegateTo}-${shortId}`
  : `${config.parentAgentName}-sub-${shortId}`;

// 6. Build thread context (D-TCX-01 — append delegation block)
const baseContext = [
  "\n\n## Subagent Thread Context",
  `You are a subagent operating in a Discord thread.`,
  `Thread: "${config.threadName}" (ID: ${thread.id})`,
  `Parent channel: ${channelId}`,
  `Parent agent: ${config.parentAgentName}`,
  "Respond to messages in this thread only.",
];
const delegationContext = config.delegateTo
  ? [
      "",
      "## Delegation Context",
      `You are acting on behalf of \`${config.parentAgentName}\` who delegated this work to you.`,
      "Use your full identity, skills, and standards. Treat the operator's request",
      "as one fulfilled THROUGH you, not BY you.",
    ]
  : [];
const threadContext = [...baseContext, ...delegationContext].join("\n");

// 7. Determine model — from sourceConfig (delegate when delegating, parent otherwise)
const model = config.model ?? sourceConfig.subagentModel ?? sourceConfig.model;

// 8. Build webhook identity per D-INH-03
//    - displayName/avatar from DELEGATE (or sessionName fallback for non-delegate)
//    - webhookUrl from PARENT (caller owns the channel)
const webhook = parentConfig.webhook?.webhookUrl
  ? {
      displayName: config.delegateTo
        ? (delegateConfig!.webhook?.displayName ?? delegateConfig!.name)
        : sessionName,
      avatarUrl: config.delegateTo
        ? delegateConfig!.webhook?.avatarUrl
        : parentConfig.webhook.avatarUrl,
      webhookUrl: parentConfig.webhook.webhookUrl,
    }
  : undefined;

// 9. Build subagent ResolvedAgentConfig — sourceConfig contributes inherited
//    fields (model/soul/identity/skills/subagentModel/mcpServers/effort);
//    parentConfig + overrides contribute the channel-scoped/quota fields.
const subagentConfig: ResolvedAgentConfig = {
  ...sourceConfig,                           // <- delegate's everything (or parent's, if no delegate)
  name: sessionName,
  model,
  channels: [],                              // never inherit channels for subagents
  soul: (config.systemPrompt ?? sourceConfig.soul ?? "") + threadContext,
  schedules: [],                             // subagents don't run schedules
  slashCommands: [],
  webhook,
  threads: parentConfig.threads,             // D-INH-02 — caller's quota
  disallowedTools: ["mcp__clawcode__spawn_subagent_thread"],  // D-TCX-03 — UNCONDITIONAL
};
```

**Notes on the proposed shape:**
- The `...sourceConfig` spread inherits the delegate's `mcpServers`, `mcpEnvOverrides`, `gsd.projectDir`, `settingSources`, `effort`, etc. This matches D-INH-01.
- `threads: parentConfig.threads` is an EXPLICIT override after the spread to enforce D-INH-02 (caller's quota). Without it, `...sourceConfig` would have spread the delegate's `threads` field.
- The webhook block reads from `parentConfig.webhook?.webhookUrl` — caller's channel-bound URL. The `displayName` + `avatarUrl` are per-message overrides (verified at `webhook-manager.ts:71-75` — discord.js `client.send({ username, avatarURL })` overrides per-call without rebinding the webhook to a new channel).

### Anti-Patterns to Avoid

- **Conditional disallowedTools (CRITICAL):** Setting `disallowedTools` ONLY on the non-delegate branch — would silently bypass the recursion guard for delegated subagents. Must be unconditional. The Phase 99-N test suite at `__tests__/subagent-recursion-guard.test.ts:218-257` (RG5) verifies this for parent identities; new tests must extend the invariant to the delegated branch.
- **Spreading delegate config last (would clobber caller's channels):** `{ ...parentConfig, ...delegateConfig }` would leak the delegate's `channels[]` into the subagent — which then propagates wrong identity to the binding. The order is `...sourceConfig` first, then explicit overrides for `channels: []`, `threads: parentConfig.threads`, etc.
- **Using `validateStringParam` for `delegateTo`:** It would reject empty strings as "Missing required parameter" — breaking D-EDG-04 (empty string treated as undefined). Use the optional-string pattern at daemon.ts:5048-5050 instead.
- **Reading delegate config inside the spawner without IPC validation:** D-ARC-02 places validation at the IPC boundary so the verbatim error surfaces to the caller. The spawner can defensively re-check (DEL-10), but the user-visible failure path is the IPC.
- **Mutating `parentConfig`:** Never. The composition pattern uses `{ ...sourceConfig, ... }` to produce a new object.

## Common Pitfalls

### Pitfall 1: Forgetting to forward `delegateTo` through the IPC layer
**What goes wrong:** MCP tool accepts the new `delegateTo` arg, but daemon.ts:5042 doesn't read `params.delegateTo` and pass it to `spawner.spawnInThread`. Result: silent failure — the subagent spawns with caller's config (no error surfaced because all validation passes).

**Why it happens:** 4-surface fan-through is easy to mis-edit. The MCP layer reads `delegateTo` from the Zod-parsed args (line 353 destructuring), then passes it in `sendIpcRequest(...)`. The daemon's `case "spawn-subagent-thread"` (line 5042) must explicitly read `params.delegateTo` and include it in the `subagentThreadSpawner.spawnInThread({ ... })` call.

**How to avoid:** Task 3 test asserts the `spawnInThread` mock was called with `delegateTo` set. If using IPC integration tests, assert the daemon-side handler reads the field. Trace the field through every layer in test (Task 1 RED tests cover the spawner; Task 3 covers IPC; Task 4 covers MCP).

**Warning signs:** Subagent's session name doesn't have `-via-` infix even though the operator passed `delegateTo` — exact field-not-forwarded symptom.

### Pitfall 2: Webhook composition — wrong `webhookUrl`
**What goes wrong:** Setting `webhookUrl: delegateConfig.webhook?.webhookUrl` instead of `parentConfig.webhook?.webhookUrl`. Webhooks are channel-bound — using the delegate's URL would post messages to the DELEGATE's channel, not the caller's thread (which lives in caller's channel).

**Why it happens:** Spread-first composition (`{...delegateConfig, ...}`) clobbers `webhook` field. Easy to miss because the spread is implicit.

**How to avoid:** Test DEL-04 explicitly asserts `webhookUrl === caller's URL`. The composition shape (Pattern 1 above) puts `webhook` AFTER the spread to enforce the correct override.

**Warning signs:** Subagent posts go to wrong channel or fail with "webhook not found" if delegate has no webhook configured.

### Pitfall 3: Recursion guard accidentally bypassed
**What goes wrong:** A clean refactor moves `disallowedTools` into a conditional branch (e.g., the non-delegate path's `subagentConfig` literal) and forgets to add it to the new delegate branch. Phase 99-N regression — 5-deep recursion returns.

**Why it happens:** Tempting to "clean up" by moving fields into the spread literal that constructs each branch.

**How to avoid:** Test DEL-05 asserts the invariant on the delegate branch. Test RG1 / RG5 in `__tests__/subagent-recursion-guard.test.ts` already asserts the non-delegate branch. The recursion guard should be set OUTSIDE both branches — in the final assembly literal (line 405 of the proposed shape).

**Warning signs:** Phase 99-N test suite fails (RG1/RG5). Or worse, tests pass but the field is conditionally absent — operator sees recursion bug return.

### Pitfall 4: Forgetting to update the Zod schema
**What goes wrong:** MCP tool definition passes `delegateTo` to IPC but the Zod schema doesn't declare it — discord-side request validation rejects it as "unknown property" (Zod 4 strict mode would reject unrecognized keys).

**Why it happens:** Easy to update the destructuring and the `sendIpcRequest` call without adding the schema entry.

**How to avoid:** Task 4 test loads the Zod schema and asserts `delegateTo` is a valid optional string field. Or simpler — the typecheck script (`npm run typecheck`) catches it because the MCP server.tool() second-arg generic infers from the schema; the destructured `{ delegateTo }` would be flagged if absent from the schema.

**Warning signs:** Tool call fails at the MCP boundary with "Invalid params" before reaching the IPC layer.

### Pitfall 5: Empty-string `delegateTo`
**What goes wrong:** Operator passes `delegateTo: ""` (perhaps from a UI default or stale config). With `.optional()`, Zod returns `""`, which is truthy as a string. The spawner's `config.delegateTo ? ... : ...` check is FALSY for empty string, so it falls through to the standard branch. But if the IPC handler validates `delegateTo` length WITHOUT the empty-treats-as-undefined logic, it could throw `Delegate agent '' not found`.

**Why it happens:** D-EDG-04 specifies empty-string-as-undefined, but the IPC handler at daemon.ts:5042 has to implement this normalization.

**How to avoid:** In the IPC handler, normalize: `const delegateTo = typeof params.delegateTo === "string" && params.delegateTo.length > 0 ? params.delegateTo : undefined;`. Test DEL-09 covers the no-delegateTo case; add DEL-09b for explicit empty-string.

**Warning signs:** Caller passes empty string and gets a "Delegate agent '' not found" error instead of a clean default-path spawn.

### Pitfall 6: Session-name collision
**What goes wrong:** Two simultaneous spawns with the same caller + delegate — `nanoid(6)` collision is statistically negligible (~36^6 = 2.1B values) but theoretically possible.

**Why it happens:** nanoid 6 chars is sufficient for this scale (handful of subagents per agent), but if the existing session manager has assertion-based name uniqueness, a collision throws.

**How to avoid:** Pin in test (DEL-02) that the regex matches the pattern; rely on existing session-manager uniqueness handling (which already worked for `-sub-` shortIds for years). No new defense needed.

**Warning signs:** N/A in practice. Documented for completeness.

### Pitfall 7: Inheriting delegate's `channels[]` accidentally
**What goes wrong:** `{ ...delegateConfig, ... }` spread includes `channels: ["delegate-channel-id"]` — the subagent gets bound to the delegate's primary channel for its identity routing. The thread is in caller's channel but the agent thinks it's in delegate's. Bridging gets confused.

**Why it happens:** Spread-clobber. Already addressed in non-delegate path by `channels: []` override (line 400).

**How to avoid:** Keep the explicit `channels: []` override in the final assembly. Test that `startedConfig.channels` is `[]` regardless of delegate's channels.

**Warning signs:** Subagent posts in wrong channel or ignores thread messages.

### Pitfall 8: `mcpEnvOverrides` and `gsd.projectDir` propagation
**What goes wrong:** D-INH-01 says delegate provides `mcpServers` — but should it also provide `mcpEnvOverrides` (the per-agent token-distribution map)? And `gsd.projectDir` (used by Phase 100 GSD-06 artifact discovery)? CONTEXT.md doesn't list these explicitly.

**Why it happens:** ResolvedAgentConfig has 30+ fields. Spread `...delegateConfig` carries them all unless explicitly overridden.

**How to avoid:** **Spread `...sourceConfig` and let it carry the delegate's full identity** — including `mcpEnvOverrides` (tokens scoped to the delegate's agent) and `gsd.projectDir` (delegate's project root). This is consistent with D-INH-01's spirit (delegate's identity = full identity). The artifact-discovery during autoRelay then surfaces from the DELEGATE's project root, not caller's — which makes sense because the delegate did the work.

**Validation:** Ask in plan-checker if the operator wants caller's `gsd.projectDir` instead. Defaults to delegate's per the "delegate's everything" spread.

**Warning signs:** autoRelay surfaces wrong artifact paths to the caller's main channel — inconsistent with the work the subagent actually did.
