## Summary

The leak is real, the root cause is mechanical, and the fix is structural — but **the architecture differs from CONTEXT.md's framing in one critical way**: the daemon does NOT directly spawn MCP server child processes. The `@anthropic-ai/claude-agent-sdk` (v0.2.97 pinned) forks a `claude` CLI subprocess per agent, and **that** subprocess spawns the MCP servers using its own `child_process.spawn` calls (no `detached`, no exposed PIDs to the daemon). The daemon's only hooks are: (1) the SDK options object (`mcpServers: Record<name, {command,args,env}>`), and (2) the `claude` CLI's process lifecycle, which is opaque to us.

This means **MCP-01 cannot be implemented as "set spawn options on our side"** — there are no spawn options on our side for these processes. The fix is necessarily indirect:

1. **Discovery, not spawn-time wiring** (MCP-01 reframe): At agent-start time, after the SDK has booted the `claude` subprocess, scan `/proc` for processes whose `ppid` matches the agent's `claude` PID and whose cmdline matches a configured MCP server. Store these PIDs in a per-agent tracker. This **gives us the PIDs we need for MCP-02 and MCP-04** without fighting the SDK.

2. **Process-group reach** (MCP-02): When the agent's `claude` subprocess exits or disconnects, the npm-wrapped MCP servers reparent their `sh -c → node` grandchildren to PID 1 (today's leak). The SDK's `Query.disconnect()` and the `persistent-session-handle.ts` teardown path are the trigger points. On those triggers, fire `process.kill(-pid, 'SIGTERM')` against each tracked MCP PID — Linux propagates the signal to the entire process group, reaching the grandchildren the npm wrapper otherwise drops.

3. **Reaper + boot scan** (MCP-03/05): Same scanner function, two callsites. 60s interval for steady-state belt-and-suspenders; one-shot at boot before any agent starts. Plain `/proc` walk filtering on `uid==clawcodeUid AND ppid==1 AND cmdline matches union-regex`.

4. **Graceful shutdown** (MCP-04): Insert `tracker.killAll()` into the existing `shutdown()` function in `daemon.ts:3962` between `manager.stopAll()` and PID-file cleanup. Don't rely on systemd's cgroup-mode kill — it backstops, but the leak window is between daemon graceful exit and systemd-final-kill, plus the next daemon start can race with cgroup teardown.

**Primary recommendation:** Build a single `McpProcessTracker` daemon-side singleton (mirrors `SecretsResolver` pattern) with three responsibilities: (1) per-agent register/unregister of MCP PIDs discovered via `/proc` scan, (2) `killGroup(pid)` helper using `process.kill(-pid, sig)`, (3) `scanOrphans()` returning PPID=1 + cmdline-match candidates. Wire MCP-01..05 around this single object. ~150 lines total across 2-3 files.

### Pattern 1: PID Discovery After Agent Start (MCP-01 reframe)

**What:** The daemon cannot intercept the SDK's MCP-server spawn. Instead, after `manager.startAgent(name)` resolves (the `claude` subprocess is up and has begun spawning MCP servers), the daemon waits a short settle window (~500ms-1s), then walks `/proc` for processes whose `ppid` equals the agent's `claude` PID and whose cmdline matches any configured MCP server's first arg.

**When to use:** Per-agent, once at agent-start. Re-run when the SDK reconnects an MCP server (capability-probe heartbeat may trigger this — out of scope here, but the tracker has an `update(agentName, pids)` API for it).

**Sketch:**
```typescript
// Source: derived from /proc semantics + persistent-session-handle.ts pattern
async function discoverAgentMcpPids(
  agentClaudePid: number,
  configuredCommands: readonly string[],   // from clawcode.yaml mcpServers[].command + args[0]
): Promise<readonly number[]> {
  const entries = await readdir("/proc");
  const out: number[] = [];
  for (const e of entries) {
    if (!/^\d+$/.test(e)) continue;
    const pid = Number(e);
    const stat = await safeRead(`/proc/${pid}/stat`);
    const cmdline = await safeRead(`/proc/${pid}/cmdline`);
    if (!stat || !cmdline) continue;            // proc disappeared
    const ppid = parseStatPpid(stat);            // see Pitfall 4 — parse after ')'
    if (ppid !== agentClaudePid) continue;
    const cmd = cmdline.split("\0").filter(Boolean).join(" ");
    if (!matchesAnyMcpCommand(cmd, configuredCommands)) continue;
    out.push(pid);
  }
  return out;
}
```

**The agent's `claude` PID:** Currently NOT tracked. The SDK does not expose the spawned subprocess PID via its public API (verified against `node_modules/@anthropic-ai/claude-agent-sdk/sdk.d.ts` — no `pid` field on `Query` or `SessionHandle`). **This is a research-surfaced gap**: MCP-01 implementation must EITHER monkey-patch / wrap the SDK's spawn (fragile, since SDK is pre-1.0) OR scan `/proc` for `claude` processes whose own `ppid` is the daemon's PID. The latter is the recommended path. See Open Questions.

### Pattern 2: Process-Group Kill (MCP-02 / MCP-04)

**What:** `process.kill(-pid, 'SIGTERM')` on POSIX sends the signal to all processes in the process group whose pgid equals `pid`. Verified on clawdy: the leaked `sh -c mcp-server-mysql` has `pgrp=3828787` (the daemon's PID), but the npm-wrapped MCP servers establish their own pgid. Each npm-spawned MCP wrapper IS a process-group leader, so `kill(-mcpPid, 'SIGTERM')` reaches `npm exec → sh -c → node` together.

**When to use:** On agent disconnect (MCP-02), on daemon shutdown (MCP-04), and from the orphan reaper (MCP-03/05) — though for orphans we kill individual PIDs, not groups, because the orphans have PPID=1 and may not retain their original pgid.

**Sketch:**
```typescript
// Source: man 2 kill (POSIX) — negative pid signals process group
function killGroup(pid: number, sig: NodeJS.Signals = "SIGTERM"): void {
  try {
    process.kill(-pid, sig);
  } catch (err: unknown) {
    const e = err as NodeJS.ErrnoException;
    if (e.code === "ESRCH") return;          // already dead — idempotent
    if (e.code === "EPERM") {                // not our process — never on UID-matched scan
      log.warn({ pid, sig, code: e.code }, "kill denied");
      return;
    }
    throw err;
  }
}
```

**Note:** `process.kill(0, sig)` is reserved (signals all processes in caller's group). Never pass `pid === 0`. Validate `pid > 0` before negation.

### Anti-Patterns to Avoid

- **`detached: true`**: Decouples the child from the parent. Even if you call `unref()`, you lose the cgroup guarantee on shutdown and create MORE orphans. CONTEXT.md locks this — researcher confirms.
- **`tree-kill` / shelling to `pkill`**: Adds dependency / shell injection surface for what `process.kill(-pid)` does in one syscall.
- **Reading `/proc/{pid}/environ` for orphan classification**: Leaks env vars into logs (security risk per `~/.claude/rules/security.md`). Use cmdline only.
- **Naive `stat.split(' ')[3]` for ppid**: The comm field (field 2) is parenthesized and can contain spaces / parens / newlines. See Pitfall 4.
- **Parsing `ps` output**: `ps` is unreliable for PPID across runs (race-prone snapshots, locale-dependent column widths). `/proc/{pid}/stat` is atomic.
- **`setInterval` without `unref()`**: Will keep the event loop alive past shutdown signal handler. Pattern from `src/heartbeat/runner.ts:152` — `intervalId` saved + `clearInterval` in `stop()` (no `unref` needed since we explicitly clear in shutdown).
- **Killing on PPID=1 alone, no UID filter**: Would target every PPID=1 process matching the cmdline regex on the whole machine — including unrelated user accounts. UID filter (`uid === clawcodeUid`) is the safety net.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Process tree walking | Custom recursive ppid traversal | `/proc/{pid}/stat` field 4 (ppid) — single read per pid | Linear walk over `/proc` is O(N processes); recursive parent-chain walks compound the race window. |
| Process listing | Shell out to `ps -ef` | `readdir('/proc')` filtering numeric entries | Atomic snapshot, no parser fragility, no shell. |
| Signal cascade to children | Custom recursive kill loop | `process.kill(-pid, sig)` POSIX process-group | One syscall, kernel-atomic. |
| Cmdline parsing | `ps -o args` and split-by-space | `/proc/{pid}/cmdline` (NUL-delimited, atomic) | `ps` truncates by terminal width; cmdline file gives full original argv. |
| UID lookup | Hard-coded UID literal | `process.getuid()` at daemon start, capture once | Deployment portability — same code runs in dev (UID=jjagpal:1000) and prod (UID=clawcode:999). |

**Key insight:** Linux `/proc` is a stable kernel API. The reaper logic is ~50 lines of pure stdlib code. Adding a dependency for this is the wrong instinct — but writing each part from scratch without checking what's there is the OTHER wrong instinct. Net: zero new deps, reuse `pino` + `node:fs/promises` + `node:child_process`.

## Common Pitfalls

### Pitfall 1: SDK Does Not Expose MCP Subprocess PIDs
**What goes wrong:** Implementer assumes `query.pid` or `sessionHandle.mcpPids` exists; writes code that compiles against `SdkQuery` types but throws at runtime, OR worse, returns undefined and silently registers nothing — so the tracker stays empty and the reaper never has work to do.
**Why it happens:** `@anthropic-ai/claude-agent-sdk@0.2.97` exposes session-level abstractions (`Query.disconnect()`, `setModel`, `interrupt`) but no child-process-level surfaces. Verified by reading `node_modules/@anthropic-ai/claude-agent-sdk/sdk.d.ts:1218,1226` — `mcpServers` is input only.
**How to avoid:** Use `/proc` discovery (Pattern 1). Verify via integration test: spawn a real `claude` subprocess (or a fake), confirm the discovery scan finds expected children.
**Warning signs:** Tracker `pids` map is empty after agent start. Heartbeat reaper logs `scanned=N orphans=0 tracked=0` — should be `tracked > 0` when agents have MCP servers.

### Pitfall 2: PPID=1 Is Not Atomic With Kill Decision
**What goes wrong:** Reaper reads `/proc/{pid}/stat`, sees PPID=1, decides to kill — but between read and kill, the proc has been reaped by a fast cleanup elsewhere, and the PID has been recycled to an unrelated process (e.g., a system daemon). SIGTERM hits the wrong target.
**Why it happens:** Linux PID space wraps quickly under load (32k default, `cat /proc/sys/kernel/pid_max`). Verified on clawdy: `pid_max` typically 4194304, but on busy boxes the wrap window can still be hit.
**How to avoid:** Re-verify ppid AND uid AND cmdline-prefix immediately before kill. Use `try/catch` on `process.kill` and treat `ESRCH` as success (proc already gone). Reaper loop: read stat → check ppid+uid → read cmdline → check match → kill (with second stat-read as final gate). Cost: 2 syscalls per candidate, dwarfed by interval scan period.
**Warning signs:** Random unrelated processes dying after deploy.

### Pitfall 3: Daemon Restart Race — New MCP Spawns During Reaper Sweep
**What goes wrong:** Reaper's 60s interval fires while an agent is mid-startAgent. The freshly-spawned MCP server has PPID=1 transiently (during the npm wrapper's exec handoff between `npm exec` → `sh -c` → `node`), gets reaped immediately. Agent never gets a working MCP.
**Why it happens:** `npm exec` spawns `sh -c <cmd>`, exits, `sh` exec's `node`. Brief window where the `node` proc has PPID=1 before parent reaping completes.
**How to avoid:** **CONTEXT.md non-negotiable**: exclude procs younger than 5s from reaper. Read process start time from `/proc/{pid}/stat` field 22 (`starttime` in clock ticks since boot) — convert via `(stat.starttime / clock_ticks_per_sec) + boot_time` and compare to `Date.now() / 1000`. Skip if `now - start_seconds < 5`.
  - Alternative: register the agent's expected MCP cmdlines into a "spawning" set with a 10s TTL when `manager.startAgent` runs, exclude any cmdline-matching proc whose ppid path traces back to that set. More complex, harder to reason about. Prefer the age filter.
**Warning signs:** Agents fail to come up after restart with "MCP server failed to initialize" — no log of why because the reaper killed it before the SDK saw it ready.

### Pitfall 4: Naive `/proc/{pid}/stat` Parsing Breaks On Comm With Spaces
**What goes wrong:** `stat` field 2 is the executable name in parens: `(node)`. But it can be `(sh -c foo)` or `(claude (test-agent))` — splitting `stat.split(' ')` indexes off-by-N because of internal spaces and parens.
**Why it happens:** Linux `/proc/{pid}/stat` format: `pid (comm) state ppid pgrp …`. The `comm` field can contain `)` because it's literally the first 16 bytes of the binary's name; the kernel just wraps it in parens. Stable parser: `lastIndexOf(')')` to find the end of comm, then split the remainder.
**How to avoid:**
```typescript
function parseStatPpid(stat: string): number {
  const close = stat.lastIndexOf(")");
  if (close === -1) throw new Error("malformed stat: no closing paren");
  // After ")  ", fields are: state ppid pgrp ...
  const tail = stat.slice(close + 2).split(" ");
  // tail[0] = state, tail[1] = ppid
  const ppid = Number(tail[1]);
  if (!Number.isFinite(ppid)) throw new Error("malformed stat: ppid not numeric");
  return ppid;
}
```
**Warning signs:** Parser throws on certain MCP servers but not others. Process names with parens aren't detected as orphans.

### Pitfall 5: SIGTERM-on-`disconnect` Fires Before MCP Has Settled
**What goes wrong:** MCP-02 wires SIGTERM to the SDK's `disconnect` event. The SDK fires `disconnect` on transient transport hiccups (handled internally by `mcp-reconnect` heartbeat), not just on agent-shutdown. SIGTERM kills MCPs that the SDK was about to reconnect.
**Why it happens:** `disconnect` is overloaded — both "session ended" and "MCP transport reconnect cycle". The reconnect cycle is owned by `src/heartbeat/checks/mcp-reconnect.ts`.
**How to avoid:** Wire MCP-02 to `Query.disconnect()` (the user-initiated session shutdown) AND the `claude` subprocess `exit` event (parent died) — NOT to per-MCP transport disconnect events. `persistent-session-handle.ts` is the right place — its existing teardown path is the correct trigger.
**Warning signs:** MCPs being killed and respawned every minute under normal operation; reconnect-storm in logs.

### Pitfall 6: Race Between Daemon-Shutdown and MCP `npm` Wrapper Exit
**What goes wrong:** MCP-04 SIGTERMs npm wrappers; npm wrappers have their own SIGTERM handler that exits cleanly without forwarding to `sh -c → node` (the original bug). Then daemon exits. Children become orphans. No improvement.
**Why it happens:** `npm exec` passes signals to the child shell, but the `sh -c <cmd>` chain may exit faster than the `node` exec it spawned (if `node` was the last thing in the script). Linux reparents the still-running `node` to PID 1.
**How to avoid:** Use `process.kill(-pid, 'SIGTERM')` (process-group form, **negative pid**), not `process.kill(pid, 'SIGTERM')`. POSIX kernel propagates to all PIDs sharing the pgid, regardless of intermediate parent exits. Verified primitive.
**Warning signs:** Acceptance test fails — `pgrep -cf mcp-server-mysql` after `systemctl restart` still shows orphans.

### Pitfall 7: Test False-Pass via Mocked `/proc`
**What goes wrong:** Tests mock `readFile('/proc/...')` returning a synthetic ppid=1 line. Code passes. Production fails because the real `/proc` parsing has a quirk (e.g., the comm-field parens issue) the mock didn't model.
**Why it happens:** Mocks model the API contract, not the real format. Linux kernel format is canonical; mock divergence is a tax.
**How to avoid:** Require at least ONE Linux-only integration test in `proc-scan.test.ts` that spawns a real child and reads its real `/proc/{pid}/stat`. Use vitest `.skipIf(process.platform !== 'linux')`. CI must run on Linux (it does — same as deploy target).
**Warning signs:** All tests green, manual SSH check shows orphans persist.

### Read /proc and parse PPID + cmdline (Linux)
```typescript
// Source: man 5 proc — /proc/{pid}/stat fields, /proc/{pid}/cmdline NUL-delimited
import { readFile, readdir } from "node:fs/promises";

type ProcInfo = {
  readonly pid: number;
  readonly ppid: number;
  readonly uid: number;
  readonly cmdline: readonly string[];
  readonly startTimeJiffies: number;       // field 22; for age filter
};

async function readProcInfo(pid: number): Promise<ProcInfo | null> {
  try {
    const [stat, cmdRaw, status] = await Promise.all([
      readFile(`/proc/${pid}/stat`, "utf-8"),
      readFile(`/proc/${pid}/cmdline`, "utf-8"),
      readFile(`/proc/${pid}/status`, "utf-8"),
    ]);
    const close = stat.lastIndexOf(")");
    if (close === -1) return null;
    const tail = stat.slice(close + 2).split(" ");
    // tail[0]=state tail[1]=ppid tail[2]=pgrp ... tail[19]=starttime (field 22 - 3 prior to comm)
    const ppid = Number(tail[1]);
    const startTimeJiffies = Number(tail[19]);
    const uidLine = status.match(/^Uid:\s*(\d+)/m);
    const uid = uidLine ? Number(uidLine[1]) : NaN;
    const cmdline = cmdRaw.split("\0").filter(Boolean);
    return { pid, ppid, uid, cmdline, startTimeJiffies };
  } catch (err: unknown) {
    const e = err as NodeJS.ErrnoException;
    if (e.code === "ENOENT" || e.code === "ESRCH") return null;
    throw err;
  }
}

async function listAllPids(): Promise<readonly number[]> {
  const entries = await readdir("/proc");
  return entries.filter((e) => /^\d+$/.test(e)).map(Number);
}
```

### Process-group SIGTERM with idempotent error handling
```typescript
// Source: man 2 kill (POSIX) — negative pid signals process group
import type { Logger } from "pino";

export function killGroup(pid: number, sig: NodeJS.Signals, log: Logger): boolean {
  if (pid <= 1) {
    log.error({ pid }, "refusing to kill pid <= 1");
    return false;
  }
  try {
    process.kill(-pid, sig);
    return true;
  } catch (err: unknown) {
    const e = err as NodeJS.ErrnoException;
    if (e.code === "ESRCH") return true;     // already gone — idempotent success
    if (e.code === "EPERM") {
      log.warn({ pid, sig }, "kill denied (not our process)");
      return false;
    }
    log.error({ pid, sig, code: e.code }, "kill failed unexpectedly");
    return false;
  }
}
```

## Open Questions

1. **How does the daemon get the per-agent `claude` subprocess PID?**
   - What we know: SDK's public types do not expose it (verified `sdk.d.ts:1218,1226`).
   - What's unclear: Whether `loadSdk().query()` returns any internal object with a `.pid` field at runtime (private API, unsafe to depend on).
   - Recommendation: Scan `/proc` for processes whose `ppid === process.pid` (daemon) and whose `cmdline[0]` ends in `/claude` or `claude`. Capture once per agent at start. Verify the count matches the number of running agents. If multiple matches, sort by start_time and take the most recent.

2. **Will the npm-wrapped MCP servers' process-group ID survive reparenting to PID 1?**
   - What we know: When a process is reparented to init, its pgid is preserved (POSIX). Verified by `cat /proc/3934295/stat` on clawdy → `pgrp=3828787` (the daemon's old PID, still preserved post-orphan).
   - What's unclear: After daemon restarts, the old daemon's PID (3828787) may be recycled. Process-group sigs to that pid would target unrelated processes.
   - Recommendation: For orphans (PPID=1), use individual `process.kill(pid, sig)`, NOT process-group signaling. The orphan reaper kills PIDs one at a time. Process-group form is for live MCP cleanup (MCP-02/04), where the pgid leader is still alive and tracked.

3. **Does mcporter (the new mysql MCP) behave the same as `npm exec mcp-server-mysql`?**
   - What we know: clawcode.example.yaml line 35 shows `command: mcporter` for the mysql server (different invocation than the orphan-producing `npm exec mcp-server-mysql@latest`).
   - What's unclear: Whether `mcporter serve mysql` has the same npm-wrapper-grandchild structure or spawns directly. Production trees still show `npm exec mcp-server-mysql@latest` (verified above), so prod hasn't migrated to mcporter yet.
   - Recommendation: Reaper regex should match both `mcp-server-mysql` AND `mcporter` cmdline patterns. Cheap insurance against future migration.
