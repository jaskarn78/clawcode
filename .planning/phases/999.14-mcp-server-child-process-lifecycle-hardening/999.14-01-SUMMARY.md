# Phase 999.14 Plan 01: Wire daemon + GREEN MCP-08/09/10 Summary

Wave 1 GREEN — wires Wave-0 substrate into the daemon (boot scan + reaper + shutdown), turns 39 RED tests for MCP-08/09/10 GREEN, and lands the operator escape hatches (CLI subcommands + IPC methods) that close today's two production incidents (orphan mcp-server-mysql + stale fin-acquisition thread bindings).

## Outcome

A daemon that, on `systemctl restart clawcode`, leaves zero orphan `mcp-server-mysql` procs with PPID=1 AND prunes registry bindings on Discord 50001/10003/404 AND runs a 60s stale-binding sweep AND offers operator CLI subcommands for thread inspection/archive/prune.

The daemon-wide McpProcessTracker singleton is constructed AFTER `secretsResolver.preResolveAll()` and BEFORE `manager.startAll()` (so the boot-time orphan scan runs first). The reaper interval starts after the daily summary cron and runs `reapOrphans` every 60s; its `onTickAfter` callback runs `sweepStaleBindings` AFTER the reap completes (locked decision). On daemon shutdown, the interval is cleared and `tracker.killAll(5s)` SIGTERMs every tracked PID with a grace period before SIGKILL.

Per-agent: `startAgent` waits 1s for npm-wrapper grandchildren to settle, walks `/proc` for the most-recently-started `claude` subprocess whose `ppid===daemonPid`, then enumerates MCP children of that PID and registers them. `stopAgent` and `onError` (crash) both call `tracker.killAgentGroup(name, 5s)` — idempotent on dead PIDs.

## Deviations from Plan

### Auto-fixed issues

**[Rule 1 - Bug] Test 13 (thread-cleanup) — write call expectation mismatch**
- **Found during:** Task 4 — running thread-cleanup tests after first impl.
- **Issue:** I added a "skip write if removeBinding returns same reference" optimization, but Test 13 mocks removeBinding to return identity; test expected writeThreadRegistry to ALWAYS be called.
- **Fix:** Removed the optimization — always write after removeBinding (Discord-side gone, want disk to reflect that even if registry didn't change in memory; idempotent on disk).
- **Files modified:** `src/discord/thread-cleanup.ts`
- **Verification:** All 17 tests GREEN.
- **Commit:** 1323f9c

**[Rule 3 - Blocking] Commander parent/child option collision — `threads --agent` vs `threads prune --agent`**
- **Found during:** Task 6 — running CLI tests; 4 failures around `--agent` flag.
- **Issue:** Commander by default propagates parent options to subcommands; the parent `threads`'s `-a, --agent` filter ate the subcommand's `--agent` parameter. Tests for `threads prune --agent fin-acq` saw `opts.agent === undefined` inside the prune handler.
- **Fix:** Called `program.enablePositionalOptions()` + `threads.passThroughOptions()` so subcommand options take precedence.
- **Files modified:** `src/cli/commands/threads.ts`
- **Verification:** All 10 CLI tests GREEN; existing formatter tests still pass.
- **Commit:** bcfe012

**[Rule 3 - Blocking] daemon.ts routeMethod: `log` not in scope**
- **Found during:** Task 5 — adding 2 new IPC handlers; tsc errors at lines 5487, 5507, 5527.
- **Issue:** `log` is a daemon-level closure variable inside `startDaemon`; routeMethod is a top-level function with no `log` in scope. The module-level `logger` import is the right replacement.
- **Fix:** Used `logger` (module-level pino) instead of `log` (daemon closure) inside the new IPC handlers.
- **Files modified:** `src/manager/daemon.ts`
- **Verification:** tsc count returned to 120 (no new errors).
- **Commit:** 1b7ccd2
