## Summary

This phase is a **pure-refactor + alias-pattern + tool-output-fix** trifecta. Nothing here requires new dependencies — the recommended path uses only already-installed primitives (MCP SDK, zod, discord.js webhook surface, vitest).

**Investigation confirms:**

1. **Call-site inventory is complete.** A repo-wide grep for `sendToAgent` outside test files finds exactly 7 production call sites — matching CONTEXT.md verbatim. There are 4 additional doc-comment mentions in production files that should be refreshed but are not call sites. There are 14 test files that mock `sendToAgent` by name — these MUST be updated alongside the rename or vitest will fail with `vi.fn()` typing errors.
2. **MCP SDK does NOT support tool aliases natively.** `server.tool(name, ...)` throws `Tool ${name} is already registered` on duplicate name. The canonical alias pattern is to extract the handler into a shared closure and call `server.tool()` twice with different names. Tools are listed in `Object.entries(this._registeredTools)` insertion order — so registering the new name BEFORE the deprecated name controls D-RNX-04's LLM-picking ordering.
3. **The smoking-gun bug is a single line.** `src/mcp/server.ts:282-294` destructures `{ok, messageId}` from the IPC result and discards the `response` field even though the IPC handler at `daemon.ts:4188-4204` already returns it (when escalation isn't triggered). Plan 03 D-SYN-02 is fundamentally a one-line text-template change in the MCP wrapper, gated on `response` becoming non-undefined post-error-propagation.
4. **The webhook-mirror infrastructure already exists.** `WebhookManager.sendAsAgent` (webhook-manager.ts:96) is exactly the API needed — no new code paths. `buildAgentMessageEmbed` (agent-message.ts:14) is reusable for the prompt; the response embed uses the same builder with target's identity as sender (cleanest approach — avoids a sibling builder and keeps the embed schema uniform).
5. **Pitfall 5 from Phase 103 does NOT apply.** That pitfall was about a name like `list-rate-limit-snapshots` colliding with an existing `rate-limit-status`. Here both old and new names (`send-message`, `ask-agent`, `send-to-agent`, `post-to-agent`) are unambiguous and don't collide with anything in `IPC_METHODS`. Confirmed by reviewing the full array at `src/ipc/protocol.ts:7-221`.

**Primary recommendation:** Execute the three plans in strict dependency order (Plan 01 → 02 → 03). Plan 01 is purely mechanical (rename + test mock updates). Plan 02 introduces the shared-handler factoring. Plan 03 is the actual bug-fix delivery and is the only plan with non-trivial behavior change.

### Pattern 1: MCP tool alias via shared-handler closure

**What:** Extract the handler body into a function/closure, then call `server.tool()` twice with the same closure but different names + descriptions.

**When to use:** Every time you need two MCP tool names to behave identically (the canonical pattern enforced by the SDK throwing on duplicate registration).

**Example (drawn from existing repo style — `src/mcp/server.ts`):**

```typescript
// Source: SDK behavior verified in node_modules/@modelcontextprotocol/sdk/dist/esm/server/mcp.js:657-660
// (`if (this._registeredTools[name]) throw new Error('Tool ${name} is already registered')`)

// Shared schema reused by both registrations.
const askAgentSchema = {
  to: z.string().describe("Target agent name"),
  content: z.string().describe("Message content"),
  from: z.string().default("mcp-client").describe("Sender name"),
  priority: z.enum(["normal", "high", "urgent"]).default("normal").describe("Message priority"),
  mirror_to_target_channel: z.boolean().default(false).describe(
    "When true, post the prompt + response as embeds in target's Discord channel for an audit trail."
  ),
} as const;

// Shared handler (Plan 03 — sync-reply behavior).
const askAgentHandler = async ({
  to, content, from, priority, mirror_to_target_channel,
}: { to: string; content: string; from: string; priority: string; mirror_to_target_channel: boolean }) => {
  try {
    const result = (await sendIpcRequest(SOCKET_PATH, "ask-agent", {
      from, to, content, priority, mirror_to_target_channel,
    })) as { ok: boolean; messageId: string; response?: string };

    if (result.response !== undefined) {
      return {
        content: [{
          type: "text" as const,
          text: `Message sent to ${to} (id: ${result.messageId})\n\n${to} replied:\n${result.response}`,
        }],
      };
    }
    return {
      content: [{
        type: "text" as const,
        text: `Message queued in ${to}'s inbox. ${to} is not running — no synchronous reply.`,
      }],
    };
  } catch (err) {
    const msg = err instanceof Error ? err.message : String(err);
    return {
      content: [{ type: "text" as const, text: `Failed to ask ${to}: ${msg}` }],
    };
  }
};

// Register canonical name FIRST so it appears first in tools/list (D-RNX-04).
server.tool(
  "ask_agent",
  "Ask another ClawCode agent a question and receive their reply synchronously. " +
    "Use this when you need a response — e.g., 'fin-acquisition, what's our LTV target?'. " +
    "Set mirror_to_target_channel=true to post the Q+A as embeds in the target's channel for visibility.",
  askAgentSchema,
  askAgentHandler,
);

// Register deprecated alias SECOND.
server.tool(
  "send_message",
  "[DEPRECATED — use ask_agent instead] " +
    "Send a synchronous message to another ClawCode agent and receive their reply. " +
    "Identical to ask_agent; kept for backwards compatibility.",
  askAgentSchema,
  askAgentHandler,
);
```

**Key points verified against the SDK:**
- The deprecated description prefix is purely string content — no SDK-level deprecation flag exists in `tool()` (registerTool has an `annotations` field but the existing repo style doesn't use it; staying consistent).
- `Object.entries(this._registeredTools)` (mcp.js:69) iterates in insertion order, so first-registered = first-listed. Verified.
- The same closure (object identity) being passed twice is fine — each registration wraps it in its own RegisteredTool entry; calling either dispatches to the same closure.

### Pattern 2: IPC alias via switch-case fall-through

**What:** Add the new method name as a sibling case clause that uses fall-through to the existing handler, OR delegate explicitly.

**When to use:** Whenever an IPC method is renamed and back-compat is required.

**Example (mirrors existing `case "..."` blocks in daemon.ts:4017-...):**

```typescript
// Source: src/manager/daemon.ts case-statement pattern (no fall-through used elsewhere
// in the file — every case has a return). Recommended explicit-delegation pattern:

case "ask-agent":
case "send-message": {  // DEPRECATED — kept for back-compat (Phase 999.2 D-RNI-IPC-01)
  const from = validateStringParam(params, "from");
  const to = validateStringParam(params, "to");
  const content = validateStringParam(params, "content");
  const priority = typeof params.priority === "string" ? params.priority : "normal";
  const mirror = params.mirror_to_target_channel === true; // Plan 03 D-SYN-04

  // ... (existing handler body — Plan 03 modifies it for sync-reply behavior) ...
}

case "post-to-agent":
case "send-to-agent": {  // DEPRECATED — kept for back-compat (Phase 999.2 D-RNI-IPC-02)
  // ... existing handler body unchanged (broadcast semantic preserved) ...
}
```

**Why stacked-case (not fall-through with explicit jumps):** The existing daemon.ts uses `case "x": { ...; return ...; }` with an internal `return`, never a `break` with fall-through. Stacked case-labels (which JavaScript treats as both labels pointing at the same block) match this style and don't introduce a new pattern.

### Pattern 3: Webhook embed mirror (D-SYN-04)

**What:** When `mirror_to_target_channel=true`, post two webhook embeds in target's channel — pre-dispatch with caller's identity, post-dispatch with target's identity.

**When to use:** Only inside the `ask-agent` IPC handler, only when target is running, only when caller-supplied flag is true. Best-effort: webhook errors during the mirror MUST NOT fail the underlying ask.

**Example:**

```typescript
// Source: webhook-manager.ts:96-119 (sendAsAgent signature) + agent-message.ts:14 (embed builder)

// Pre-dispatch — prompt embed posted FROM caller TO target's channel.
if (mirror && webhookManager.hasWebhook(to)) {
  try {
    const senderConfig = configs.find((c) => c.name === from);
    const senderDisplayName = senderConfig?.webhook?.displayName ?? from;
    const senderAvatarUrl = senderConfig?.webhook?.avatarUrl;
    const embed = buildAgentMessageEmbed(from, senderDisplayName, content);
    await webhookManager.sendAsAgent(to, senderDisplayName, senderAvatarUrl, embed);
  } catch (err) {
    // Best-effort — webhook failure does NOT abort the ask.
    log.warn({ from, to, err: (err as Error).message }, "[ask-agent] mirror prompt webhook failed");
  }
}

// dispatch ...
const response = await manager.dispatchTurn(to, content);

// Post-dispatch — response embed posted FROM target TO target's channel
// (target's identity = same channel, target's webhook = self-attribution).
if (mirror && response && webhookManager.hasWebhook(to)) {
  try {
    const targetConfig = configs.find((c) => c.name === to);
    const targetDisplayName = targetConfig?.webhook?.displayName ?? to;
    const targetAvatarUrl = targetConfig?.webhook?.avatarUrl;
    // Reuse buildAgentMessageEmbed; passing `to` as senderName makes the
    // footer read "Agent-to-agent message from {to}" — semantically correct
    // because the EMBED is from the target acknowledging the caller.
    // Optional: prefix description with `> reply to ${from}:\n\n` to make
    // the threading explicit. (Recommended — see Pitfall 4 below.)
    const embedBody = `> reply to ${from}:\n\n${response}`;
    const embed = buildAgentMessageEmbed(to, targetDisplayName, embedBody);
    await webhookManager.sendAsAgent(to, targetDisplayName, targetAvatarUrl, embed);
  } catch (err) {
    log.warn({ from, to, err: (err as Error).message }, "[ask-agent] mirror response webhook failed");
  }
}
```

### Anti-Patterns to Avoid

- **Re-registering an MCP tool by mutating `_registeredTools` directly.** The SDK's private map is not part of the supported API; bypassing the `server.tool()` throw means losing list-changed notifications and breaking any future SDK upgrade. Use the shared-closure pattern instead.
- **Using `break` + fall-through in the daemon switch.** No other case uses fall-through; introducing it creates a stylistic inconsistency. Stacked `case` labels (single block) is the right shape.
- **Deleting the `try {...} catch {}` block at daemon.ts:4205-4208 without reading what it was hiding.** D-SYN-05 requires this — but the removal MUST land in Plan 03 ALONGSIDE the test that pins error propagation. Removing it earlier (in Plan 01 or 02) breaks Plan 03's TDD discipline.
- **Sharing a `mirror_to_target_channel` flag with `post_to_agent`.** D-PST-01 specifies `post_to_agent` is unchanged. Adding the param to that tool would conflate the semantics. Keep the flag exclusive to `ask_agent` / `ask-agent`.

## Common Pitfalls

### Pitfall 1: Test files mock `sendToAgent` by name (~14 files)

**What goes wrong:** After renaming the production method to `dispatchTurn`, ANY test file with `sessionManager.sendToAgent: vi.fn(...)` will pass type-check (the mock is just a fake-typed object), but assertions like `expect(sessionManager.sendToAgent).toHaveBeenCalledWith(...)` will pass against a stale mock that the production code never calls — leading to FALSE POSITIVES.

**Why it happens:** vitest mocks don't enforce real method names; the property is just a key in the mock object.

### Pitfall 3: MCP tool list ordering controls LLM behavior (D-RNX-04)

**What goes wrong:** If the deprecated `send_message` is registered BEFORE `ask_agent`, LLM tool-picking heuristics may prefer the older name (because trained on prior data + appears first). Users won't see the rename benefit.

**Why it happens:** `Object.entries(this._registeredTools)` iterates in insertion order (verified at `node_modules/@modelcontextprotocol/sdk/dist/esm/server/mcp.js:69`).

**How to avoid:** Register `ask_agent` and `post_to_agent` BEFORE their deprecated aliases in `src/mcp/server.ts`. Plan 02 should add a static-grep test that asserts the line number of `server.tool("ask_agent",...)` is LESS than the line number of `server.tool("send_message",...)` — guaranteeing insertion order at file-edit time.

**Warning signs:** After deploy, agents continue using the old name. (Confirm via daemon log inspection — every IPC call logs the method name.)

### Pitfall 4: Mirror response embed has no implicit thread/reply marker

**What goes wrong:** The two mirror embeds (prompt + response) appear in target's channel as two independent webhook posts. Without an explicit "reply to" marker, an operator scrolling target's channel sees a question and an unrelated answer with no visual link.

**Why it happens:** Webhooks in Discord can't natively reply-thread (no `message_reference`); embeds can only convey the relationship via content.

**How to avoid:** Recommended approach: in the response embed, prepend the description with `> reply to {from}:\n\n` (Discord blockquote rendering). This reuses `buildAgentMessageEmbed` with the response body augmented at the call-site — no schema change. Tests should pin this prefix on the response embed.

**Alternative considered:** Add an `inReplyTo` field via `EmbedBuilder.addFields(...)`. Rejected — diverges from the existing single-description embed shape.

### Pitfall 5: `IPC_METHODS` collision (Phase 103 reference)

**What goes wrong:** Phase 103 had a near-collision between `rate-limit-status` and `list-rate-limit-snapshots`. Could `ask-agent` or `post-to-agent` collide with anything?

**Verification:** Full review of `src/ipc/protocol.ts:7-221` — no method named `ask-*` exists; no method named `post-*` exists. The closest substring match is `ask-advisor` (Phase 39 model tiering) which is unambiguously distinct from `ask-agent`. **No collision risk.**

**Sanity test:** Plan 02 includes a unit test asserting all 4 names (`send-message`, `ask-agent`, `send-to-agent`, `post-to-agent`) appear EXACTLY ONCE in `IPC_METHODS`.

### Pitfall 6: Removing the silent-error catch may surface previously-hidden flakiness

**What goes wrong:** The current `try {...} catch {}` block at `daemon.ts:4205-4208` swallows EVERY exception from `manager.sendToAgent` — including transient SDK errors, abort signals, and the "shutting down, agent X" SessionError. Removing it (D-SYN-05) means ANY of these errors now propagate as `IpcError` to the caller. This is correct (operator visibility) BUT may surface latent bugs that the catch was hiding.

**Why it happens:** Silent error catches accumulate over time. Each one masks a different failure mode.

**How to avoid:**
1. Plan 03 Task 03 should add explicit tests for each error class — `dispatchTurn rejects → IPC returns IpcError`, `dispatchTurn aborted → IPC returns IpcError`, `target offline → IPC returns ok with no response (NOT an error)`.
2. Plan 03 should ALSO audit recent daemon logs (last 7 days) for any `[send-message]` or `[ask-agent]` failures to estimate the surface — if there's a known-flaky path (e.g., MCP server warm-up race), the plan should call it out.
3. The escalation path (existing logic — escalation triggers on error indicators in the response, NOT on a thrown exception) is UNCHANGED. Verify: the silent catch was wrapping `escalationMonitor.escalate` calls too — when escalation itself throws (budget exceeded, fork failed), today the caller saw `{ok: true}`. Post-Plan-03 they will see the budget error. Operator may need to update agent SOULs to expect this.

**Warning signs:** Sudden spike in `Failed to ask {to}: ...` messages in caller channels post-deploy that weren't there before. Diagnose with daemon log archeology, not by reinstating the catch.

### Pitfall 7: `mirror_to_target_channel` failure should NOT fail the ask

**What goes wrong:** If the target has no webhook configured (or webhook delivery throws), naively wrapping the mirror around the dispatch fails the WHOLE call, losing the response.

**Why it happens:** Operator-set flag triggers an OPTIONAL side-effect; that side-effect should never gate the primary purpose.

**How to avoid:** Wrap each webhook call in its OWN `try/catch` and log warnings (mirroring `case "send-to-agent"` at daemon.ts:4257-4262, which already does this). The dispatch + response logic must be the only code path that can fail the IPC call. Test this explicitly in Plan 03: "mirror=true + target has no webhook → ask still succeeds with response."

### Pitfall 8: Schema drift between `ask-agent` IPC params and the wire format

**What goes wrong:** The IPC handler reads `params.priority`, `params.mirror_to_target_channel`, etc. There's no zod schema for IPC PARAMS — only for the request envelope (`ipcRequestSchema`). If the MCP wrapper sends `mirror_to_target_channel` but the daemon reads `mirrorToTargetChannel`, no validation catches it.

**Why it happens:** Existing IPC params use `validateStringParam` (manual). No zod for the params payload.

**How to avoid:** Test pins both sides — one MCP-server test sending `mirror_to_target_channel` and asserting the IPC payload field name; one daemon-handler test asserting the read uses the same field name. Snake-case is the existing convention (`from`, `to`, `content`, `priority` already exist) — keep snake_case for `mirror_to_target_channel` and DO NOT camelCase it.

## Open Questions

1. **Should the `ask_agent` MCP tool description mention the `mirror_to_target_channel` flag prominently?**
   - What we know: D-SYN-04 specifies the flag exists with default `false`. D-RNX-01 says descriptions clarify the rename.
   - What's unclear: Whether the description should describe WHEN to use the flag (audit trails, debugging, multi-party visibility) or just list it as a parameter.
   - Recommendation: Include a one-line "Set mirror_to_target_channel=true to post the Q+A as embeds in {to}'s channel for visibility." in the tool description. LLMs read tool descriptions during action selection — adding the use-case helps them pick the flag correctly.

2. **Do we need a metric/counter for alias usage?**
   - What we know: D-RNX-03 says aliases stay 30+ days then a quick-task removes them. The decision-trigger is "operator confirms no agent SOUL/SKILL still references them."
   - What's unclear: How does the operator confirm? By checking daemon logs for `case "send-message"` hits, or by some explicit metric?
   - Recommendation: Plan 02 adds a one-line `log.info({alias: "send-message"}, "deprecated IPC alias used")` in the deprecated branches. Operator runs `grep deprecated.*alias.*used ~/.clawcode/logs/manager.jsonl` after 30 days; if zero hits → safe to remove.

3. **Does the escalation path interact with the silent-catch removal?**
   - What we know: Today the catch wraps `escalationMonitor.escalate(to, content)` too — if escalation itself throws (budget exceeded, fork stop fails), caller sees `{ok: true, messageId}` with no error.
   - What's unclear: Whether operators have come to rely on this — i.e., a budget-exceeded escalation today silently succeeds; post-Plan-03 it errors.
   - Recommendation: Plan 03 verification step manually triggers an escalation budget overflow in a test agent and confirms the error is now visible. Document the change in the phase's deploy notes.
