<domain>
## Phase Boundary

Fix the agent-to-agent comms architectural debt surfaced 2026-04-29 when admin-clawdy → fin-acquisition produced no reply back. The phase covers TWO of the three originally-scoped concerns and EXPLICITLY DEFERS the third to a future phase:

**In scope (this phase):**

1. **Rename + back-compat aliases (Option C).** Three layers of naming-debt cleanup:
   - **Internal method**: `SessionManager.sendToAgent(name, content)` → `SessionManager.dispatchTurn(name, content)`. The method dispatches a turn and awaits the result — the "to agent" framing was always misleading (heartbeat/inbox uses it on its own agent; escalation uses it on a fork). 7 production call sites.
   - **MCP tool: synchronous Q&A**: `send_message` → `ask_agent`. The semantic IS asking + receiving; "send_message" undersells it. Backwards-compat alias lets old agents keep working until they're updated.
   - **MCP tool: async broadcast**: `send_to_agent` → `post_to_agent`. The semantic IS posting in target's channel for visibility; "send_to_agent" implied a reply that never came. Backwards-compat alias preserved.
   - **IPC methods**: `send-message` → `ask-agent`, `send-to-agent` → `post-to-agent`. Daemon handler accepts both old + new names.

2. **v2 sync-reply behavior** on `ask_agent`:
   - **Response in tool result**: caller's LLM sees target's reply in the tool result text (today: silently discarded — the smoking-gun bug found 2026-04-29).
   - **`mirror_to_target_channel: boolean` flag** (default false): when true, post Q+A as webhook embeds in target's Discord channel for visible audit trail.
   - **Error propagation**: stop swallowing `dispatchTurn` errors silently; surface to caller as a tool error so the LLM can react.

**Out of scope — explicitly deferred to follow-up phase:**

3. **Async correlation IDs**: alternative non-blocking pattern where caller fires-and-forgets, target's reply auto-posts back to caller's channel via correlation-ID lookup. Requires reply hook in turn-dispatcher + per-message correlation tracking + Discord routing for the relayed reply. Bigger redesign — files as **backlog 999.8** post-promotion. The synchronous v2 (in scope above) already solves the operator-blocking pain; correlation IDs are a polish/pacing optimization.

</domain>

<decisions>
## Implementation Decisions

### Rename strategy
- **D-RNI-01:** `SessionManager.sendToAgent` → `SessionManager.dispatchTurn`. All 7 production call sites updated:
  - `src/manager/turn-dispatcher.ts:582, 597`
  - `src/manager/daemon.ts:1895, 4188, 4967, 5591`
  - `src/manager/escalation.ts:127`
  - `src/heartbeat/checks/inbox.ts:90`
- **D-RNI-02:** No alias for the internal method — pure rename. Internal method has no external API consumers; tests update simultaneously. Comment in session-manager.ts notes the rename + Phase 999.2 reference.
- **D-RNI-03:** Method signature unchanged: `dispatchTurn(name: string, content: string, turn?: Turn, opts?: {signal?: AbortSignal}): Promise<string>`. Same param ordering, same return type. Only the name changes.

### MCP tool renames + back-compat aliases
- **D-RNX-01:** `send_message` MCP tool renamed to `ask_agent`. The OLD name remains registered as an alias that calls into the new handler. Tool descriptions updated to clearly state "ask_agent (preferred) — send_message is the deprecated alias for backwards compat".
- **D-RNX-02:** `send_to_agent` MCP tool renamed to `post_to_agent`. Same alias treatment as above.
- **D-RNX-03:** Alias deprecation horizon: aliases stay for at least 30 days post-deploy, then a dedicated quick-task removes them once operator confirms no agent SOUL/SKILL still references them. Not a hard time-bomb; soft cleanup later.
- **D-RNX-04:** New tool names appear FIRST in the registered tool list so the LLM is more likely to pick them. Old names remain registered but their description starts with `[DEPRECATED — use ask_agent instead]` / `[DEPRECATED — use post_to_agent instead]`.

### IPC method renames + back-compat
- **D-RNI-IPC-01:** `send-message` IPC method renamed to `ask-agent`. Daemon's `routeMethod` switch keeps the old `send-message` case as a passthrough that calls the new handler. Both work.
- **D-RNI-IPC-02:** `send-to-agent` IPC method renamed to `post-to-agent`. Same passthrough treatment.
- **D-RNI-IPC-03:** `IPC_METHODS` array in `src/ipc/protocol.ts` includes BOTH old and new names so existing IPC clients (CLI tools, MCP wrappers) keep working. Static-grep regression tests pin both as registered.
- **D-RNI-IPC-04:** Static-grep tests pin the deprecated-alias annotations in the IPC `protocol.ts` definition (`// DEPRECATED — use ask-agent`).

### v2 sync-reply behavior on ask_agent
- **D-SYN-01:** When target agent is running, `ask_agent` IPC handler dispatches a turn (via `dispatchTurn` after rename) and **synchronously waits for the response**. The response field of the IPC return is populated.
- **D-SYN-02:** **MCP tool wrapper** for `ask_agent` includes the response in the returned text:
  ```
  Message sent to {to} (id: {messageId})

  {to} replied:
  {response}
  ```
- **D-SYN-03:** When target is NOT running, the IPC handler still writes the inbox file (existing behavior preserved) but returns `{ok: true, messageId, response: undefined}`. MCP tool returns: `Message queued in {to}'s inbox. {to} is not running — no synchronous reply.` This makes the offline case explicit to the calling LLM.
- **D-SYN-04:** New `mirror_to_target_channel: boolean` (default `false`) param accepted at MCP + IPC layers. When `true` AND target was running:
  1. Before turn dispatch, post a webhook embed in target's Discord channel from caller's identity showing the prompt.
  2. After response received, post a webhook embed in target's channel from target's identity showing the response.
  3. Both posts use the existing `webhookManager.sendAsAgent` pattern.
- **D-SYN-05:** Errors from `dispatchTurn` (target throws, signal aborted, MCP server unreachable, etc.) are NO LONGER caught + swallowed in the IPC handler. They propagate as `IpcError` → MCP tool wrapper surfaces as plain text: `Failed to ask {to}: {error.message}`. Operator (and the calling LLM) sees the failure.
- **D-SYN-06:** Phase 99-M auto-relay infra is unchanged — `ask_agent` synchronous reply complements (does not replace) the subagent autoRelay.

### `post_to_agent` (the broadcast tool) — minimal change
- **D-PST-01:** `post_to_agent` keeps its current behavior: writes inbox + posts webhook embed in target's channel from caller's identity. NO synchronous reply path added (it's the broadcast/fire-and-forget tool by design).
- **D-PST-02:** Tool description updated to be clear: *"Broadcast a message to another agent's Discord channel. The agent sees it as a normal message. Does NOT wait for a reply — for synchronous Q&A use ask_agent."*
- **D-PST-03:** Errors are NOT silently swallowed (mirroring D-SYN-05) — webhook delivery failure propagates as IpcError so caller sees it.

### Plan structure
- **D-PLN-01:** 3 plans, each TDD-disciplined:
  - **Plan 01:** Rename internal `SessionManager.sendToAgent` → `dispatchTurn` (7 production call sites + tests). Pure rename, no behavior changes. Wave 1.
  - **Plan 02:** Rename MCP tools + IPC methods with back-compat aliases. New tool names registered, old names registered as aliases pointing to new handlers. IPC `protocol.ts` includes both. Tests pin both old + new names work. Wave 2 (depends on Plan 01).
  - **Plan 03:** v2 sync-reply behavior on `ask_agent` — response in tool result, `mirror_to_target_channel` flag, error propagation. Tests pin all three sub-behaviors. Wave 3 (depends on Plan 02).
- **D-PLN-02:** NO production deploy step in any plan. Operator confirms deploy explicitly.
- **D-PLN-03:** Async correlation IDs (D-CORR-01 below) are deferred — NOT a plan in this phase.

### Async correlation IDs (deferred to future phase)
- **D-CORR-01:** Async non-blocking ask pattern requires:
  - Per-message correlation IDs persisted somewhere (likely a per-agent tasks.db extension or new correlations table)
  - Reply hook in turn-dispatcher that detects "this turn was triggered by correlation X" and routes the reply back to caller's channel as a webhook embed
  - Tool variant `ask_agent_async` or a `block: false` flag on `ask_agent`
  - Discord-side: caller's channel sees the relayed reply as an embed, possibly with reply-to threading
  
  Estimated 3-4 plans of work. Defer to a backlog item (will file post-promotion as Phase 999.8 — "Async correlation IDs for non-blocking ask_agent").
