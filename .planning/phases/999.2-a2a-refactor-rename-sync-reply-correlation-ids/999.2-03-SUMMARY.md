# Phase 999.2 Plan 03: v2 Sync-Reply Behavior on `ask_agent` Summary

**Fixed the 2026-04-29 smoking-gun bug — `ask_agent` MCP tool now surfaces the target agent's reply in the tool-result text — by extracting the daemon `case "ask-agent":` body into a pure-DI handler (mirroring the Phase 103 blueprint) that adds the `mirror_to_target_channel` flag for optional Q+A webhook embeds in target's channel, propagates `dispatchTurn` errors instead of silently swallowing them, and renders explicit offline text when the target isn't running. The previous silent `try {...} catch {}` at daemon.ts:4205-4208 is gone; webhook failures during the mirror are best-effort logged but never abort the ask. Escalation behavior is preserved unchanged at the daemon edge.**

## The Bug-Fix Diff

The 2026-04-29 smoking-gun bug at `src/mcp/server.ts:282-294`:

```typescript
// BEFORE (Plan 02 — what the bug looked like):
const result = (await sendIpcRequest(SOCKET_PATH, "ask-agent", {
  from, to, content, priority,
})) as { ok: boolean; messageId: string };  // <-- DISCARDED `response` field

return {
  content: [{ type: "text" as const, text: `Message sent to ${to} (id: ${result.messageId})` }],
};
```

```typescript
// AFTER (Plan 03 — bug fixed):
const result = (await sendIpcRequest(SOCKET_PATH, "ask-agent", {
  from, to, content, priority, mirror_to_target_channel,
})) as { ok: boolean; messageId: string; response?: string };  // <-- response now read

if (result.response !== undefined) {
  // D-SYN-02 — surface the reply in the tool-result text. THE BUG-FIX LINE.
  return {
    content: [{
      type: "text" as const,
      text: `Message sent to ${to} (id: ${result.messageId})\n\n${to} replied:\n${result.response}`,
    }],
  };
}
// D-SYN-03 — offline target.
return {
  content: [{
    type: "text" as const,
    text: `Message queued in ${to}'s inbox. ${to} is not running — no synchronous reply.`,
  }],
};
```

The IPC handler at `daemon.ts:4188-4204` was already returning `response` correctly when escalation didn't trigger — only the MCP wrapper destructure was discarding it. Plan 03 fixes the wrapper AND removes the silent error-catch downstream.

## The Silent-Error-Catch Removal Diff (D-SYN-05)

```typescript
// BEFORE (daemon.ts:4202-4224 — the error masker):
if (running.includes(to)) {
  try {
    let response = await manager.dispatchTurn(to, content);
    // ... escalation check ...
    return { ok: true, messageId: message.id, response, escalated };
  } catch {
    // Direct send failed -- inbox write already succeeded, return ok
    return { ok: true, messageId: message.id };  // <-- ANY error → false-success
  }
}
```

```typescript
// AFTER (daemon.ts — delegates to pure-DI handler that does NOT catch):
const { handleAskAgentIpc } = await import("./daemon-ask-agent-ipc.js");
const askResult = await handleAskAgentIpc(
  { from, to, content, priority, mirror_to_target_channel: mirror },
  { runningAgents: manager.getRunningAgents(), dispatchTurn: ..., writeInbox: ..., webhookManager, configs, log: console-shim },
);
// errors from dispatchTurn now propagate as IpcError → MCP wrapper renders
// `Failed to ask ${to}: ${msg}` so the caller sees the failure.
```

The pure-DI handler has NO try/catch around `await deps.dispatchTurn(...)`. Mirror webhook calls have their own narrow try/catch (best-effort, never abort the ask).

## Decisions Made

- **Static-grep over McpServer build-and-extract for the wrapper tests.** The MCP wrapper is a deterministic text-template + try/catch around `sendIpcRequest`. The pure-DI handler tests already pin the IPC contract behavior (response shape, error propagation, mirror calls). Adding McpServer construction + handler-closure extraction via `_registeredTools` private API would have required hoisting `vi.mock("../ipc/client.js")` at the top of server.test.ts (which would affect the entire 417-line existing test file's import graph). Static-grep is the established Plan 02 precedent in the same file.
- **Escalation kept at the daemon edge.** `escalationMonitor.escalate` needs `SessionManager` to fork a session and re-dispatch. Pushing it into the pure-DI handler would have inflated the deps surface significantly (manager-shaped object with fork primitives) for one pre-existing path that has its own regression coverage. Plan 03 D-SYN-06 ("escalation path unchanged") is satisfied: indicator-text scan + escalate call live at daemon.ts in the case body, AFTER the handler returns.
- **Console-shim logger for the dynamic-import call site.** `routeMethod` is a top-level function (line 3990); the daemon-scoped pino `log` lives inside `startDaemon`'s closure. Threading it through `routeMethod`'s 25-arg signature is far outside Plan 03 scope. The handler's `log.warn` calls go through `console.warn` — same pattern Plan 02 used for the deprecated-alias metric, same pattern as the existing `case "post-to-agent"` webhook-failure log at daemon.ts:4322-4324.
- **Dynamic import for the new module.** Matches daemon-rate-limit-ipc.ts (4096), daemon-fs-ipc.ts (2638, 2702). Lazy-loads only when an ask-agent / send-message IPC dispatch fires.
- **Snake-case wire field `mirror_to_target_channel`.** Existing IPC params (from, to, content, priority) are all snake-case-equivalent. Pitfall 8 explicitly warns against camelCase drift between MCP wrapper (snake) and IPC handler (camel). Test #9 statically pins the exact field name in the schema.
- **Escalation result shape preserved (`escalated: true|false`).** Production callers may already inspect this — the daemon edge returns `{ok: true, messageId, response, escalated}` matching the pre-Plan-03 envelope. The pure-DI handler returns just `{ok, messageId, response}`; the daemon edge layers `escalated` on top.
