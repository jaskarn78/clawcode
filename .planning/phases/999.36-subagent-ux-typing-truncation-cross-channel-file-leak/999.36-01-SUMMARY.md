# Phase 999.36 Plan 01: Sub-bug C cross-channel file leak — agent-identity routing rebind — Summary

**One-liner:** P0 correctness fix routing `clawcode_share_file` through the thread binding registry by `sessionName` FIRST, eliminating the shared-workspace leak class that landed fin-acquisition's Schwab AIP markdown in finmentum-content-creator's primary channel on 2026-05-06.

## What Was Built

### 1. Pure registry-lookup helper (Task 1)

`src/discord/thread-registry.ts` — added `getBindingForSession(registry, sessionName)`. Mirrors the existing `getBindingForThread` / `getBindingsForAgent` patterns at lines 147-165: 30 LoC including JSDoc, no I/O, no logging, no mutations. Disambiguates the failure class where two siblings in a shared workspace have the same `agentName` field on their bindings but distinct `sessionName` fields.

### 2. Pure channel-resolution helper (Task 2)

`src/manager/tools/share-file-channel-resolver.ts` (new, 58 LoC). Resolution order:

1. Operator override (`channel_id` param) wins.
2. Subagent thread binding (`sessionName` lookup) → `binding.threadId`.
3. `agentConfig.channels[0]` fallback (regular agents — pre-Phase-999.36 behavior).
4. `undefined` → caller refuses with structured error.

Daemon IPC handler at `src/manager/daemon.ts:6689-6788` (case "share-file") rewired:

- Reads `threadRegistryForShare` ONCE.
- Looks up `subagentBinding = getBindingForSession(threadRegistryForShare, agentName)`.
- Uses `subagentBinding.agentName` (the PARENT) as the `configs.find()` lookup key when it exists, falling back to `agentName` itself for regular invocations.
- Calls `resolveShareFileChannel(agentName, channelIdParam, threadRegistryForShare, agentConfig)` for the final channel.
- Refused-cases now include `"no thread binding"` in the error message so misconfigured agents surface deterministically.

### 3. Regression test — shared-workspace invariant (Task 3)

The `expect(resolved).not.toBe(CONTENT_CREATOR_CHANNEL)` anti-leak assertion is the load-bearing pin per D-11.

### 4. Defense-in-depth prompt instruction (Task 4)

`src/discord/subagent-thread-spawner.ts:540-552` — added two lines to `baseContext`:

```typescript
`Your session name: ${sessionName}`,

`When calling clawcode_share_file, pass agent="${sessionName}" (your session name above) — NOT "${config.parentAgentName}". This routes the upload to THIS thread.`,
```

The defense matrix per LLM-supplied `agent` parameter:

| LLM passes | Layer 1 (prompt) | Layer 2 (daemon resolver) | Result |
|---|---|---|---|
| sessionName (intended) | catches | binding lookup → bound thread | no leak |
| parent name (drift) | drifts | falls to `parent.channels[0]` (= Ramy's thread by coincidence for finmentum) | no leak |
| **sibling name (actual incident)** | **drifts** | **falls to `sibling.channels[0]`** | **LEAK without Layer 1** |

Layer 1 (prompt) catches all three drift paths. Layer 2 (resolver) catches only the sessionName path; the parent-name path is saved by coincidence (parent's `channels[0]` happens to equal the bound thread for fin-acquisition); the sibling-name path — which IS the 2026-05-06 incident path — relies on Layer 1 alone. This is documented in the threat model: Layer 2's resolver is a defense for the *sessionName-correct* case, while Layer 1 is the actual barrier against the sibling-name drift that produced the incident.

## Approach taken

**Approach A (pure helper extraction) — committed upfront per advisor reconciliation.** Avoids Task 3 having to refactor inline logic into a helper just so it can be tested. The trade-off is a new file (`share-file-channel-resolver.ts`) versus a 4-line inline change in daemon.ts; the helper file is preferable because:

- It's pure (no I/O, no `await readThreadRegistry()` inside) → testable without daemon spin-up.
- Documents the resolution order in JSDoc near the code.
- Future operator-override paths (e.g. config-driven routing) extend cleanly.

The daemon's `case "share-file"` block reads the registry ONCE and threads it through both the agent-config lookup AND the channel resolver — single I/O round-trip, no race window between the two consumers.

## Backwards compatibility

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Daemon agent-config lookup needed sessionName indirection**

- **Found during:** Task 2 (daemon wire-in)
- **Issue:** The plan's `<action>` block specified `agentConfig = configs.find((c) => c.name === agentName)` would run BEFORE the binding lookup. Subagents are spawned dynamically and their sessionNames are NOT in the static `configs[]` array. Following the plan literally would have made every subagent's `clawcode_share_file` call fail with `"Agent 'fin-acquisition-sub-OV9rkf' not found in config"`. Task 4's prompt instruction (LLM passes sessionName) would have made the bug 100% reproducible.
- **Fix:** Read the registry FIRST. If `getBindingForSession(registry, agentName)` returns a binding, use `binding.agentName` (the PARENT's name) as the `configs.find()` key. Subagents legitimately inherit the parent's workspace + memoryPath + fileAccess for the security gate (allowedRoots) — only channel routing differs.
- **Files modified:** `src/manager/daemon.ts` (case "share-file" reorder)
- **Commit:** `73417c1`
- **Verification:** new test added in `clawcode-share-file-shared-workspace.test.ts` exercises this exact path; existing `clawcode-share-file.test.ts` (14 tests) still pass — backwards compat for regular agents.

### Minor implementation choices (within Claude's discretion)

1. **Test count:** plan asked for 4 regression tests; shipped 5. The 5th covers the daemon's `if (!channelId)` refusal path at the helper boundary. Coverage gain, no acceptance-criteria conflict.

2. **Anti-leak assertion present in 2 places:** Test 1 (Schwab AIP failure class) AND Test 4 (explicit override). Test 4's `not.toBe(CONTENT_CREATOR_CHANNEL)` is defense-in-depth — the override path explicitly rules out silent fallthrough.

After Rule 3 deviation surfaced, the daemon DOES need `getBindingForSession` directly for the agent-config lookup reorder. Final import shape: `getBindingForSession` joins the existing destructured thread-registry import; `resolveShareFileChannel` is a separate single-symbol import. Acceptance criteria pinned only `resolveShareFileChannel` count (2 matches: 1 import, 1 invocation) — met exactly.
