<threat_model>
- **Behavior change risk**: changing `cursor = 2000` to `cursor = 1997` causes overflow chunk 1 to start 3 bytes EARLIER. This means overflow chunk 1's first 3 bytes are now the bytes that the editor truncated. From the operator's perspective, the visible "..." in the editor message + the first 3 chars of the next chunk visually JUMPED — but content-wise nothing is lost. Acceptable per operator's stated success criterion: "Subagent output is delivered in full, never truncated mid-sentence."
- **wrapMarkdownTablesInCodeFence interaction**: the wrap function may insert backticks at table boundaries, which can shift char positions. Use the WRAPPED string's length, not the raw content length, for cutoff calc. The current code already uses `wrapped` for the truncate; do the same in the overflow path. Specifically: the editor's `editFn` receives `content` and computes `wrapped = wrapMarkdownTablesInCodeFence(content)`. The overflow loop uses `text = wrapMarkdownTablesInCodeFence((reply ?? "").trim())`. Both are wrapped — the boundary alignment must use the wrapped-string indices.
- **Editor's actual visible cutoff vs overflow start**: the editor truncates at index 1997 + "..." (visible bytes 0-1996, 3 dots, total 2000 chars displayed). The overflow path starts at index 1997 of the same wrapped string. Discord shows: edited-message (0-1996 + "...") followed by overflow-chunk-1 (1997-3996, up to 2000 chars). Content bytes 1997-1999 appear in overflow-chunk-1's first 3 bytes — recovered.
- **wrapMarkdownTablesInCodeFence at chunk boundary**: if a code fence opens in the editor's visible text but closes in overflow chunk 1, the rendering may briefly look broken (close-fence missing in editor's view, open-fence missing in overflow's view). Pre-fix this is ALREADY broken (3-byte loss could be inside a code fence anyway). The fix doesn't make this worse. Mitigation deferred — operator decides if fence-aware chunking is needed in a future phase.
- **Test fixture must be a single contiguous string**, not multiple short responses. Boundary truncation only happens when a single response > 2000 chars. The reelforge-build-v2 thread evidence is a single subagent turn that exceeded 2000 chars.
