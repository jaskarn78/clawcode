## Phase Boundary

Four operator-reported subagent UX/correctness bugs in the same observation window (2026-05-05/06), all sitting on the subagent dispatch surface (`src/discord/subagent-thread-spawner.ts` + adjacent dispatcher / file-share / Progressive editor). Bundled into one phase because they share helper code and one machine cycle tests all four together.

### Sub-bug A — No typing indicator on subagent dispatch (P1)
While a subagent is mid-turn, the Discord thread shows nothing. No typing indicator, no progress message. Combined with subagent boot latency (~5min), operator sees an empty thread for minutes and can't tell if work is happening.

### Sub-bug B — Subagent output truncation (P1)
"Often don't produce full output" — operator-reported pattern. Subagent runs, posts a partial response, work is incomplete or cut off mid-sentence. **2026-05-06 amendment evidence (reelforge build threads `1501361804012687504` + `1501302129782952126`):** truncation happens at chunk boundaries — turn 1 ends mid-word ("Now updating..."), turn 2 starts mid-word ("o-end"; missing "end-t"). Same pattern across two different subagent threads. Strong evidence for **pipeline-side chunking mismatch in `ProgressiveMessageEditor`** (cause #3 in PHASE.md hypothesis tree), NOT max_tokens or tool-loop cap.

### Sub-bug C — Cross-channel file leak (P0 correctness)
fin-acquisition spawned the Schwab AIP deep-dive subagent. TL;DR text landed correctly in Ramy's channel thread. But the markdown file attachment (`schwab-aip-timing-research-2026-05-05.md`) posted at 2026-05-06 04:28:03 UTC into channel `1486348188763029648` (**finmentum-content-creator's primary channel**) — not Ramy's `1481670479017414767`. Compliance-relevant for Finmentum (could be PII routed to wrong audience in different scenario). Silent failure — operator got no error, file just landed in the wrong place. Root cause hypothesis: `clawcode_share_file` resolution walks workspace → agent binding, and finmentum-family agents share `/home/clawcode/.clawcode/agents/finmentum` workspace so the lookup grabs the wrong agent (likely first in config = content-creator).

### Sub-bug D — Completion event fires before subagent finishes (P0 — added 2026-05-06)
Operator-reported: "sometimes it seems like the completion event fires before agent finishes." When this fires early, the parent's `autoRelay` summary lands in main channel announcing work is done, the thread autoarchives, operator is told the subagent shipped — but the subagent's actual final chunks never arrived. Compounds with sub-bug B (cumulative effect: confident "Phase 2 complete" relay summary while last 2 minutes of work silently disappeared). Hypothesis: `subagent_complete` fires on tool-result-with-no-followup OR heartbeat-quiescence (Phase 999.30 quiescence sweep `quiescenceMinutes` default 5), instead of strictly on stream `end`.

## Implementation Decisions

### Bundling + Scope (D-01 through D-03)

**D-01 — Bundle all 4 sub-bugs in one phase.** Operator's PHASE.md already recommends this: "they'll all touch `subagent-thread-spawner.ts` and testing all three together is one machine cycle." Sub-bug D was added later but lives in the same surface. Lock: 4 sub-bugs, one phase.

**D-02 — Sub-bug C blast radius: list-don't-fix the audit.** PHASE.md flags concern about audit step 3 ("other resolution paths keyed off workspace") surfacing more bugs. **Decision:** the share-file resolution gets fixed in this phase. Other workspace-keyed lookup paths (memory writes, Discord posts, attachment uploads) get *catalogued* in a deferred-items list under this phase dir but NOT fixed. Operator can promote them to a follow-up phase if any are problematic. Don't scope-creep this phase — it has 4 bugs already.

**D-03 — Investigation order: P0s first, with diagnostics shipped before code fixes for B.** P0 bugs (C cross-channel leak, D premature completion) get fixed first in code. For sub-bug B, ship diagnostic logs in the same wave (relay-side `relayLen`/`chunkCount`/`endReason` per PHASE.md investigation step), but the actual fix lands AFTER one observation cycle — pipeline chunking root-cause is the strong hypothesis from amendment evidence; ship the fix that targets it, but instrument first to confirm.

### Sub-bug A — Typing indicator (D-04 through D-05)

**D-04 — Mirror `bridge.ts:361` pattern for subagent thread dispatch.** The existing `sendTyping()` call at the top-level dispatcher (`src/discord/bridge.ts:361,608`) is the reference. Wire equivalent emit into the subagent thread dispatcher path. Cap loop per Discord rate limit: typing extends 10s per call, refresh at ≤10s intervals. Use same `void channel.sendTyping().catch(...)` pattern (observational, non-fatal failure mode).

**D-05 — Refresh cadence: 8 seconds (not 10).** Discord's typing indicator extends ~10s per call but races against drift. 8s gives a 2s safety margin; if the indicator ever flickers off, operator sees it and we know the cadence needs tightening. Test: spawn a subagent that takes >30s; typing indicator must show throughout with no gaps.

### Sub-bug B — Output truncation (D-06 through D-08)

**D-06 — Root cause assumption: pipeline-side chunking mismatch.** Amendment evidence (2026-05-06) eliminates max_tokens and tool-loop cap as causes. Truncation always happens at chunk boundaries; head of one chunk gets eaten and tail of previous chunk is preserved elsewhere. This is consistent with `ProgressiveMessageEditor` chunking where chunk N+1's leading whitespace/word gets concatenated into chunk N's trailing edit, but the actual text content drops on the floor at the seam.

**D-07 — Fix: explicit chunk boundary tracking with overlap detection.** Track `lastChunkTail` and `nextChunkHead` per stream; when posting/editing the next message, verify the head doesn't overlap the previous tail (which would imply double-write) AND doesn't skip content (which would imply drop). On detected drop, log structured warning with byte offset + content sample, and patch the previous edit to include the dropped bytes.

**D-08 — Diagnostic instrumentation lands before the fix.** Per D-03, ship a structured log around relay-send (mirror quick task `260501-i3r` pattern): `{ relayLen, chunkCount, endReason, lastChunkTail, nextChunkHead }` when a subagent stream closes or chunks transition. The diagnostic data validates D-06's assumption before the fix in D-07 ships. If the data shows a different pattern (e.g., max_tokens or tool-loop), pivot D-07's approach. Two-task structure: T-instrumentation, then T-fix.

### Sub-bug C — Cross-channel file leak (D-09 through D-11)

**D-09 — Resolution path rebind: agent identity → channel bindings → originating thread's parent channel.** Workspace must NOT be a routing primitive. The fix path (per PHASE.md):
1. Subagent's parent agent identity (passed via TurnDispatcher origin) → that agent's channel bindings
2. Then route to the originating thread's parent channel (the one the subagent was spawned FROM, not just any channel the agent binds to)
3. Workspace value is irrelevant to this lookup — it's a coincidence, not a primitive

**D-10 — Dispatch-context `currentChannelId` already has the right value; the bug is upstream.** `clawcode-share-file.ts:347-349` reads `deps.currentChannelId()` correctly. The bug is in WHO populates that getter when a subagent is dispatching. Confirm: `currentChannelId()` should return the SUBAGENT THREAD ID, not just "first channel for this agent's workspace." Trace where `currentChannelId` is bound during subagent dispatch and verify it's correctly set.

**D-11 — Regression test: the shared-workspace invariant.** New test in `src/manager/tools/__tests__/clawcode-share-file-shared-workspace.test.ts`: spawn fixtures of two agents sharing a workspace (mimic finmentum family), invoke `clawcode_share_file` from agent A's dispatch context, assert the channel ID is A's originating thread, NOT agent B's primary channel. Catches this entire bug class.

### Sub-bug D — Premature completion event (D-12 through D-14)

**D-12 — Investigation goal: confirm the completion event source.** The PHASE.md amendment lists three candidate sources:
1. Stream `end` (correct)
2. Tool-result-with-no-followup (premature)
3. Heartbeat-quiescence (Phase 999.30 default 5min — premature for mid-think subagents)

Add structured `[diag] subagent-complete-fired` log at every site that fires the event, with `source` field naming which path fired. One observation cycle (see PHASE.md cross-reference to quick task `260501-i3r`'s diagnostic logs) before code fix.

**D-13 — Fix: gate completion on `streamFullyDrained && deliveryConfirmed` AND-clause.** The completion event must fire ONLY when:
- The subagent's stream has emitted `end` (not just inactive)
- The Discord post pipeline has confirmed delivery of the LAST chunk
- (NOT just on tool-result-with-no-followup or heartbeat-quiescence)

If quiescence is truly mid-think, the agent will resume; quiescence detection should NOT fire `subagent_complete` — it should fire a SEPARATE `subagent_idle_warning` (operator visibility, no autoArchive impact).

**D-14 — `autoArchive` waits for relay confirmation.** Per PHASE.md success criteria amendment: when `autoArchive=true`, the archive must wait for the relay to confirm delivery, not race ahead. Currently `autoArchive` implies `autoRelay`; ordering must be explicit: relay-confirmed-delivery → archive (NOT relay-fired → archive).
