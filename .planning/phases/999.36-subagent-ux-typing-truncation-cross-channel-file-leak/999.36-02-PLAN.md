P0 fix for sub-bug D — eliminate the premature `subagent_complete` event firing that causes parent's autoRelay summary to land in main channel announcing "Phase 2 complete" while the subagent's actual final chunks never arrived (compound failure with sub-bug B).

Per CONTEXT D-12 (confirmed via Plan 00's diagnostic logs running on prod for one observation cycle), the suspect firing paths are:
1. Stream `end` (correct)
2. Tool-result-with-no-followup (premature)
3. Heartbeat-quiescence (Phase 999.30 default 5min — premature for mid-think subagents)

Fix per D-13/D-14:
- Gate `subagent_complete` on `streamFullyDrained && deliveryConfirmed` AND-clause.
- Quiescence-sweep emits `subagent_idle_warning` (NEW log; no autoArchive impact) instead of firing `subagent_complete`.
- `autoArchive=true` waits for relay to CONFIRM delivery (not just to fire).

- `lastDeliveryAt: number | null` field added to `ThreadBinding` JSON registry (D-16 forbids SQL columns; JSON registry fields are ALLOWED — `completedAt` set the precedent).
- `streamFullyDrained` is implicit when `lastDeliveryAt` is set (the stamp happens AFTER editor.flush + overflow loop fully drain).
- `deliveryConfirmed` is the field name in code; `lastDeliveryAt` is the persisted value.

<threat_model>
- **Compound with sub-bug B**: until Plan 03 ships, sub-bug B can still cause chunk-boundary loss within the delivered text. Plan 02's gate prevents the COMPLETION event from firing on partial work — but doesn't fix the truncation itself. Operator must still check thread until Plan 03 lands.
- **Backwards compat**: existing bindings on disk lack `lastDeliveryAt`. Treat undefined as "delivery not confirmed" → relay gate refuses to fire. This means a bug-class case where Plan 02 deploys mid-stream-of-pre-existing-subagent will block that subagent's relay forever. Mitigation: use a one-time migration on daemon startup that stamps `lastDeliveryAt = lastActivity` for any binding with `completedAt === null && lastActivity !== null`. Pinned by a startup test.
- **Storage compat with D-16**: `lastDeliveryAt` is a JSON registry field on `ThreadBinding`. CONTEXT D-16 forbids "new MCP tools, IPC methods, schema columns." `ThreadBinding` is in a JSON file (`thread-bindings.json`), not a SQL table. Treat D-16 as SQL-only per advisor reconciliation. If checker flags it, the executor can document the interpretation in SUMMARY.
- **Zombie subagent**: agent crashes mid-stream → never stamps `lastDeliveryAt`. Without a backstop, `completedAt` never fires → thread accumulates as ghost. Mitigation: the existing session-end callback (`daemon.ts:7329-7350`) is the backstop; it already calls `relayCompletionToParent` directly. Plan 02 makes this path stamp `lastDeliveryAt` to `Date.now()` itself (treating session-end as a delivery-equivalent terminal state). This preserves the current "session ended → relay fires" behavior and is correct: if the agent's session ended without delivery, the relay should still go (operator deserves notification).
- **autoArchive race**: even with the gate, if `autoArchive=true` and the relay wait times out, the archive should NOT happen (don't archive without operator awareness). Use the existing `try/catch` pattern; on timeout, log warn and leave thread open.
- **Quiescence-warning storm**: if the warning fires every quiescenceMinutes for the same idle binding, log volume could grow. Use the existing `binding.lastActivity` field plus a new in-memory `idleWarningEmittedAt` Map keyed by threadId to dedupe (one warning per quiescence cycle per thread). Lives in daemon process — dies on restart.
</threat_model>

       export async function stampLastDeliveryAt(
         registryPath: string,
         threadId: string,
         deliveredAt: number,
       ): Promise<{ ok: true } | { ok: false; reason: string }> {
         const registry = await readThreadRegistry(registryPath);
         const idx = registry.bindings.findIndex((b) => b.threadId === threadId);
         if (idx === -1) return { ok: false, reason: "no-binding" };
         const updated: ThreadBinding = {
           ...registry.bindings[idx],
           lastDeliveryAt: deliveredAt,
         };
         const newRegistry: ThreadBindingRegistry = {
           bindings: [
             ...registry.bindings.slice(0, idx),
             updated,
             ...registry.bindings.slice(idx + 1),
           ],
           updatedAt: Date.now(),
         };
         try {
           await writeThreadRegistry(registryPath, newRegistry);
           return { ok: true };
         } catch (err) {
           return { ok: false, reason: (err as Error).message };
         }
       }
