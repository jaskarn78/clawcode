  <files>src/discord/message-coalescer.ts</files>
  <action>
    Add a new `requeue(agentName, messages)` method to `MessageCoalescer`. Semantics: messages were already accepted once (they passed the cap check on initial `addMessage`), so push-back must NOT re-check the cap. Otherwise a buffer at exactly the cap loses messages on every gate-deferral or cap-hit path.

    Mirror the existing immutable-append style at line 45-55 (per RESEARCH.md "Existing immutable-append in coalescer"):

    ```typescript
    /**
     * Push messages back into the buffer without enforcing perAgentCap.
     * Used by the bridge's drain block when the hasActiveTurn gate fires or
     * the depth cap is hit — the messages came OUT of the buffer and need
     * to go back in for the next drain attempt. Cap was already enforced on
     * initial addMessage, so re-checking here would silently drop accepted
     * messages.
     *
     * Prepends so re-queued messages drain first on next take (FIFO across
     * the original arrival order is preserved because the original take
     * already drained them in order).
     */
    requeue(agentName: string, messages: ReadonlyArray<CoalescedMessage>): void {
      if (messages.length === 0) return;
      const list = this.pending.get(agentName) ?? [];
      // Immutable concat — new array with re-queued at the head.
      const next: CoalescedMessage[] = [...messages, ...list];
      this.pending.set(agentName, next);
    }
    ```

    Constraints:
    - Immutable concat (spread, not push) — per CLAUDE.md coding-style.md.
    - No mutation of the input `messages` array (it's `ReadonlyArray<CoalescedMessage>` typed, but defense-in-depth).
    - No log line — this is a low-level buffer operation. The bridge logs the WHY (gate-defer or cap-hit) at the call site.
    - Update the file's top doc-comment block (`/** … */` at line 1-30) if it enumerates the public API methods, to include `requeue` (per the existing convention: `addMessage(...) — appends ...`, `takePending(...) — atomically returns ...`).

    ```typescript
    private static readonly COMBINED_PREFIX = "[Combined:";
    private static readonly MAX_DRAIN_DEPTH = 3;

    private formatCoalescedPayload(
      pending: ReadonlyArray<{ readonly content: string; readonly messageId: string }>,
    ): string {
      // Idempotent guard: a single pending entry that's already a wrapped payload
      // (re-queued after a prior coalesced dispatch failed with QUEUE_FULL) must
      // not gain a second [Combined: …] header. The storm trace from clawdy
      // 2026-04-30 09:47–09:58 PT showed +54 chars per spin-loop iteration from
      // exactly this nesting bug.
      if (pending.length === 1 && pending[0].content.startsWith(DiscordBridge.COMBINED_PREFIX)) {
        return pending[0].content;
      }
      const header = `[Combined: ${pending.length} message${pending.length === 1 ? "" : "s"} received during prior turn]`;
      const body = pending.map((m, i) => `(${i + 1}) ${m.content}`).join("\n\n---\n\n");
      return `${header}\n\n${body}`;
    }
    ```

    **Change 3 — Drain block: gate + cap (COAL-02 + COAL-03 + COAL-04).** Replace lines ~733-744 with:

    ```typescript
    // Drain pending coalesced messages.
    const pending = this.messageCoalescer.takePending(sessionName);
    if (pending.length === 0) return;

    // (a) Depth cap — prevent unbounded recursion regardless of root cause.
    //     If we hit the cap, push messages back and let the next message-arrival
    //     drain pick them up (or the in-flight turn's success/error path will
    //     re-trigger this block once the queue actually has room).
    if (drainDepth >= DiscordBridge.MAX_DRAIN_DEPTH) {
      this.log.warn(
        { agent: sessionName, count: pending.length, drainDepth },
        "coalescer drain depth cap reached — leaving messages buffered for next arrival",
      );
      this.messageCoalescer.requeue(sessionName, pending);
      return;
    }

    // (b) hasActiveTurn gate — defer drain if another turn is still occupying
    //     the per-agent queue. The next message-arrival or in-flight settle
    //     will re-trigger this block.
    if (this.sessionManager.hasActiveTurn(sessionName)) {
      this.log.debug(
        { agent: sessionName, count: pending.length },
        "coalescer drain deferred — turn still in-flight",
      );
      this.messageCoalescer.requeue(sessionName, pending);
      return;
    }

    // (c) Drain — proceed with combined dispatch.
    const combinedPayload = this.formatCoalescedPayload(pending);
    this.log.info(
      { agent: sessionName, count: pending.length, drainDepth },
      "draining coalesced messages as combined dispatch",
    );
    await this.streamAndPostResponse(message, sessionName, combinedPayload, undefined, drainDepth + 1);
    ```

    Notes:
    - The order matters: depth cap FIRST (cheapest check, prevents the runaway), gate SECOND, drain THIRD. If both cap and gate would fire, cap wins (we stop draining and leave buffered).
    - `requeue` is used in BOTH push-back paths so messages don't re-trigger cap rejection on `addMessage`.
    - Log levels: cap-hit = `warn` (operator should see one of these on real storms); gate-defer = `debug` (chatty in normal operation, useful only when debugging). Per CLAUDE.md skills/rules, use pino structured fields.
    - `drainDepth + 1` is passed only on the recursive call. The `combinedPayload`'s downstream catch handler (if it fails again with QUEUE_FULL) will run its own drain block at the incremented depth.
