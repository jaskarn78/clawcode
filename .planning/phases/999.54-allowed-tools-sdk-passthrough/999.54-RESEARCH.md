## Architecture Patterns

### Pattern 1: Spread-conditional at every hop (Phase 99 disallowedTools precedent — EXACT CLONE)

**What:** At every type-cascade hop, the new field is **OMITTED** from the constructed object when unset — never set to `undefined`. This preserves byte-stable deep-equality regression pins for the existing fleet (agents that don't opt in produce identical serialized objects pre- and post-merge).

**When to use:** Always for additive cascade fields. Repeated at 5 sites in the disallowedTools precedent.

**Example (clone these idioms verbatim):**

```typescript
// session-config.ts:347-349 (bootstrap path)
...(config.disallowedTools && config.disallowedTools.length > 0
  ? { disallowedTools: config.disallowedTools }
  : {}),

// session-config.ts:1094-1096 (main path)
...(config.disallowedTools && config.disallowedTools.length > 0
  ? { disallowedTools: config.disallowedTools }
  : {}),

// session-adapter.ts:1116-1118 (createSession)
...(config.disallowedTools && config.disallowedTools.length > 0
  ? { disallowedTools: [...config.disallowedTools] }
  : {}),

// session-adapter.ts:1240-1242 (resumeSession — MUST mirror createSession)
...(config.disallowedTools && config.disallowedTools.length > 0
  ? { disallowedTools: [...config.disallowedTools] }
  : {}),
```

**Adaptation for `alwaysLoad`:** Since `alwaysLoad` lives on each `mcpServers[]` entry (not at top-level), the spread-conditional goes **inside the `Object.fromEntries` map** in `transformMcpServersForSdk`, NOT at the top level of `baseOptions`. The cascade fields above (resolved → session-config) pass the whole `mcpServers` array through, so widening the readonly type is sufficient at those hops — the conditional spread is needed only at:

1. `loader.ts:463-479` — when pushing into the resolved mcpServers list, spread `alwaysLoad` conditionally to preserve byte-stable output for legacy server entries without it set.
2. `session-adapter.ts:1323-1325` — inside the `Object.fromEntries` map, spread `alwaysLoad: true` only when the source entry has it.

### Pattern 2: createSession ↔ resumeSession symmetric edit (Rule 3)

**What:** Every change to `createSession`'s `baseOptions` shape MUST be mirrored in `resumeSession`. A resumed session must receive identical SDK options to a fresh one; daemon restart cannot drift a session into a different SDK preload state.

**Where this matters here:** `transformMcpServersForSdk` is called from both `createSession` (line 1075) and `resumeSession` (line 1209). Since the SAME function is the chokepoint, widening it once automatically satisfies symmetric-edit — but the **test pin** must assert both call sites produce identical SDK options for the same input.

### Pattern 3: NON_RELOADABLE_FIELDS documentation-of-intent

**What:** Even though `differ.ts:144-149` already falls through to `reloadable: false` for any unclassified field, Phase 100 GSD-07 and Phase 115 sub-scope 2/5 both added explicit entries to `NON_RELOADABLE_FIELDS` for grep-ability.

**Recommended path strings to add:**

```typescript

"defaults.mcpServers.*.alwaysLoad",
"agents.*.mcpServers.*.alwaysLoad",
```

Note: the existing `mcpServerSchema` is shared between top-level `defaults.mcpServers` (a `z.record(string, mcpServerSchema)` at `schema.ts:2365`) and per-agent `agents[].mcpServers` (a `z.array(z.union([mcpServerSchema, z.string()]))` at `schema.ts:1499`). Adding the field to `mcpServerSchema` (line 621-639) is the single edit point — both call sites pick it up.

### Pattern 4: pino.info startup log line (D-05a)

**What:** `session-adapter.ts` already logs per-agent config (`disallowedTools` count, etc.). Add a log line listing servers with `alwaysLoad: true` for grep-ability.

**Suggested placement:** Right before the `sdk.query` initial drain call at `session-adapter.ts:1186`, after `baseOptions` is fully assembled. Use the existing `console.info` JSON pattern already in place at line 1166-1174 (for `phase115-quickwin`):

```typescript

  .filter(([, cfg]) => (cfg as { alwaysLoad?: boolean }).alwaysLoad === true)
  .map(([name]) => name);
if (preloadedServers.length > 0) {
  console.info(
    "phase999.54-preload",
    JSON.stringify({ agent: config.name, alwaysLoadServers: preloadedServers, flow: "create" }),

```

Mirror in `resumeSession` with `flow: "resume"`.

### Anti-Patterns to Avoid

- **Adding `alwaysLoad` to `mcpServerSchema` without widening `transformMcpServersForSdk`'s param type:** The function strips by **inline type signature**, not by intent. A field that exists on `ResolvedAgentConfig.mcpServers[]` but not on `transformMcpServersForSdk`'s narrow input type will be silently stripped. See Pitfall 3.
- **Explicit `undefined` instead of omission:** `{ alwaysLoad: undefined }` is NOT equivalent to omitting the field for byte-stable deep-equality. The Phase 99 cascade uses `...(cond ? {field: val} : {})` everywhere; clone the idiom.
- **Setting `alwaysLoad` at top-level `Options`:** SDK 0.2.140 only accepts it on per-server entries inside `Options.mcpServers`. There is no top-level `Options.alwaysLoad`.

## Common Pitfalls

### Pitfall 1: `clawcode` auto-inject site is the actual landing-PR target (NOT yaml)

**What goes wrong:** Operator writes a plan that adds `alwaysLoad: true` to `defaults.mcpServers.clawcode` in `clawcode.example.yaml`, deploys, and observes NO change in ToolSearch call patterns — because `clawcode.yaml` doesn't actually have a `clawcode` entry in its top-level `mcpServers:` block. Every agent in the fleet picks up the `clawcode` server via auto-inject at `loader.ts:295-304`, which constructs the server entry without `alwaysLoad`.

**Why it happens:** D-03a's "at the `defaults.mcpServers` level" phrasing is geographically accurate but operationally misleading. The locked-policy effect (preload `clawcode` for the whole fleet) requires either:
- (a) An operator-level yaml change AND an explicit per-agent `mcpServers: [..., clawcode]` reference for every agent — high-touch, easy to miss agents.
- (b) A code edit at the auto-inject site to bake in `alwaysLoad: true` — single line, affects whole fleet.

Option (b) is the simpler, safer landing-PR path. **The plan must include the loader.ts:295-304 edit (Step 5 above) as a mandatory task.**

**How to avoid:** Plan-checker should verify that the landing-PR includes a code edit at `loader.ts:295-304` (or, alternatively, that the plan explicitly threads `clawcode` into every agent's `mcpServers:` array — but this is the high-touch path D-03a is trying to avoid).

**Warning signs:** A plan that touches only schema.ts + types.ts + session-adapter.ts and "documents the default in yaml" without changing the loader is incomplete.

**Confidence:** HIGH. Verified by reading `loader.ts:295-304` (auto-inject confirmed) and `clawcode.example.yaml` (no `clawcode` in top-level mcpServers).

### Pitfall 2: `transformMcpServersForSdk` strips by inline type signature, not by intent

**What goes wrong:** Developer adds `alwaysLoad` to `ResolvedAgentConfig.mcpServers[]` and `AgentSessionConfig.mcpServers[]`, runs `tsc`, sees green, deploys — and `alwaysLoad` never reaches the SDK. The transform function silently strips it because its inline param type was never widened.

**Why it happens:** `transformMcpServersForSdk` at `session-adapter.ts:1317-1326` is a defensive narrow-type function: it accepts a subset of the broader `AgentSessionConfig.mcpServers[]` shape (only 4 fields), so passing a wider object doesn't trigger any type error — TypeScript happily accepts a superset. The function then reconstructs the output object with ONLY the 4 fields it knows about. Any additive field on the input is dropped.

**How to avoid:** When widening any type in the cascade above this function, ALSO widen this function's param type signature in lockstep. The `transformMcpServersForSdk` function's signature is the **actual contract** for "what reaches the SDK" — not the broader `AgentSessionConfig.mcpServers[]` type.

**Test pin:** Add a regression test that calls `transformMcpServersForSdk` with `{name, command, args, env, alwaysLoad: true}` and asserts the output contains `alwaysLoad: true`. This single test would have caught the existing pre-Phase-999.54 strip behavior immediately.

**Warning signs:** No direct test exists today for `transformMcpServersForSdk`. The function is unexported. Both should be addressed as part of this phase.

**Confidence:** HIGH. Read the function body directly at session-adapter.ts:1317-1326.

### Pitfall 3: Byte-stable deep-equality regression pins

**What goes wrong:** Developer uses `alwaysLoad: config.mcpServers[i].alwaysLoad ?? false` (explicit fallback) instead of spread-conditional. Existing fleet agents that don't opt in now have `{alwaysLoad: false}` in their serialized config, which differs from the pre-merge `{}` shape. Deep-equality regression tests fail.

**Why it happens:** Subtle JavaScript: `{}` and `{alwaysLoad: undefined}` and `{alwaysLoad: false}` are all *truthy-equivalent* but produce different serialized JSON. Phase 99 / 100 / 115 cascades explicitly call out "preserves byte-stable deep-equality regression pins" — the spread-conditional idiom is mandatory.

**How to avoid:** Every conditional addition uses `...(cond ? {field: val} : {})`. Never use `field: cond ? val : undefined` or `field: cond ? val : defaultVal`.

**Confidence:** HIGH. Pattern is repeated 4+ times in Phase 99 cascade with explicit comments calling this out.

### Pitfall 4: SDK 0.2.140 `Options.allowedTools` has SEMANTIC COLLISION with the BACKLOG's proposed field name

**What goes wrong:** A plan that originated from the BACKLOG might attempt to wire `Options.allowedTools` (which DOES exist at sdk.d.ts:1208) thinking it does what the BACKLOG describes. It doesn't — that field means "auto-allow without permission prompt," not "preload these tool schemas." The BACKLOG's "Acceptance criteria" line about `allowedTools` is wrong about the SDK contract.

**How to avoid:** CONTEXT.md D-01 supersedes the BACKLOG on this point. The plan MUST use `mcpServers[].alwaysLoad`, not `Options.allowedTools`. Plan-checker should reject any task that references `Options.allowedTools`.

**Confidence:** HIGH. Verified at sdk.d.ts:1208 (Options.allowedTools — permission-grant) vs sdk.d.ts:1064/1075 (alwaysLoad — preload).

### Pitfall 5: `alwaysLoad` adds a 5-second startup blocking window per server

**What goes wrong:** Per the SDK docstring at `sdk.d.ts:1062, 1073`: "as a side effect this also blocks startup until the server is connected (capped at the standard 5s connect timeout) even when MCP_CONNECTION_NONBLOCKING=1, since the tools must be present when the turn-1 prompt is built."

This means setting `alwaysLoad: true` on a slow-to-connect MCP server delays the agent's first turn by up to 5 seconds. For the `clawcode` MCP (which is a local process spawn — fast), this is unlikely to matter. For a hypothetical future broker server that depends on network handshake, it would.

**How to avoid:** D-03a's locked set is `clawcode` (fast, local). The 1password broker (already gated behind `OP_SERVICE_ACCOUNT_TOKEN` + handshake) is explicitly NOT in the locked set. If a future tuning pass adds `alwaysLoad: true` to a slow server, the operator should validate startup time impact before merging.

**How to validate:** Wave 2 verification should include a startup-time measurement (one `clawcode restart admin-clawdy` invocation, time-to-first-turn, compared against pre-merge baseline if available).

**Confidence:** HIGH (SDK docstring is explicit) — but the impact for `clawcode` server specifically is MEDIUM (depends on local-process spawn latency, which is sub-second in normal operation).

### Pitfall 6: Schema added to `mcpServerSchema` reaches BOTH `defaults.mcpServers` (record) AND `agents[].mcpServers` (array of union)

**What goes wrong:** Plan adds the field to schema and assumes per-agent overrides "just work" — without realizing that `agents[].mcpServers` accepts `z.union([mcpServerSchema, z.string()])`. The string variant is the by-name reference to a `defaults.mcpServers` entry. A per-agent override via the string-ref form CANNOT set `alwaysLoad` (the string carries no extra fields). A per-agent override REQUIRES an inline object form.

**How to avoid:** Document in the schema comment + plan that per-agent `alwaysLoad` overrides require the inline form (`{name: clawcode, alwaysLoad: false}`), not the string-ref form (`clawcode`). For the landing PR, this is moot (auto-inject sets the fleet-wide default); for deferred per-agent tuning, it's a footgun.

**Confidence:** HIGH. Read `schema.ts:1499`.
