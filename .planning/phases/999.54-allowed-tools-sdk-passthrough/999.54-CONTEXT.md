# Phase 999.54: allowed-tools-sdk-passthrough - Context

**Gathered:** 2026-05-14
**Status:** Ready for planning

<domain>
## Phase Boundary

Eliminate first-use ToolSearch round-trips for hot-path MCP tools by exposing
the Claude Agent SDK's `mcpServers[].alwaysLoad` knob through `clawcode.yaml`.
When set, the listed MCP server's tools appear in the agent's main tool
surface from session start — no per-session "fetch this schema, then call it
next turn" detour.

**In scope:**
- New yaml field controlling MCP-server preload (alwaysLoad passthrough).
- `defaults.*` baseline + `agents.*` per-agent override, merged at resolve time.
- Validation that warns (does NOT crash) on unknown server names.
- Schema → ResolvedAgentConfig → AgentSessionConfig → SDK Options wiring,
  mirroring the existing `disallowedTools` spread-conditional pattern.
- `tools` shape passthrough updated so `mcpServers[name].alwaysLoad` actually
  reaches the SDK (today `transformMcpServersForSdk` strips everything except
  `{command, args, env}`).

**Out of scope (deferred):**
- Per-tool `_meta['anthropic/alwaysLoad']` injection (would need a custom SDK
  shim — high complexity, marginal token savings over server-level preload).
- Preloading built-in tools (Bash/Read/Edit/etc) — separate `tools: string[]`
  vs preset mechanism; BACKLOG evidence is entirely MCP-tool-driven.
- Hot-reload via ConfigWatcher — classified non-reloadable (see D-02 below).
- Per-tool name allowlist in yaml — SDK 0.2.140 has no clean top-level Options
  field for this; the closest path is the deferred per-tool `_meta` route above.

</domain>

<decisions>
## Implementation Decisions

### Mechanism (SDK surface to expose)
- **D-01:** Use **server-level `mcpServers[].alwaysLoad: true` SDK passthrough**, not the BACKLOG's originally-proposed per-tool `allowedTools` list.
  - **Rationale (critical finding):** SDK 0.2.140 has *no* top-level `Options.allowedTools` field that means "preload these tool schemas." `Options.allowedTools` exists (`sdk.d.ts:1208`) but means "auto-allow without permission prompt" — different semantics. The actual preload mechanisms are `mcpServers[].alwaysLoad: boolean` (server-wide, `sdk.d.ts:1062, 1075`) and per-SDK-tool `tool({alwaysLoad: true})` (`sdk.d.ts:5618`). Per-tool `_meta['anthropic/alwaysLoad']` is the underlying API mechanism but is not surfaced as a clean Options field.
  - **Best performance + simplicity:** Server-level eliminates 100% of first-use deferred-fetch turns for the targeted MCP server in one flag. The cost (preloading whole server's tool schemas into prompt) is acceptable because the hot-path MCP servers ClawCode operates (`clawcode__*`, `1password`, `mysql`, `discord`) have small, fully-utilized tool surfaces. User direction: "whatever is likely to give the best performance gains."
- **D-01a:** Wire the field through `transformMcpServersForSdk` (currently strips anything except `{command, args, env}` — `session-adapter.ts:1317-1326`). New shape forwards `alwaysLoad` when present, spread-conditional like every other Phase-100+ field.

### Naming + shape
- **D-02:** Name the yaml knob `alwaysLoad: true` directly under each `mcpServers[]` entry — mirrors the SDK field name 1:1 so operators can correlate yaml → SDK docs without translation. Reject the BACKLOG's `allowedTools` / `preloadedTools` names (both misleading: the former collides with SDK's permission-grant field, the latter implies per-tool granularity we are explicitly NOT shipping).
- **D-02a:** Boolean only — no per-tool list inside the server entry. SDK's `alwaysLoad` is a single boolean per server; matching that keeps the schema simple and review-able.

### Scope (where in yaml)
- **D-03:** Both `defaults.mcpServers[].alwaysLoad` and `agents.<name>.mcpServers[].alwaysLoad` accepted. **However:** ClawCode already manages MCP servers at the `defaults.mcpServers` level (the resolver merges them onto every agent). The natural placement is the existing `mcpServers[]` array in defaults, with per-agent overrides allowed via the existing merge mechanism. No new top-level config block needed — the field rides on the existing MCP server schema.
- **D-03a:** **Concrete locked policy for landing PR:** `mcpServers` named `clawcode` and `clawcode-broker-discord` get `alwaysLoad: true` at the `defaults.mcpServers` level (covers `spawn_subagent_thread`, `post_to_agent`, `ask_advisor`, Discord operational tools — every agent uses these). All other servers remain default-deferred. Per-agent overrides (e.g. `finmentum` setting `1password.alwaysLoad: true`) land in their respective yaml stanzas in this phase OR in a follow-up tuning pass.

### Hot-reload classification
- **D-04:** **NON-reloadable.** `alwaysLoad` is an SDK session-boot baseOptions field (captured into `query()` at session start, not re-read per turn). Same architectural pattern as `settingSources`, `disallowedTools`, `excludeDynamicSections` — all explicitly documented in `src/config/types.ts:178-244` as NON_RELOADABLE_FIELDS. The BACKLOG's reload ask conflicts with the existing architectural pattern; the classifier in `differ.ts` already falls through to `reloadable: false` for paths not in `RELOADABLE_FIELDS`. Documentation-of-intent entries go in `NON_RELOADABLE_FIELDS` for the new field path. Operators apply changes via `clawcode restart <agent>`.

### Validation
- **D-05:** Zod accepts any boolean. No cross-validation that the server actually exists in `mcpServers` — `alwaysLoad` is a property OF an `mcpServers` entry, so the existence question is structural (zod already requires the entry to exist to set the flag on it). The "unknown server" failure mode from the BACKLOG doesn't apply to this design (it applied to a per-tool list, which we rejected).
- **D-05a:** Resolver logs a `pino.info` line at agent start listing which servers were marked `alwaysLoad: true` for that agent — gives operators a single grep target to confirm config flowed. No warnings, no crashes.

### Validation gates / verification
- **D-06:** Validate by `clawcode restart admin-clawdy` after merge, then spawn a fresh subagent. The 💭-emoji + advisor footer landing on turn 1 (instead of turn 2 with a ToolSearch round trip) confirms preload. Cross-check via traces.db: count ToolSearch calls for `mcp__clawcode__spawn_subagent_thread` per session before/after — expectation is 0 after preload.

### Claude's Discretion
- Source-of-truth file locations: schema declaration goes in `src/config/schema.ts` (mcpServers entry shape, near the existing entry definition); type field goes in `src/shared/types.ts` `ResolvedAgentConfig.mcpServers` shape; passthrough goes in `src/manager/session-config.ts` (both bootstrap-needed path ~line 335 and main path ~line 1077) and `src/manager/session-adapter.ts` `transformMcpServersForSdk` at line 1317 (BOTH `createSession` and `resumeSession` paths — Rule 3 symmetric edit, same as the Phase 99 disallowedTools precedent).
- Test pins: extend existing schema.test.ts disallowedTools test pattern + a regression test against `transformMcpServersForSdk` confirming `alwaysLoad: true` survives the transform when set and is omitted when unset (byte-stable for fleet agents that don't opt in).
- Default state in `defaults.mcpServers`: do NOT flip every server to `alwaysLoad: true` — only `clawcode` and `clawcode-broker-discord` per D-03a. Preloading 1password / mysql / drive eats tokens for agents that don't use them.

</decisions>

## Existing Code Insights

### Reusable Assets
- **`transformMcpServersForSdk` (`session-adapter.ts:1317`)** — single chokepoint for AgentSessionConfig → SDK Record shape. Adding `alwaysLoad` here is one line + a sibling property in the input type.
- **Phase 99 disallowedTools cascade** — exact passthrough pattern to clone. Schema → ResolvedAgentConfig → AgentSessionConfig → SDK Options, spread-conditional at every hop so existing fleet stays byte-identical when the field is absent. 5 source files touched per the cascade, ~30-50 lines of net addition.
- **`pino.info` startup logs in `session-adapter.ts`** — already logs `disallowedTools` count and other per-agent config; adding "preloaded servers: [list]" rides on the existing log line.

### Established Patterns
- **Spread-conditional at every hop (Rule 3 symmetric edit):** Phase 99 sub-scope N comments explicitly call out "preserves byte-stable deep-equality regression pins." Apply the same. Field MUST be OMITTED (not `undefined`) when not set. Pattern repeats in `session-config.ts:347, 1094` and `session-adapter.ts:1116, 1240`.
- **createSession ↔ resumeSession symmetric edits:** A resumed session must receive identical SDK options to a fresh one; daemon restart cannot drift a session into a different SDK preload state. Apply per the disallowedTools / advisorModel precedent.
- **NON_RELOADABLE_FIELDS documentation-of-intent entries:** Phase 100 GSD-07 and Phase 115 sub-scope 2 / 5 both added documentation entries even though the classifier already falls through to false. Follow this pattern for grep-ability.

### Integration Points
- **`mcpServers[]` array shape in `src/config/schema.ts`** — schema entry where the new `alwaysLoad?: z.boolean().optional()` lives. Already accepts `{name, command, args, env}`; this is a pure additive field.
- **`ResolvedAgentConfig.mcpServers` in `src/shared/types.ts`** — type shape for the resolved per-agent server list. Must gain `alwaysLoad?: boolean`.
- **`AgentSessionConfig.mcpServers` in `src/manager/types.ts`** — already mirrors ResolvedAgentConfig shape; must mirror the new field.
- **Loader resolver (wherever `defaults.mcpServers` merges into per-agent)** — must propagate `alwaysLoad` through the merge so a `defaults` setting flows to all agents and a per-agent setting wins on conflict. Locate via `grep -rn "mcpServers" src/config/loader.ts`.

<specifics>
## Specific Ideas

- **Concrete first targets:** `clawcode` and `clawcode-broker-discord` MCP servers get `alwaysLoad: true` at `defaults.mcpServers` in the landing PR. These two cover `spawn_subagent_thread`, `post_to_agent`, `ask_advisor`, and Discord operational tools — every agent's hot path.
- **Don't preload everything:** Resist the temptation to flip every MCP server. `1password`, `mysql`, `drive`, `playwright` etc. are agent-specific. Preloading them on agents that never use them wastes tokens.
- **Verify via traces.db, not just vibes:** Pre/post ToolSearch call counts for `mcp__clawcode__spawn_subagent_thread` per session is the objective metric (BACKLOG Related §3 — once 999.49 lands, the dashboard tool-rollup will surface this directly).

</specifics>

<deferred>
## Deferred Ideas

- **Per-tool `_meta['anthropic/alwaysLoad']` preload** — surgical preload of single tools without preloading their whole MCP server. Requires synthesizing `_meta` markers on individual tool definitions via a custom SDK shim path. Defer until evidence shows token cost of server-level preload is a problem (it won't be for the clawcode server's small tool surface).
- **Built-in tool preload (Bash, Read, Edit, etc.)** — controlled via SDK's `tools: string[] | preset` option, not `mcpServers[].alwaysLoad`. BACKLOG mentions it but provides no hot-path evidence; built-in tools rarely defer. Separate phase if metrics later justify.
- **Hot-reload of alwaysLoad** — would require a session-rebuild pathway in ConfigReloader, conflicting with the architectural pattern at `types.ts:178-186`. Defer indefinitely; operators run `clawcode restart <agent>`.
- **Per-agent `mcpServers[].alwaysLoad` overrides for finmentum (`1password`), projects (`drive`), atlas (`mysql`)** — natural follow-up tuning once D-03a's defaults land and we measure per-agent ToolSearch call patterns.
- **Re-running the 2026-05-14 tool preload audit** — per BACKLOG Related §1; depends on 999.49 dashboard fix to get per-tool invocation frequency data.

</deferred>
