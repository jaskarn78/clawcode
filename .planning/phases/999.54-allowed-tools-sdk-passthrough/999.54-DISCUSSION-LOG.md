## Mechanism (which SDK surface to expose)

| Option | Description | Selected |
|--------|-------------|----------|
| Server-level `alwaysLoad` in mcpServers yaml | Add `alwaysLoad: true` to each mcpServers[] entry. Simple, 1:1 SDK passthrough. Preloads whole server. Matches Phase 110 mcpServers passthrough pattern. | ✓ (via "best performance gains") |
| Per-tool name list | Synthesize per-tool `_meta['anthropic/alwaysLoad']` via custom adapter. Surgical but SDK 0.2.140 has no clean top-level Options field for this. | |
| Hybrid (server-level for MCP + per-tool only for SDK-internal tools) | Phase 1 server-level, Phase 2 per-tool if needed. | |

**User's choice:** "whatever is likely to give the best performance gains"
**Notes:** Server-level `alwaysLoad: true` is the right call. (1) Simpler implementation — single chokepoint in `transformMcpServersForSdk`. (2) Eliminates 100% of first-use deferred-fetch turns for the targeted server. (3) Token cost is acceptable because the hot-path servers (`clawcode`, `clawcode-broker-discord`) have small, fully-utilized tool surfaces. Per-tool granularity gains marginal token savings at much higher complexity — deferred indefinitely.

**Critical finding shared with user:** SDK 0.2.140's `Options.allowedTools` does NOT mean "preload these tool schemas." It means "auto-allow without permission prompt" — different semantics. The BACKLOG's premise of mirroring `disallowedTools` with `allowedTools` doesn't match the SDK API. The real preload knob is `mcpServers[].alwaysLoad: boolean`. This reframes the entire phase design.

## Reload classification

| Option | Description | Selected |
|--------|-------------|----------|
| Agent restart required (non-reloadable) | Classify alongside settingSources / disallowedTools / excludeDynamicSections. SDK session-boot baseOptions cannot retroactively change a live session. Operators run `clawcode restart <agent>`. | ✓ |
| Hot-reload via ConfigWatcher (BACKLOG's ask) | Add to RELOADABLE_FIELDS and rebuild session on diff. Conflicts with the documented architectural pattern at `types.ts:178-186`. Would require new reloader plumbing. | |

**User's choice:** Agent restart required (Recommended)
**Notes:** Matches the existing architectural pattern. Documentation-of-intent entry goes in NON_RELOADABLE_FIELDS even though the classifier already falls through to false.

## Scope (where in yaml)

| Option | Description | Selected |
|--------|-------------|----------|
| Both — `defaults.*` baseline + `agents.*.*` override | Matches every other tool/agent knob in schema.ts. Fleet shares a baseline, specific agents add their own. | ✓ |
| Per-agent only | Simpler, no merge logic. Every agent yaml repeats the same baseline. Diverges from established schema pattern. | |

**User's choice:** Both — defaults + per-agent override (Recommended)
**Notes:** Concrete landing-PR policy locked: `defaults.mcpServers` for `clawcode` and `clawcode-broker-discord` get `alwaysLoad: true` (every agent uses these). Per-agent overrides for 1password/mysql/drive/etc. handled in follow-up tuning per agent's actual usage pattern.

## Validation strictness

| Option | Description | Selected |
|--------|-------------|----------|
| Warn on unknown server names, never crash | Permissive zod schema + resolver-time pino warnings. Matches `mcpEnvOverrides` posture. | ✓ |
| Hard-reject unknown server names at parse | Zod refine() cross-validates against mcpServers. Catches typos but brittle to server removal. | |

**User's choice:** Warn on unknown, never crash (Recommended)
**Notes:** Note D-05 in CONTEXT.md narrows this further — because we landed on `alwaysLoad` as a property OF an mcpServers entry (D-02), the "unknown server" failure mode the BACKLOG was anticipating doesn't structurally apply. Zod just accepts a boolean. Resolver still logs an info line at agent start listing which servers were preload-marked, giving operators a single grep target.
