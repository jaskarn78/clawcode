<objective>
Land the fleet-wide leverage edit at `src/config/loader.ts:295-304` — bake `alwaysLoad: true` into the auto-injected `clawcode` MCP server entry so every agent in the fleet picks up the preload default on next restart without requiring any yaml change. This is the actual mechanism behind CONTEXT.md D-03a (RESEARCH.md Pitfall 1: a yaml-level `defaults.mcpServers.clawcode.alwaysLoad: true` would silently no-op because no production yaml declares `clawcode` in its top-level `mcpServers:` block — the auto-inject site is the only fleet-wide path).

Companion edit at the resolved-server construction site (line 463-479) propagates yaml-declared `alwaysLoad` from the schema (Plan 01) into the resolved config via spread-conditional, matching the existing `description` / `accessPattern` siblings. This is what makes per-agent overrides work (D-03 / RESEARCH.md OQ-3) — an operator who declares `clawcode` explicitly in their per-agent `mcpServers:` block (inline-object form) with `alwaysLoad: false` causes `resolvedMcpMap.has(\"clawcode\")` at line 295 to return true, skipping the auto-inject, and the per-agent entry flows through the resolved-construction spread instead.

D-05a structured `console.info` log emitted after the mcpServers list is constructed — gives operators a single grep target (`phase999.54-resolver`) on clawdy systemd journal to confirm preload-marked servers per agent at agent boot.

CRITICAL — `clawcode-broker-discord` is NOT included in this plan (RESEARCH.md OQ-1 — phantom server, zero grep hits in src/ or scripts/). Only `clawcode` gets the auto-inject `alwaysLoad: true`. The 1password / browser / search / image auto-inject sites (lines 341-356, 395-435) are NOT touched per D-03a's explicit "don't preload everything" guidance.
