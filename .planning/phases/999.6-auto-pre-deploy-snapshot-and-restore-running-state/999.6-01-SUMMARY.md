# Phase 999.6 Plan 01: Wave 1 GREEN — snapshot-manager + daemon wiring + schema field — Summary

**One-liner:** Implemented atomic snapshot-manager module + wired daemon shutdown writer (top of `shutdown()`, before drain) + boot reader (before autoStartAgents filter) + added `defaults.preDeploySnapshotMaxAgeHours` zod field — turns 38/38 Wave 0 RED tests GREEN with zero new dependencies.

## Outcome

Operator-pain fix shipped at the source-code level: `systemctl restart clawcode` now preserves the running fleet across restarts. `autoStart=false` agents that were running at SIGTERM time get recorded in `~/.clawcode/manager/pre-deploy-snapshot.json` and auto-revived on the next boot (one-shot — the file is deleted before `startAll`, so the *next* normal restart honors yaml `autoStart` only). 24h staleness guard prevents long-stale snapshots from auto-reviving agents the operator never wanted back.

Plan 02 (deploy gate) is the next step — this plan only ships the code; production smoke is gated to Plan 02 per `feedback_no_auto_deploy`.

## Deviations from Plan

### [Variable name] Renamed `knownAgentNames` → `snapshotKnownAgentNames`

- **Found during:** Task 3 (typecheck after wiring)
- **Issue:** The plan's `<interfaces>` snippet uses `const knownAgentNames = new Set(resolvedAgents.map((c) => c.name))`. The same variable name is already declared in the same `startDaemon` function at line 1804 (used by `reconcileRegistry`).
- **Fix:** Renamed our local to `snapshotKnownAgentNames` to disambiguate.
- **Files modified:** src/manager/daemon.ts (one line)
- **Verification:** `npx vitest run src/manager/__tests__/daemon-snapshot-restore.test.ts` — 5/5 still GREEN. SNAP-WIRE source-grep tests pin the function-call shape (`readAndConsumePreDeploySnapshot([^)]*config\.defaults\.preDeploySnapshotMaxAgeHours`), not the local variable name, so the rename doesn't break the wiring contract.
- **Commit:** `7bb7ff4`

This is a syntactic mechanic, not a behavioral change. The plan's `<interfaces>` block was a verbatim-copy template; the planner couldn't have anticipated the existing in-scope binding.

### [Wave 0 cleanup] Removed @ts-expect-error directives

- **Found during:** Task 3 (typecheck reported `Unused '@ts-expect-error' directive` at 4 sites)
- **Issue:** Wave 0 added directives to suppress TS2307 (module-not-found) and TS2339 (property-not-found) errors that exist before the implementation lands. Once the module + field exist, the directives become "unused" → typecheck error.
- **Fix:** Removed 1 directive from `snapshot-manager.test.ts` import block + 3 directives from `schema-snapshot-defaults.test.ts` field accesses.
- **Files modified:** snapshot-manager.test.ts, schema-snapshot-defaults.test.ts
- **Verification:** typecheck on those files clean; all 33 tests still GREEN
- **Commit:** `7bb7ff4`

This was anticipated by the Wave 0 SUMMARY ("Wave 1 removes this directive when the module exists").

**Total deviations:** 2 cleanup-class (one variable rename for collision, one Wave 0 directive cleanup). **Zero behavioral deviations** from the plan's `<interfaces>` block. **Impact:** none — wiring contract preserved, all RED tests turn GREEN as designed.
