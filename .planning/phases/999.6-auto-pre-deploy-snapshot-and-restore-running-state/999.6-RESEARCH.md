## Summary

This phase adds a small, self-contained "running-fleet snapshot" subsystem to the clawcode daemon. On SIGTERM/SIGINT, the daemon serializes its in-memory list of running agent names to `~/.clawcode/manager/pre-deploy-snapshot.json` via tmp+rename. On next boot, the daemon reads that snapshot, unions the names into the static `autoStart=true` set from `clawcode.yaml`, passes the union to `manager.startAll(...)`, and deletes the file. A 24h staleness guard prevents long-stale snapshots from auto-reviving agents the operator never wanted back.

Every primitive needed is already in the repo:
- `getRunningAgents()` (`session-manager.ts:1899`) — returns `[...this.sessions.keys()]`, which is exactly the running set (`sessions.set` happens at line 970 after `status: 'running'` registry write; `sessions.delete` happens in stopAgent).
- Atomic write pattern — `effort-state-store.ts:131-148` is the canonical idiom; copy verbatim.
- SIGTERM/SIGINT handlers — already registered at `daemon.ts:4316-4322`. Hook the snapshot **write** at the top of the existing `shutdown()` function (line 4180), BEFORE `manager.drain()`. Hook the snapshot **read** in the boot path between line 4324 ("manager daemon started" log) and the `autoStartAgents` filter at line 4337.
- `defaults.preDeploySnapshotMaxAgeHours` — additive optional zod field in `defaultsSchema` (line 1300 of `src/config/schema.ts`); follows the same `z.number().int().positive().default(24)` shape as `greetCoolDownMs` (line 1316) and `memoryFlushIntervalMs` (line 1334).

**Primary recommendation:** Build a single `src/manager/snapshot-manager.ts` module exporting `writePreDeploySnapshot()` and `readAndConsumePreDeploySnapshot()`. Wire the writer into the **first line** of `shutdown()` (before drain). Wire the reader into the boot path immediately before the `resolvedAgents.filter(...)` block. Total daemon.ts diff: ~25 lines. Total new code: one ~150-line module + one ~200-line test file + one ~80-line integration test.

### Pattern 1: Atomic JSON Persistence (verbatim from effort-state-store.ts)

**What:** Write JSON to `<path>.<rand>.tmp` in same directory, then `rename()` to final path. POSIX rename is atomic within a filesystem — readers either see the old file or the new file, never half-written content.

**When to use:** Any daemon-side state file where a partial write would break boot.

**Example (canonical idiom from effort-state-store.ts:143-146):**
```typescript
// Source: src/manager/effort-state-store.ts:143-146
await mkdir(dirname(filePath), { recursive: true });
const tmp = `${filePath}.${randomBytes(6).toString("hex")}.tmp`;
await writeFile(tmp, JSON.stringify(next, null, 2), "utf8");
await rename(tmp, filePath);
```

**Why 12 hex chars:** Avoids tmp-file collisions if two callers race (effort-state-store.ts:126-129 explains — even though current callers serialize, the store must be safe for future concurrent writers). For our case, only the daemon writes during shutdown, but mirroring the pattern keeps drift-resistance.

### Pattern 2: Boot-time Read-Then-Consume

**What:** Read snapshot fully into memory, apply override, delete file unconditionally (even on partial-startAll-failure) to prevent auto-restart loops.

**When to use:** One-shot recovery state files where re-application would be incorrect.

**Example (skeleton):**
```typescript
// src/manager/snapshot-manager.ts (NEW)
export async function readAndConsumePreDeploySnapshot(
  filePath: string,
  knownAgentNames: ReadonlySet<string>,
  maxAgeHours: number,
  log: Logger,
): Promise<ReadonlySet<string>> {
  const restoreLog = log.child({ component: "snapshot-restore" });

  let raw: string;
  try {
    raw = await readFile(filePath, "utf8");
  } catch (err) {
    if ((err as NodeJS.ErrnoException).code === "ENOENT") return new Set(); // Normal boot
    restoreLog.warn({ filePath, error: (err as Error).message }, "snapshot read failed — falling back to static autoStart");
    return new Set();
  }

  // Parse + validate (Zod safeParse — malformed → warn + delete + empty set)
  let parsed: PreDeploySnapshot;
  try {
    parsed = preDeploySnapshotSchema.parse(JSON.parse(raw));
  } catch (err) {
    restoreLog.warn({ filePath, error: (err as Error).message }, "snapshot malformed — deleting and falling back");
    await unlink(filePath).catch(() => {});
    return new Set();
  }

  // Stale guard
  const ageMs = Date.now() - Date.parse(parsed.snapshotAt);
  const ageHours = ageMs / (1000 * 60 * 60);
  if (ageHours > maxAgeHours) {
    restoreLog.warn(
      { snapshotAgeHours: Number(ageHours.toFixed(1)), maxAgeHours },
      "discarding stale snapshot (>maxAgeHours) — falling back to static autoStart",
    );
    await unlink(filePath).catch(() => {});
    return new Set();
  }

  // Filter to names that still exist in current config
  const restored = new Set<string>();
  for (const entry of parsed.runningAgents) {
    if (knownAgentNames.has(entry.name)) {
      restored.add(entry.name);
    } else {
      restoreLog.warn({ agent: entry.name }, "snapshot references unknown agent — skipping");
    }
  }

  restoreLog.info(
    { snapshotAt: parsed.snapshotAt, agentCount: restored.size },
    `applying pre-deploy snapshot — ${restored.size} agents will be auto-started`,
  );

  // Delete BEFORE startAll — guarantees never-replayed semantics even if startAll throws
  await unlink(filePath).catch((err) =>
    restoreLog.warn({ err: (err as Error).message }, "snapshot delete failed (will be overwritten next shutdown)"),
  );
  restoreLog.info({ deletedSnapshotPath: filePath }, "snapshot consumed — next restart will honor static autoStart config");

  return restored;
}
```

**Note:** delete happens BEFORE startAll returns. The CONTEXT.md mentions "after startAll(... or 60s hard cap" but unconditional pre-startAll delete is simpler and safer — partial startAll failure produces a registry with the right intent (status=crashed → reconcile loop retries via existing recovery), and we never get into a "snapshot replayed twice" failure mode.

### Pattern 4: Shutdown-path Wiring (daemon.ts diff shape)

**Anchor (daemon.ts:4180):**
```typescript

  log.info("shutdown signal received");

  // PHASE 999.6 — write snapshot FIRST, before drain begins evicting sessions.
  // getRunningAgents() reflects the in-memory sessions map; once drain settles
  // SDK requests, stopAgent calls will start removing entries.
  try {
    await writePreDeploySnapshot(
      PRE_DEPLOY_SNAPSHOT_PATH,

        name,
        sessionId: manager.getSessionHandle(name)?.sessionId ?? null,
      })),
      log,

  } catch (err) {
    log.warn(
      { err: (err as Error).message },
      "pre-deploy snapshot write failed (non-fatal — boot will fall back to static autoStart)",
    );
  }

  // Existing shutdown sequence:
  try {
    const drainResult = await manager.drain(15_000);
    ...
```

**Pitfall: ordering vs. drain.** `manager.drain()` settles in-flight summaries but does NOT itself remove sessions from the map. However, downstream lines (4276 `manager.stopAll()`) DO remove them. Writing the snapshot at the top of `shutdown()` is the safest place — the registry status is exactly what was running when SIGTERM arrived.

**Pitfall: `getSessionHandle(name)?.sessionId`.** Snapshot stores `sessionId` informationally only (per CONTEXT.md "NOT used for restore — daemon spawns fresh sessions"). The handle exposes a `sessionId` getter — verify exact shape during plan; if the property name differs (`id`, `sessionID`), normalize to `null`. Schema accepts `string | null` so any quirk degrades gracefully.

### Anti-Patterns to Avoid

- **Writing snapshot AFTER `manager.drain()` or `manager.stopAll()`** — sessions map starts emptying; snapshot loses entries. **Always write at the top of `shutdown()`.**
- **Reading registry.json instead of in-memory `getRunningAgents()`** — registry can include `crashed` / `restarting` entries we explicitly do NOT want to auto-revive (per CONTEXT.md filter rules). The in-memory `sessions` map is exactly the right filter.
- **Using `unlink` after `startAll` returns** — if startAll throws, snapshot persists and gets re-applied on next boot, doubling the auto-start. Always delete BEFORE startAll.
- **Throwing on snapshot read failure** — boot must never block on snapshot. Every error path in the reader returns an empty set + warn log.
- **Synchronous fs calls** — match the rest of the codebase: `node:fs/promises` only. (effort-state-store.ts and dream-log-writer.ts are both async.)
- **Hardcoding the 24h threshold** — per CONTEXT.md it MUST be configurable via `defaults.preDeploySnapshotMaxAgeHours`.

## Common Pitfalls

### Pitfall 1: Snapshot written after sessions are stopped → empty/incorrect snapshot

**What goes wrong:** If the snapshot writer is wired AFTER `manager.drain()` or `manager.stopAll()` in the shutdown sequence, `getRunningAgents()` returns an empty (or partial) array. Boot then sees a snapshot with zero agents and auto-starts nothing extra — bug looks like "the feature did nothing."

**Why it happens:** The shutdown function in daemon.ts (line 4180) has many ordered steps. Adding one new line "somewhere" in there is tempting. But sessions begin evicting from the map at line 4276 (`manager.stopAll()`).

**How to avoid:** Hook the snapshot writer at the **first line** of `shutdown()`, right after `log.info("shutdown signal received")`. Tested by the source-grep test: snapshot-write call must appear textually BEFORE `manager.drain` in `shutdown`.

**Warning signs:** Boot log shows `applying pre-deploy snapshot — 0 agents` for every restart. Or snapshot file present but `runningAgents: []`.

### Pitfall 2: Snapshot delete happens after startAll throws → infinite auto-revive loop

**What goes wrong:** If startAll partial-fails (one agent's MCP server is down, say) and we delete the snapshot only on success, the next boot replays the same snapshot. If the underlying failure is persistent, the loop never terminates.

**Why it happens:** Intuitive ordering says "consume the snapshot only if startAll succeeded." But that conflates "I tried to apply it" with "every agent in it started." Only the former matters for never-replay semantics.

**How to avoid:** Delete the snapshot **before** `manager.startAll(...)` is invoked. The reader function is responsible for the delete; the daemon just receives the resolved name set. Tested by an integration test: start with a snapshot referencing one valid + one nonexistent agent, mock startAll to throw, assert snapshot file is gone.

**Warning signs:** Boot log shows the same set of names repeatedly auto-starting across multiple restarts despite the operator never re-running them.

### Pitfall 3: SIGKILL bypasses the writer — operator confusion

**What goes wrong:** SIGKILL is uncatchable in Node.js. If systemd's `TimeoutStopSec` (default 90s) elapses and systemd sends SIGKILL, the snapshot is never written and the next boot loses the running fleet.

**Why it happens:** `manager.drain(15_000)` allows up to 15s + cleanup; total shutdown is normally well under 30s. But edge cases (hung MCP child, dashboard server not closing) can blow past TimeoutStopSec.

**How to avoid:**
1. Write the snapshot at the **top** of `shutdown()` — fastest possible. A single ~1KB JSON file write is sub-100ms. This way, even if a later step hangs, the snapshot is already on disk.
2. Document in commit message that operators using `kill -9` or watching SIGKILL get the same fleet-loss behavior they have today.
3. (Out of scope, but worth noting for ops:) the production `clawcode.service` on the clawdy host should set `TimeoutStopSec=60` or higher to give graceful shutdown room — this is unrelated to phase 999.6 but operator-relevant.

**Warning signs:** journal shows `Killing process ... with signal SIGKILL` from systemd, no snapshot file present, fleet doesn't restore.

### Pitfall 4: Reading the snapshot, then SIGTERM hits during startAll, snapshot writer fires with EMPTY running set, snapshot file gets clobbered

**What goes wrong:** Boot sequence is: (1) read snapshot, (2) compute autoStartAgents, (3) start IIFE that calls `manager.startAll(...)`. If SIGTERM arrives DURING step 3, the `getRunningAgents()` call inside the new shutdown writer sees a partial sessions map (some agents started, others not yet) and writes an INCORRECT snapshot — losing fleet members that hadn't been spawned yet.

**Why it happens:** The new shutdown writer reads `getRunningAgents()`, which is the live map. During boot it grows incrementally as `startAll` iterates.

**How to avoid (practical mitigation):** This is a real but very rare edge case (SIGTERM during boot — operator double-restarting). Mitigation: the snapshot file is unconditionally deleted DURING boot read (before startAll). So if SIGTERM hits mid-boot, the writer still records the partial set, but the prior snapshot is already gone. The operator loses the agents not-yet-started, but doesn't get worse-than-today behavior. Document this limitation in the module docstring; do NOT add complex "boot still in progress" guards (would expand scope).

**Warning signs:** Operator restarts twice in <5s; second boot shows fewer agents than expected. Recoverable via manual `clawcode start <name>`.

### Pitfall 5: `defaults.preDeploySnapshotMaxAgeHours` zod default vs. resolver

**What goes wrong:** zod schema sets `default(24)`, but the loader's resolver layer (which materializes resolved configs) might bypass the default if the field is not propagated through `resolveAgentConfig`. The reader then sees `undefined` and `Date.now() - undefined` is NaN.

**Why it happens:** Several `defaults.*` fields are explicitly listed in `loader.ts` resolveAgentConfig — fields not listed there can fall through.

**How to avoid:** This field is **not** a per-agent override (it's a daemon-level config). Read it directly from `config.defaults.preDeploySnapshotMaxAgeHours` in daemon.ts, NOT from a resolved agent config. Belt-and-suspenders: in the reader, default to 24 if the value is undefined/NaN: `const maxAgeHours = (typeof maxAge === "number" && Number.isFinite(maxAge)) ? maxAge : 24;`

**Warning signs:** Snapshot is never discarded as stale; or always discarded as stale (NaN compare returns false in JS).

### Pitfall 6: Zod field placement breaks v2.5/v2.6 backward-compat tests

**What goes wrong:** `defaultsSchema` has migration tests that pin "configs without field X parse unchanged." Adding a non-default-bearing required field would break this.

**Why it happens:** Several phases (83/86/89/90/92/96/100) explicitly call this out in schema.ts comments — see line 1349 "8th application of the Phase 83/86/89/90/92 schema blueprint."

**How to avoid:** Make the field `z.number().int().positive().default(24).optional()` — additive-optional, default-bearing. Existing configs parse unchanged. Mirrors `greetCoolDownMs` (line 1316), `memoryFlushIntervalMs` (line 1334) shapes.

**Warning signs:** existing schema migration test failures referencing unknown field on parse.

### Example 1: Snapshot file schema (zod, embedded in snapshot-manager.ts)

```typescript
// src/manager/snapshot-manager.ts (NEW)
import { z } from "zod/v4";

export const preDeploySnapshotSchema = z.object({
  version: z.literal(1),
  snapshotAt: z.string().refine(
    (s) => !Number.isNaN(Date.parse(s)),
```

### Example 2: Atomic write (mirrors effort-state-store.ts:131-148)

```typescript
// Source: src/manager/effort-state-store.ts:131-148 (verbatim idiom)
export async function writePreDeploySnapshot(
  filePath: string,
  runningAgents: ReadonlyArray<{ name: string; sessionId: string | null }>,
  log: Logger,
): Promise<void> {
  const writeLog = log.child({ component: "snapshot-restore" });
  const next: PreDeploySnapshot = {
    version: 1,
    snapshotAt: new Date().toISOString(),
    snapshotPid: process.pid,
    runningAgents: [...runningAgents],
  };
  await mkdir(dirname(filePath), { recursive: true });
  const tmp = `${filePath}.${randomBytes(6).toString("hex")}.tmp`;
  await writeFile(tmp, JSON.stringify(next, null, 2), "utf8");
  await rename(tmp, filePath);
  writeLog.info(
    { filePath, agentCount: next.runningAgents.length },
    "pre-deploy snapshot written",
  );
}
```

## Open Questions

1. **Do we want to log the snapshot's `snapshotPid` in the apply log line?**
   - What we know: pid is informational only; helps operator correlate with journal entries from prior daemon process.
   - What's unclear: noise vs. signal at log level info.
   - **Recommendation:** Include it — `log.info({ snapshotAt, snapshotPid, agentCount }, "...")`. Trivially cheap and disambiguates "this snapshot is from the daemon I just stopped" vs. "this is a stale snapshot from a crash 6h ago."

2. **What happens if SIGTERM arrives during boot — specifically before `readAndConsumePreDeploySnapshot` returns?**
   - What we know: handlers are registered before boot's IIFE in current daemon.ts (signal handlers at 4316-4322 sit AFTER the auto-start IIFE at 4345 — so during the snapshot read window, signal handlers are not yet installed; default Node.js behavior is to terminate the process on SIGTERM).
   - What's unclear: actually, default SIGTERM handling exits Node with no `process.exit(0)` cleanup → partial-state on next boot is the same as a crash today.
   - **Recommendation:** No change — accept default Node behavior. Document in module docstring: "if SIGTERM arrives before the snapshot reader runs, the prior snapshot persists (re-applied next boot) — this is the desired behavior."

3. **Should the snapshot writer fire on SIGHUP (config reload)?**
   - What we know: the existing config-watcher / reloader does NOT call `shutdown()`. Only SIGTERM/SIGINT do.
   - What's unclear: if the daemon ever adds SIGHUP-triggered restart, the snapshot path needs updating.
   - **Recommendation:** Out of scope for this phase. If/when a SIGHUP handler is added, it should explicitly call the snapshot writer.

4. **Sub-second restart races: SIGTERM #2 before SIGTERM #1's shutdown completes?**
   - What we know: `process.on("SIGTERM", () => { void shutdown().then(() => process.exit(0)) })` registers the same handler twice if the operator hits Ctrl-C twice. Node's signal handlers are idempotent: same listener, single registration. But two SIGTERMs → two `shutdown()` invocations.
   - What's unclear: writePreDeploySnapshot is called twice rapidly. Atomic rename semantics mean each rename clobbers the prior tmp file's destination → final state is consistent.
   - **Recommendation:** No fix needed — atomic rename guarantees the second writer wins. Document in docstring.
