# Phase 13: Discord Attachment & Image Support - Context

**Gathered:** 2026-04-09
**Status:** Ready for planning

<domain>
## Phase Boundary

This phase adds the ability for agents to send and receive attachments/images in Discord. After this phase, agents can read images sent by users (via Claude's multimodal capabilities), download attachments, and send files/images back in responses.

</domain>

<decisions>
## Implementation Decisions

### Receiving Attachments
- **D-01:** When the bridge forwards a message with attachments, include attachment metadata (name, type, size, URL) in the formatted message
- **D-02:** For images, download them to a temp directory and pass the path to the agent session so Claude can read them (multimodal)
- **D-03:** For other files, download and provide the path for the agent to read
- **D-04:** Attachment download uses a shared temp directory at `{workspace}/inbox/attachments/`

### Sending Attachments
- **D-05:** Agent sessions can send files to Discord using the Discord plugin's reply tool with `files` parameter (already supported by the MCP plugin)
- **D-06:** For images generated by the agent, the file path is passed to the Discord reply tool
- **D-07:** Support sending from agent workspace paths

### Message Formatting
- **D-08:** Update the bridge's message formatting to include attachment info as structured metadata
- **D-09:** Attachment URLs are Discord CDN URLs — download before they expire
- **D-10:** Large attachments (>25MB) are logged as warnings but not downloaded

### Architecture
- **D-11:** Attachment handling is a module at `src/discord/attachments.ts`
- **D-12:** Downloads are async with timeout (30s default)
- **D-13:** Temp files cleaned up on a schedule or when inbox is processed

### Claude's Discretion
- Exact temp file naming convention
- Image format handling details
- Cleanup schedule for downloaded attachments

</decisions>
