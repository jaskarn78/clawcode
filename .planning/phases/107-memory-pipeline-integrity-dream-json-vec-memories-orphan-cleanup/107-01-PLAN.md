<objective>
Pillar A — Dream pass JSON output enforcement. Strengthen the dream system prompt with a schema-correct fallback envelope (DREAM-OUT-01), downgrade the parse-failure log to a structured `warn` with capped `responsePrefix` (DREAM-OUT-03), add unit tests covering all four input branches (DREAM-OUT-04), and document DREAM-OUT-02 (SDK structured-output mode) as deferred with file anchors so a future phase can pick it up.

Purpose: Today the dream pass already survives parse failures (returns `{kind:"failed"}` per dream-pass.ts:271-280) — but it logs at `error` level, polluting operator alerting, and the recovery is unstructured. The model also slips into chat-style prose despite Phase 95's tightened rules. Two-pronged fix: (1) give the model an EXACT escape hatch (the fallback envelope) so even uncertain runs produce schema-valid output, (2) make the recovery log structured + warn-level so it grep-counts cleanly across the fleet.

Output: A model that produces valid no-op envelopes when uncertain, a daemon that warns instead of erroring on prose, and a test file that pins the recovery contract for future regressions.

**Constraint reconciliation (from RESEARCH.md):**
- The CONTEXT.md fallback envelope `{"newWikilinks":[],"promotionCandidates":[],"summary":"","errors":["<reason>"]}` would FAIL zod validation (no `summary` or `errors` keys in dreamResultSchema). Use the schema-matching shape: `{"newWikilinks":[],"promotionCandidates":[],"themedReflection":"","suggestedConsolidations":[]}`.
- DREAM-OUT-02 (SDK `outputFormat: { type: "json_schema", schema }`) is DEFERRED. SDK supports it (sdk.d.ts:694, 1244) but production dispatch goes through the agent's persistent shared `sdk.query` handle (session-adapter.ts:653, 1286-1311) where `outputFormat` is session-scoped, not per-turn.
