<objective>
Suppress the `HEARTBEAT_OK` no-op message that the `projects` agent's cron-poll skill is leaking into the operator's user-facing Discord channel (A2A-04 / D-06 / 999.48). The fix lives in the agent's own cron skill workspace under `~/.clawcode/agents/projects/skills/cron-poll/` — NOT in this daemon repo. REQUIREMENTS forbids a daemon-side string-match filter (wrong layer, wrong abstraction); the cron skill knows what its own no-op pattern looks like.

This plan is FULLY PARALLEL with Plans 01/02/03 — different workspace, different deploy. `depends_on=[]` and wave=1.

`autonomous=false` because the agent-side workspace deploy is operator-driven (separate from `scripts/deploy-clawdy.sh` which targets the daemon, not the agent skill directory). The operator triggers the agent skill rollout; this plan documents what to change and how to verify.

Per D-06 + D-08:
- D-06: HEARTBEAT_OK filter at agent layer ONLY. No daemon code change.
- D-08: SC-4 verified by 24h journalctl grep for `HEARTBEAT_OK` in the operator's channel logs == 0.

Per D-07 sequencing: Plan 04 is fully parallel — no dependency on Plans 01/02/03.

Output:
- A single guard inserted in the `projects` agent's cron-poll skill at the post-poll dispatch site.
- Agent-workspace commit (if the agent's workspace is git-tracked — Task 1 verifies).
- Verification artifact at `.planning/phases/119-a2a-delivery-reliability/119-04-VERIFICATION.md` recording the change + the 24h soak result.
