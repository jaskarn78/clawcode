    Stage 0b is shipped (plan 110-07 complete). The Go static shim has been running production traffic across all 11 agents for some operator-defined settle period. The question now is whether to keep the Node shim code as a safety net.

    **Stage 0b runtime savings achieved:** ≥ 2.7 GiB at full fleet (per 110-07 measurements).

    **Node shim cost at runtime if retained as fallback:** zero — Node shims only spawn if `shimRuntime.<type>: node` is explicitly set. They are dormant code in the daemon's bundled CLI (~2.16 MB ESM in process memory regardless, per docs/phase-110/preflight-procs.md).

    **Path A — Keep Node fallback (conservative):**
    - No source code changes
    - Schema enum stays `["node", "static", "python"]`
    - Operators can flip back to Node at any time without code change
    - ~2.16 MB Node bundle stays in daemon process memory (small relative to total daemon RSS)
    - Adds documentation note: "Node shims are deprecated runtime — Stage 0b static is production. Flip back only as emergency rollback."

    **Path B — Remove Node shims (aggressive):**
    - Delete `src/cli/commands/{search,image,browser}-mcp.ts` (3 files, ~50-100 LOC each)
    - Remove their entries from the CLI registry (likely `src/cli/index.ts` or similar)
    - Narrow schema enum to `["static", "python"]`
    - Delete loader's `case "node":` branch from `resolveShimCommand` helper (plan 110-02 introduced)
    - Update rollback runbook: "If Go shim misbehaves, flip to `python` runtime (FastMCP fallback — separate plan to implement) OR `git revert` Stage 0b commits."
    - Reclaims ~2.16 MB Node bundle from daemon process memory (small but real)
    - Eliminates ~2 KB of dead loader code paths
    - **Risk:** if a Go shim bug surfaces post-cleanup, rollback options are reduced to `python` (not yet implemented) or git revert
  </context>
  <options>
    <option id="path-a-keep-fallback">
      <name>Path A — Keep Node shims as flippable fallback</name>
      <pros>
        - Maximum rollback flexibility — operators can flip ANY shim type back to Node any time without code change
        - Zero risk of regression — no code deletion, no test breakage
        - Documentation-only change (lowest cost)
        - Matches conservative ops posture suggested by ramy_active_no_deploy memory
      </pros>
      <cons>
        - ~2.16 MB Node bundle stays in daemon process memory indefinitely (small relative to Stage 0b 2.7 GiB savings — < 0.1%)
        - Schema enum has 3 values instead of 2 (cosmetic complexity)
        - Dead loader code path (`case "node":`) survives — minor maintenance cost
      </cons>
    </option>
    <option id="path-b-remove">
      <name>Path B — Remove Node shim code entirely</name>
      <pros>
        - Reclaims ~2.16 MB from daemon process memory
        - Eliminates dead loader code paths
        - Forces commitment to Go runtime — no half-measures
        - Reduces schema/test surface area
      </pros>
      <cons>
        - Rollback options reduced — `python` runtime is reserved but not implemented in Stage 0b; git revert is the only escape
        - Higher risk if Go shim bug surfaces post-cleanup
        - Requires test churn (every test referencing Node shim path needs update)
        - Cost > savings if Stage 0b proves stable but a niche bug appears in a year
      </cons>
    </option>
  </options>
