# Phase 999.25 — Subagent relay on work-completion, not session-end

**Status**: backlog (queued for planning)
**Caller-of-record**: Jas, 2026-05-04 admin-clawdy ClawdyV2 nuclear-deep-dive observation
**Predecessor**: Phase 999.X (subagent auto-prune, deployed) +
  closure-capture fix (PR #8, deployed)
**Sibling**: Phase 999.18 (relay-reliability — different bug, same area)

## Symptom

Operator delegates a research task to a subagent in the morning.
Parent agent posts "response will land in that thread" to the
operator's channel. Subagent finishes its work shortly after.

Operator never sees the result delivered to their channel until the
subagent session is **stopped** (operator-initiated, parent-driven
auto-archive, or — as of Phase 999.X — the 2h reaper). In the
2026-05-04 incident, the ClawdyV2 nuclear-deep-dive completion was
delivered hours late, only when sessions got cleaned up around
3:36 PM PT.

## Root cause

`relayCompletionToParent` (`src/discord/subagent-thread-spawner.ts:805`)
fires from the `autoRelay` path, which runs only when the **session
ends**. Today session-end happens via:

- Operator `clawcode stop`
- Parent agent's auto-archive flow on its OWN session end
- Phase 999.X subagent-session-reaper (now: 2h after binding idle)

But there is no signal for "the subagent finished its WORK"
independent of session lifecycle. A subagent that posts its final
answer to its thread, then sits idle waiting for follow-ups that
never come, holds the relay until session-end fires hours later.

## Constraints

- Subagent threads are normal Claude Code sessions; they don't have
  a "task complete" callback today.
- Adding a `mark_complete` tool requires skill-side cooperation
  (every subagent skill would need to call it).
- Heuristics ("no new turns for N minutes after a substantive
  message") risk false-positives on slow tool chains.
- The relay should not double-fire — if it relays on work-completion,
  the session-end path must be a no-op.

## Possible approaches

1. **Mark-complete tool**: skill author calls `mcp__clawcode__subagent_complete` after the final answer. Relay fires immediately; session-end relay becomes idempotent skip. Requires updating spawn-subagent-thread skill + every dependent skill.
2. **Final-message detection**: subagent posts a message that matches `^Done\b|^Final:\b|<final/>` etc. Heuristic, brittle, but no skill changes.
3. **Quiescence timer**: N minutes (e.g. 5) after last turn that produced a substantive message, fire relay. Tunable per-skill via `quiescenceMinutes`. Catches the common case without skill cooperation; longer than chain-of-tool-calls but shorter than full idle threshold.
4. **Hybrid**: support the explicit `subagent_complete` tool for skills that adopt it; fall back to quiescence timer for legacy skills.

Default recommendation: option 4 (hybrid). Explicit signal wins;
quiescence timer is the safety net.

## Out of scope (separate phases)

- Per-parent-agent relay channel routing (already exists)
- Relay artifact attachments (already implemented in `relayCompletionToParent`)
- Phase 999.18's "relay skipped due to error" path (different bug)

## Verification (pre-shipping)

- E2E: spawn subagent, watch parent channel get the relay within
  the configured quiescence window (vs today: hours, until reaper)
- Idempotence: when session eventually ends, no second relay
- False-positive guard: subagent in long tool chain (>quiescence
  window mid-tool-call) does NOT trigger relay — wait until tool
  output, then re-arm timer
- Operator override: `quiescenceMinutes: 0` disables, restoring
  current session-end-only behavior
