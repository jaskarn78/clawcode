# Phase 999.57 Plan 01: factory + HeartbeatRunner wiring Summary

Plugged the four config-inheritance leak surfaces that caused 2026-05-15
production failures by extracting a `buildSubagentOverrides` factory + threading
`HeartbeatRunner` through the spawner so the existing heartbeat gate finally
fires for subagent sessions.

## What changed

- **New factory** `src/discord/subagent-config-overrides.ts` (139 lines) —
  single source of truth for the override slice spread last into `subagentConfig`.
  Covers `heartbeat.enabled`, `memoryAutoLoad`, `memoryRetrievalTopK` +
  `memoryRetrievalTokenBudget`, `memoryScannerEnabled`,
  `memoryFlushIntervalMs`, and the optional
  `memory.conversation.flushIntervalMinutes`. Includes the Pitfall 3
  null-safety guard on the conversation block.
- **`SubagentThreadSpawnerConfig.heartbeatRunner`** — new REQUIRED field on
  the ctor opts. The spawner now calls
  `this.heartbeatRunner.setAgentConfigs([subagentConfig])` immediately
  after `sessionManager.startAgent(...)` in `spawnInThread`.
- **`daemon.ts:7644`** — passes the existing `heartbeatRunner` (declared at
  3300) into the spawner ctor. Single call site.
- **21 test ctor sites updated** — 18 in
  `src/discord/subagent-thread-spawner.test.ts` + 3 in
  `src/discord/__tests__/{subagent-chunk-boundary-fixture, subagent-recursion-guard, subagent-delegates-scoping}.test.ts`.
  Each file got a local `makeMockHeartbeatRunner()` helper alongside its
  existing `makeMockSessionManager` helper; the helper spies on
  `setAgentConfigs` only since that's the spawner's sole runner contact.

## Why this matters

RESEARCH.md Finding 1 showed CONTEXT.md's `heartbeat.enabled: false`
override was functionally dead — the runner's local `agentConfigs` map is
populated once at daemon boot from `resolvedAgents`, so
`agentConfigs.get(subagentSessionName)` always returns `undefined`, and
the gate at `runner.ts:271` (`if (agentConfig && agentConfig.heartbeat.enabled === false)`)
short-circuits truthy. Result: heartbeat checks (including the inbox
check that triggered the original failure) fire for every subagent
session regardless of the config-side override. The fix is the
post-startAgent additive `setAgentConfigs` upsert — `runner.ts:143-147`
is a `for...of` loop with `set()`, no clear, verified additive.

D-05's `inboxEnabled` field was dropped — Finding 2 confirmed the field
does not exist on `ResolvedAgentConfig`, defaults schema, or loader. The
"inbox watcher leak" the BACKLOG postulated is actually the
heartbeat-driven inbox check at `src/heartbeat/checks/inbox.ts:61-70`,
which the Finding-1 runner-side fix already closes.
