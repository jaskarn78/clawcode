## Summary

The 999.57 work is an inheritance-leak fix. The grep evidence confirms the underlying problem (subagent inherits source agent's full `ResolvedAgentConfig` via spread at `subagent-thread-spawner.ts:657-668`, and that config drives a memory retriever closure, two flush timers, MEMORY.md auto-load, and heartbeat tick eligibility).

However, **three of the five CONTEXT.md decisions land on dead code or non-existent fields as written**:

1. **D-01 (heartbeat override) is functionally dead** ‚Äî `HeartbeatRunner.agentConfigs` is a local Map populated only at daemon boot; subagent configs never reach it. The gate at `runner.ts:271` short-circuits on `agentConfig === undefined` (the `agentConfig &&` clause), so the check loop FIRES on the subagent session regardless of the override.
2. **D-05 (inbox override) targets a field that doesn't exist** ‚Äî there is no `inboxEnabled` on `ResolvedAgentConfig`. The chokidar `InboxSource` is constructed once at daemon boot from `config.triggers.sources.inbox` and never iterates subagent configs. The "inbox leak" the BACKLOG postulated is actually a heartbeat-check leak (inbox.ts heartbeat check fires per agent via the same broken gate as D-01).
3. **D-03 partial (memoryScannerEnabled) is dead code for subagents** ‚Äî the scanner registration loop at `daemon.ts:3239` runs once at boot over `resolvedAgents`. Subagents are never iterated, so override has no consumer.

The override-only approach (CONTEXT.md's preferred shape) cannot fix the heartbeat/inbox-check leak alone. The planner must adopt ONE of two minimal runtime-side patches:

- **(a) Preferred:** `subagent-thread-spawner` calls `heartbeatRunner.setAgentConfigs([subagentConfig])` after `sessionManager.startAgent`. The existing `setAgentConfigs` is an additive upsert (runner.ts:143-147 ‚Äî `for...of` loop, no clear). Smallest blast radius.
- **(b) Alternative:** Change `runner.ts:270-271` to read from `sessionManager.getAgentConfig(agentName)` instead of its local map. One-line semantic shift.

**Primary recommendation:** Implement the override factory per D-07 (correctness of D-02 + D-04 + the flush-timer override is HIGH confidence). Add path (a) to make D-01 actually gate the heartbeat. Treat D-03 `memoryScannerEnabled` and D-05 `inboxEnabled` as defensive markers (no functional effect, but pin via tests so a future refactor that activates those paths can't silently leak). Add a sixth override: `memory.conversation.flushIntervalMinutes: 0` (a second flush timer not mentioned in CONTEXT.md ‚Äî verified at `session-manager.ts:2624-2640`).

## Findings That Modify CONTEXT.md Decisions

### Finding 1: D-01 heartbeat override is dead code without a runner-side fix (HIGH)

**Evidence chain:**

- `HeartbeatRunner` constructor: `this.agentConfigs` is a private `Map<string, ResolvedAgentConfig>` populated only via `setAgentConfigs()` (`src/heartbeat/runner.ts:143-147`).
- `setAgentConfigs(resolvedAgents)` is called once at daemon boot (`src/manager/daemon.ts:3336`) and on YAML hot-reload (`src/manager/config-reloader.ts:129`). Both pass `resolvedAgents` (the static agent fleet) ‚Äî subagent sessions never appear.
- The gate at `runner.ts:270-271`:
  ```
  const agentConfig = this.agentConfigs.get(agentName);
  if (agentConfig && agentConfig.heartbeat.enabled === false) { continue; }
  ```
  When `agentName` is a subagent session, `this.agentConfigs.get(...)` returns `undefined` ‚Üí `agentConfig` is falsy ‚Üí gate's `&&` short-circuits ‚Üí **the continue does not fire** ‚Üí check loop runs.
- `HeartbeatRunner.tick()` iterates `this.sessionManager.getRunningAgents()` (`runner.ts:251`), which returns `[...this.sessions.keys()]` (`session-manager.ts:2136`) ‚Äî includes subagent session names.
- Inbox heartbeat check (`src/heartbeat/checks/inbox.ts:61-70`) reads `sessionManager.getAgentConfig(subagentName)` which DOES return the subagent's config (set in `startAgent` at `session-manager.ts:794`) ‚Äî gets `memoryPath` (inherited from source) ‚Üí reads `{sourceMemoryPath}/inbox/` ‚Üí fires the heartbeat-driven inbox check ON THE SUBAGENT.

**Implication for the plan:** D-01's `heartbeat.enabled: false` override has no functional effect. The planner MUST add one of:

- **(a) Preferred ‚Äî additive registration.** After `sessionManager.startAgent(sessionName, subagentConfig)` in `subagent-thread-spawner.ts:671`, call `heartbeatRunner.setAgentConfigs([subagentConfig])`. The existing method is additive (loop, no clear), so the upsert is safe. Requires threading `heartbeatRunner` into `SubagentThreadSpawner` constructor (currently it has `sessionManager`, `registryPath`, `discordClient` ‚Äî see `subagent-thread-spawner.ts` line 1565 in tests).
- **(b) Alternative ‚Äî runner gate rewrite.** Change `runner.ts:270-271` to consult `this.sessionManager.getAgentConfig(agentName)` instead of `this.agentConfigs.get(agentName)`. One-line shift. Larger semantic implication: the runner stops keeping its own copy.

**Recommend (a)** ‚Äî keeps the runner's data-source pattern intact, smaller blast radius, easier to test (the spawner test already mocks `sessionManager`; add a `heartbeatRunner` mock with `setAgentConfigs: vi.fn()` and assert it's called with `[subagentConfig]`).

CONTEXT.md `<deferred>` claims "Subagent-aware gating inside HeartbeatRunner / InboxSource ‚Äî defense-in-depth, not needed for the current failure pattern." The grep evidence contradicts this: it IS needed for D-01 to do anything. Surfacing this conflict for the planner.

### Finding 2: D-05 `inboxEnabled` field does not exist; "inbox leak" is actually a heartbeat-check leak (HIGH)

**Evidence chain:**

- Grep for `inboxEnabled`, `inbox.enabled` across `src/shared/types.ts`, `src/config/schema.ts`, `src/config/loader.ts`: **no matches**. There is no per-agent inbox toggle in `ResolvedAgentConfig`.
- `InboxSource` (chokidar) is constructed at daemon boot from `config.triggers.sources.inbox` (`src/manager/daemon.ts:3107-3125`). It iterates the daemon-level trigger config, NOT `resolvedAgents`. Subagent sessions are never iterated.
- `InboxSource` events route via `triggerEngine.ingest(event)` ‚Üí `decision.targetAgent` ‚Üí `turnDispatcher.dispatch(origin, decision.targetAgent, payloadStr)` (`src/triggers/engine.ts:158-162`). `targetAgent` is fixed at source-construction time to the **configured** agent (the parent), never the subagent session.
- The actual leak path is the **heartbeat inbox check** (`src/heartbeat/checks/inbox.ts`). It reads `sessionManager.getAgentConfig(agentName).memoryPath` where `agentName` is a subagent session, gets the source's `memoryPath` (inherited via spread), and watches the source's inbox dir from inside the subagent's heartbeat tick. This is the same leak path as Finding 1 ‚Äî its fix is Finding 1's (a) or (b), not a separate inbox knob.

**Implication for the plan:** D-05 as written cannot be implemented (field doesn't exist on `ResolvedAgentConfig`). Three options:

- **(i)** Drop D-05 entirely. The leak it targets is real but addressed by the Finding 1 fix.
- **(ii)** Reframe D-05 as a defensive marker ‚Äî e.g., set an unused `subagentScope: "transient"` field (new optional field) so future subagent-aware code branches have a discriminator. NOT recommended ‚Äî adds schema surface for no immediate consumer.
- **(iii) Recommended:** Keep DEL-18 as a regression test that asserts the heartbeat inbox check does NOT execute against the subagent (after Finding 1's fix lands). The assertion becomes: "spawn a subagent, advance heartbeatRunner's clock, assert no inbox check result lands in latestResults for the subagent."

Update CONTEXT.md interpretation: D-05 is the GOAL ("inbox watcher doesn't fire on subagent"), not the MECHANISM (no field exists). Mechanism is D-01's runner-side fix.

**Evidence chain:**

### Finding 4: A second flush timer is not covered by D-03 (HIGH)

**Evidence chain:**

- `startMemoryFileFlushTimer` (`session-manager.ts:1390-1423`) reads `config.memoryFlushIntervalMs` ‚Äî disk flush, writes to `{workspace}/memory/YYYY-MM-DD-HHMM.md`. D-03 covers this (`memoryFlushIntervalMs` override).
- **Separately**, `startFlushTimer` (`session-manager.ts:2624-2671`) reads `config.memory.conversation?.flushIntervalMinutes ?? 15` ‚Äî DB flush, writes mid-session summaries into the shared `memories.db` (which subagent shares with source via the inherited `memoryPath`). This is the path that produces the `mid-session` MemoryEntries that Phase 115 sub-scope 4 specifically excludes from retrieval.
- The intervalMs <= 0 short-circuit at line 2634 confirms `flushIntervalMinutes: 0` cleanly disables this timer.

**Implication:** Planner should add a sixth override:
```
memory: {
  ...subagentSourceConfig.memory,
  conversation: {
    ...(subagentSourceConfig.memory.conversation ?? <conversation defaults>),
    flushIntervalMinutes: 0,
  },
}
```

For short-lived subagents (<15 min) this is a no-op. For long-running delegated subagents (e.g., research tasks running 20-30 min), this prevents mid-session summaries from leaking into the source agent's memories.db. The leak is real because `memoryPath` is shared (Finding 5).

### Finding 5: `memoryPath` shared ‚Üí `memories.db` shared (HIGH, informational)

Subagent spread inherits `memoryPath` unchanged from source. `initMemory` (`session-memory.ts:52-138`) opens `memories.db` at `getAgentMemoryDbPath(config.memoryPath)`. The subagent gets a SEPARATE `MemoryStore` instance (better-sqlite3 connection) pointing at the SAME DB file as the source. WAL allows multiple connections, but any write from the subagent's session (flush timers, cue writes, subagent-return capture, dedup) lands in the source's DB.

This is the underlying architectural reason all the override decisions matter. A cleaner architectural fix would be to point subagent `memoryPath` at a transient subagent-scoped temp dir ‚Äî but per CONTEXT.md D-08 we're shipping the safer narrow fix. The planner should record this as a Deferred Idea for a future phase if the narrow fix proves insufficient.

**What:** Subagent config is built by spreading source first, then overriding fields. The 999.57 factory adds new overrides to the existing list.

### Pattern 2: Factory returning Partial<ResolvedAgentConfig>

**What:** Override factory returns a slice; spread-last position guarantees it wins. Existing test factories already produce this shape.

```typescript
// src/discord/subagent-config-overrides.ts
export function buildSubagentOverrides(
  parentConfig: ResolvedAgentConfig,
  sourceConfig: ResolvedAgentConfig,
  isDelegated: boolean,
): Partial<ResolvedAgentConfig> {
  return {
    heartbeat: {
      ...sourceConfig.heartbeat,
      enabled: false,
    },
    memoryAutoLoad: false,
    memoryRetrievalTopK: 0,
    memoryRetrievalTokenBudget: 0,
    memoryScannerEnabled: false,
    memoryFlushIntervalMs: 900_000, // schema default; no-op for short-lived
    memory: {
      ...sourceConfig.memory,
      conversation: sourceConfig.memory.conversation
        ? { ...sourceConfig.memory.conversation, flushIntervalMinutes: 0 }
        : undefined,
    },
  };
}
```

### Anti-Patterns to Avoid

- **Don't add `inboxEnabled` to `ResolvedAgentConfig` schema.** No consumer exists. Bloats type surface for zero runtime effect.
- **Don't write `memoryRetrievalTopK: 0` via zod-parsed YAML.** Schema rejects `positive()`. Runtime override-only.
- **Don't restructure `HeartbeatRunner.tick()` to skip on `agentConfig === undefined`.** That breaks the legitimate case where a non-subagent runs without registered config (boot race).
- **Don't override `memoryPath`.** Out of scope for this phase. The deferred architectural fix is bigger than the current narrow patch.

## Common Pitfalls

### Pitfall 1: Heartbeat override appears to work in unit tests but does nothing in production
**What goes wrong:** Test mocks `heartbeatRunner.tick()` not at all, just asserts `startedConfig.heartbeat.enabled === false`. Production has the gate-falsy bug (Finding 1).
**Why it happens:** Test pins the override plumbing, not the runner behavior.
**How to avoid:** Add an integration-style test that constructs a real `HeartbeatRunner` with mocked checks, calls `setAgentConfigs([sourceConfig])` (boot), then triggers subagent spawn (which should ALSO call `setAgentConfigs([subagentConfig])`), then `runner.tick()`, then assert `latestResults` has no entry for the subagent session name. Or: assert `heartbeatRunner.setAgentConfigs` was called with `[subagentConfig]` immediately after `sessionManager.startAgent` in the spawner.
**Warning signs:** Production journalctl shows `heartbeat check critical` for subagent session names like `agent-a-via-research-XXX`.

### Pitfall 2: `memoryRetrievalTopK: 0` still triggers an embed + 3 DB queries
**What goes wrong:** Performance overhead on every subagent turn even though no content surfaces.
**Why it happens:** `retrieveMemoryChunks` does not gate on topK=0 upfront (`src/memory/memory-retrieval.ts:158-173`).
**How to avoid:** Acceptable for now ‚Äî overhead is ~5-20ms per turn. If becomes a problem, add `if (topK === 0) return Object.freeze([]);` at line 170. Not in scope for 999.57.
**Warning signs:** Subagent turn latency > parent agent's typical turn by > 50ms.

### Pitfall 3: `memory.conversation.flushIntervalMinutes` undefined causes spread to crash
**What goes wrong:** `sourceConfig.memory.conversation` may be undefined (schema makes it optional, see `types.ts:327`). Spreading `undefined` works in TS but the conditional override needs care.
**Why it happens:** Test factories may build configs without the conversation block.
**How to avoid:** Use the conditional spread shown in Pattern 2: only override conversation if it exists; the planner's factory should null-check.
**Warning signs:** Test failures with "Cannot spread undefined".

### Pitfall 4: `MemoryFlushTimer` writes a final flush on `stopAgent` for the subagent ‚Äî into source's workspace
**What goes wrong:** Even with `memoryFlushIntervalMs` set to a no-op-large value, `stopAgent` invokes `awaitMemoryFileFinalFlush` (`session-manager.ts:1444+`) which flushes pending content. If the subagent accumulated any turn content, that gets written to `{sourceWorkspace}/memory/`.
**Why it happens:** The flush is unconditional on agent stop.
**How to avoid:** Out of scope for the 999.57 narrow fix. Document as Deferred Idea ‚Äî final-flush-on-stop for subagents should write to a transient location OR be suppressed by a separate `memoryFinalFlushOnStop: false` knob.
**Warning signs:** New files in source agent's `memory/` directory timestamped at subagent stop time.

### Pitfall 5: D-05 test (DEL-18) asserts a field that doesn't exist
**What goes wrong:** Test as written in CONTEXT.md `expect(startedConfig.inboxEnabled).toBe(false)` won't even compile (no such field).
**Why it happens:** CONTEXT.md left the field name as TBD per researcher.
**How to avoid:** Reframe DEL-18 as: "spawn subagent, advance heartbeatRunner.tick(), assert no inbox check result for subagent session". Tests the BEHAVIOR (Finding 2's `(iii)`) not a non-existent field.
