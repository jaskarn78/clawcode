Reference pattern — SchedulerSource (src/triggers/scheduler-source.ts):
- Constructor receives `ingest: (event: TriggerEvent) => Promise<void>` callback
- `start()` creates timers, calls `this.ingestFn(event)` on fire
- `stop()` clears timers
- `poll(since)` returns events for replay without calling ingestFn
- Timer handles use `.unref()` to avoid blocking shutdown

    import type { InboxMessage } from "../../collaboration/types.js";

    export type InboxSourceOptions = Readonly<{
      agentName: string;

    Implement `InboxSource` class:
    - `sourceId`: `inbox:{agentName}` (per-agent, unique)
    - Constructor: stores options fields
    - `start()`:
      1. Create chokidar `watch(inboxDir, { ignoreInitial: true, awaitWriteFinish: { stabilityThreshold: stabilityThresholdMs } })`
      2. On `'add'` event handler:
         a. Read file content as UTF-8
         b. JSON.parse — if fails, log warn and return (skip non-JSON)
         c. Validate it has `id`, `from`, `content`, `timestamp` fields (InboxMessage shape). If not, log warn and return.
         d. Build TriggerEvent: `{ sourceId: this.sourceId, idempotencyKey: parsed.id, targetAgent: this.targetAgent, payload: \`[Message from ${parsed.from}]: ${parsed.content}\`, timestamp: parsed.timestamp }`
         e. Call `this.ingestFn(event)` — wrapped in try/catch
         f. If ingest succeeds: call `markProcessed(inboxDir, parsed.id)` to move file to processed/
         g. If ingest fails: log error, do NOT move file (will retry on next heartbeat reconciliation)
    - `stop()`:
      1. If watcher exists, call `watcher.close()`, set to null
    - `poll(since)`:
      1. Call `readMessages(inboxDir)` to get all unprocessed messages
      2. Filter by `msg.timestamp > sinceTs` where `sinceTs = since ? parseInt(since, 10) : 0`
      3. Map to TriggerEvent[] with same shape as the 'add' handler
      4. Return sorted by timestamp ascending

    CRITICAL per Research Pitfall 3: Use `ignoreInitial: true` — without it, chokidar fires 'add' for every existing file on start, causing duplicate processing. Existing files are handled by `poll(since)` on daemon restart via `triggerEngine.replayMissed()`.

    CRITICAL per Research Pitfall 7: InboxSource is the PRIMARY delivery path. The existing heartbeat inbox check becomes a reconciler/fallback. Plan 61-03 will handle the heartbeat coordination in daemon.ts. This task only creates InboxSource — it does not modify the heartbeat check.

    import { Client } from "@modelcontextprotocol/sdk/client/index.js";

    import type { Logger } from "pino";
    import type { TriggerEvent, TriggerSource } from "../types.js";
    import type { TaskStore } from "../../tasks/store.js";

    export type CalendarSourceOptions = Readonly<{
      user: string;
      targetAgent: string;
      calendarId: string;
      pollIntervalMs: number;
      offsetMs: number;
      maxResults: number;
      eventRetentionDays: number;
      mcpServer: {

    Implement `CalendarSource` class:
    - `sourceId`: `calendar:{user}:{calendarId}` (unique per user+calendar)
    - Constructor:
      1. Store all options
      2. Load existing fired event IDs from `taskStore.getTriggerState(sourceId)?.cursor_blob`:
         ```typescript
         // cursor_blob format: JSON array of [eventId, endTimeMs] tuples
         // e.g. [["abc123", 1713400000000], ["def456", 1713500000000]]
         const existing = taskStore.getTriggerState(this.sourceId)?.cursor_blob;
         this.firedIds = existing
           ? new Map<string, number>(JSON.parse(existing) as [string, number][])
           : new Map<string, number>();
         ```
    - `start()`:
      1. Spawn MCP client as long-lived subprocess (per research recommendation):
         ```typescript
         this.transport = new StdioClientTransport({
           command: mcpServer.command,
           args: mcpServer.args ?? [],
           env: mcpServer.env,
         });
         this.mcpClient = new Client({ name: "calendar-source", version: "1.0.0" });
         await this.mcpClient.connect(this.transport);
         ```
      2. Create `setInterval(pollIntervalMs)` calling `pollOnce()`. Call `.unref()`.
      3. Run initial `pollOnce()` immediately after connecting.
      NOTE: Because `start()` needs async for MCP connect, use the pattern `start(): void { void this._startAsync(); }` and store a private `_startAsync` that does the real work. This preserves the TriggerSource interface's sync `start()` signature.
    - `stop()`:
      1. Clear interval
      2. Close MCP transport via `this.transport?.close()` in try/catch (per research pitfall 5: process leak)
      3. Set mcpClient and transport to null
    - `pollOnce()`: (private async)
      1. Call `this.mcpClient.callTool({ name: "calendar_list_events", arguments: { user, calendar_id: calendarId, time_min: new Date().toISOString(), time_max: new Date(Date.now() + offsetMs).toISOString(), max_results: maxResults } })`
      2. Parse response — MCP callTool returns `{ content: [{ type: "text", text: "..." }] }`. Parse the text as JSON to get an array of calendar events.
      3. For each event: extract event ID (e.g., `event.id`) and event end time.
      4. If event ID is already in `firedIds` map: skip (already fired).
      5. If event ID is new: build TriggerEvent `{ sourceId, idempotencyKey: \`cal:${eventId}\`, targetAgent, payload: event, timestamp: Date.now() }`, call `ingestFn(event)`.
      6. Add event ID to `firedIds` map with end time.
      7. Prune stale entries: remove entries where endTime + (eventRetentionDays * 86400000) < now.
      8. Persist fired IDs using consistent tuple-array format:
         ```typescript
         // Serialize as array of [eventId, endTimeMs] tuples for lossless Map round-trip
         taskStore.upsertTriggerState(
           this.sourceId,
           String(Date.now()),
           JSON.stringify([...this.firedIds.entries()]),
         );
         ```
      9. Wrap entire pollOnce in try/catch — on error, log.error and continue (don't crash the daemon).
    - `poll(since)`: (for replay)
      1. If MCP client not connected, return []
      2. Same calendar query as pollOnce but return events as TriggerEvent[] without calling ingestFn
      3. Filter against firedIds set

    **IMPORTANT — cursor_blob format consistency:** Both serialize and deserialize MUST use the same format: an array of `[eventId, endTimeMs]` tuples. Serialize with `JSON.stringify([...firedIds.entries()])`, deserialize with `new Map(JSON.parse(blob) as [string, number][])`. This ensures lossless round-trip of the Map. Do NOT use `Object.fromEntries()` for serialization (loses numeric precision for large timestamps) or `new Set()` for deserialization (wrong data structure — we need the endTimeMs values for retention pruning).

    **IMPORTANT — Push channel renewal DROPPED from scope:** CONTEXT.md mentions "push channel renewal via croner cron job inside CalendarSource" but RESEARCH.md (open question 2) found that the google-workspace MCP server does NOT expose a push channel API (watch/stop). It only has list/create/delete for events. Push channels would require direct googleapis dependency or extending the MCP server. This supersedes the CONTEXT.md decision. Time-window polling with fired-ID dedup is the sole calendar delivery mechanism for Phase 61.

    Per research pitfall 4: Prune event IDs older than eventRetentionDays (default 7) to prevent unbounded growth.

    Per research pitfall 5: `transport.close()` in stop() to prevent MCP server process leak.
