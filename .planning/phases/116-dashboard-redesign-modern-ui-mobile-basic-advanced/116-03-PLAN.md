# Plan 116-03: Tier 1.5 operator workflow

## Goal

Land the three new operator-workflow features the current dashboard lacks entirely. F26 = in-UI agent config editor (no more `vim clawcode.yaml`). F27 = active streaming + FTS5 past search of conversations. F28 = Kanban task board with drag-drop state transitions.

All three features reuse existing daemon-side primitives — no new SQLite tables, no new state machines. The bulk of work is API surface + React components.

## Existing primitives reused

| Primitive | File | What it provides |
|-----------|------|------------------|
| `writeClawcodeYaml()` | `src/migration/yaml-writer.ts` | Atomic temp+rename writes to `clawcode.yaml`; F26 backend |
| `ConfigWatcher.onChange()` | `src/config/watcher.ts` | Debounced file-watch + diff callback; F26 hot-reload |
| `ConversationStore.searchTurnsFts()` | `src/memory/conversation-store.ts` | FTS5 search; F27 past search |
| `ConversationStore.insertTurn()` write hooks | same | F27 active streaming SSE event source |
| `TaskStore.transitionTo()` | `src/tasks/store.ts` | 8-state machine; F28 drag-drop backend |
| `tasks.db` | same | F28 storage |

## Risk notes

- **F26 hot-reload race condition:** if operator changes config mid-turn, the new config applies on NEXT turn (per existing ConfigWatcher behavior). UI should communicate "applies on next turn" in the save success toast.
- **F27 high-cardinality SSE:** broadcasting `conversation-turn` on every turn write could be ~10-50 events/sec at peak. Metadata payload only (`{agent, turnId, ts, role}`) — keeps it well under 1KB/sec total. UI uses TanStack Query batching to avoid re-render per event.
- **F28 drag-drop on mobile:** `@dnd-kit/core` has touch support but operator on iPhone 14 may find Kanban cramped — provide a "list view" mobile fallback that uses dropdown menus to change status instead of drag.
