## Decisions Made

### Drift marker source-keying (advisor-caught during T01)

The plan's Test 2 expects `<!-- drift:virsh:WildcatVM-NEW -->`, NOT `<!-- drift:vm:WildcatVM-NEW -->`. The marker encodes the **source command** (tailscale / virsh / docker), not the entity kind (host / vm / container). Operator can `grep -E '^<!-- drift:(virsh|docker)' DRIFT.md` to scope triage by discovery surface. Caught and fixed before vitest pass — final smoke shows `drift:virsh:WildcatVM-NEW` correctly.

### Counts semantics (advisor-resolved during T01)

Plan Test 1 pins specific counts (`hostCount: 4, vmCount: 4, driftCount: 1`) that didn't match a naive "count raw discovery output" implementation. Resolution:

- `host/vm/containerCount` = **intersection** size between inventory anchors and discovery output (i.e., "how many inventoried items are alive in discovery this tick"). Moltbot-VM in the fixture is `shut off` in virsh — its anchor exists AND virsh sees it, so vmCount=4 includes it. The managed-block `status:` field reflects state (running → ok, shut off → unreachable).
- `driftCount` = host/vm/container discovery items WITHOUT an inventory anchor (excluded: tunnels + op which have no inventory comparison surface in v1).
- `tunnelCount` = raw discovered tunnels (no anchors in v1).
- `dnsCount` = 0 (no v1 poller — deferred).

### DRIFT.md filing scope (option A from advisor)

Tunnels and op items are NOT filed to DRIFT.md in v1. They have no inventory comparison surface (no `<!-- refresh.sh: managed -->` anchors for them in INVENTORY.md), so filing them would re-file the same 3 tunnels + N op items on every run forever — churning the file with zero operator value. They surface via `tunnelCount` in `.refresh-last.json` (operator-grep visible) and via the cloudflared CLI output if the operator wants to enumerate manually. Future plan could anchor tunnels in NETWORK.md's managed block and bring them into the drift/match flow.

### Stale-since timestamp freeze (D-04b refinement, caught during T01 idempotency testing)

A naive `status: unreachable since <ts>` rewrite uses the current tick's `ts`, which differs every run → INVENTORY.md diff every tick → no idempotency. Resolution: when prior `status:` starts with `unreachable since ` AND new computed status would also be `unreachable since <new_ts>`, **preserve the prior status text + prior last_seen** verbatim. This gives the operator a stable first-disappearance timestamp (not last-seen-disappearance) which is the operationally useful signal.

### Plan 01 `.gitignore` is load-bearing for Test 5 (caught during T01 idempotency testing)

Without `.refresh-last.json` in `.gitignore`, the first run's untracked transient telemetry gets picked up by the second run's `git add -A`, producing a diff and a new commit. Plan 01's bootstrap already includes this `.gitignore` on clawdy. The vitest harness explicitly seeds the same `.gitignore` to mirror production — captured in `setupWorkspace()` with a comment pointing at this issue.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] verify.sh's `record_status()` helper leaked false-branch exit code under `set -e`**

- **Found during:** Task 2 vitest spec development (Test 6 unknown-status path)
- **Issue:** When `record_status()` ended with `[[ "$STRICT" == "1" ]] && failed=$(($failed + 1))` and `STRICT=0`, the `[[ ]]` returned 1, the `&&` short-circuited, and the function's final statement returned non-zero — `set -e` in the caller killed the script after the first unknown-status item. Symptom: empty stdout + exit 1 with no error message.
- **Fix:** Replaced `&&` short-circuit with explicit `if` + explicit `return 0` at function end. Symmetric with the same guard already in `homelab_log_struct` / `homelab_fail`.
- **Files modified:** `scripts/homelab/verify.sh`
- **Committed in:** `efcb5a7` (T02 — fix applied before vitest pass)

**2. [Rule 1 — Bug] Drift markers were kind-keyed instead of source-keyed**

- **Found during:** Task 1 advisor consult before vitest spec write-up
- **Issue:** Initial implementation emitted `<!-- drift:vm:WildcatVM-NEW -->` (kind). Plan Test 2 pins `<!-- drift:virsh:WildcatVM-NEW -->` (source command). Would have shipped a green workspace-coding test suite that disagreed with the plan's contract on the very thing operators grep for.
- **Fix:** Rewired `__hl_emit_drift_for_kind` to accept a source key (`tailscale` / `virsh` / `docker`) per call and emit it into the marker.
- **Files modified:** `scripts/homelab/lib/diff-and-quarantine.sh`
- **Committed in:** `7a058e6` (T01)

**3. [Rule 1 — Bug] Counts semantics didn't match Plan Test 1 expectations**

- **Found during:** Task 1 first end-to-end smoke run
- **Issue:** Naive implementation emitted raw discovery counts (`hostCount=4, vmCount=5, driftCount=6`). Plan Test 1 pins `hostCount=4, vmCount=4, driftCount=1`. The intended semantics (intersection of inventory + discovery, plus host/vm/container-only drift, plus tunnel/op excluded) was implicit in the plan but not spelled out.
- **Fix:** `diff-and-quarantine.sh` now emits `matchedHosts / matchedVms / matchedContainers` to `counts.json`; `refresh.sh` reads those instead of raw discovery lengths; tunnels/op excluded from DRIFT.md filing.
- **Files modified:** `scripts/homelab/lib/diff-and-quarantine.sh`, `scripts/homelab/refresh.sh`
- **Committed in:** `7a058e6` (T01)
