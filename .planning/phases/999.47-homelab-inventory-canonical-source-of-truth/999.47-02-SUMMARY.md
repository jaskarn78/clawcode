`src/homelab/refresh-output-schema.ts` exports the Zod v4 schema Plan 03's `scripts/refresh.sh` writes against:

```ts

  schemaVersion: 1,
  ranAt: <iso-8601 UTC>,
  ok: boolean,
```

### Task 2 ŌĆö `homelab-refresh` heartbeat check (commit `164c966`)

`src/heartbeat/checks/homelab-refresh.ts` is a fleet-level CheckModule that runs once per fleet per `refreshIntervalMinutes` window. Sentinel-agent gating: only the alphabetically-first running agent's iteration triggers the actual refresh; every other agent returns `{status: "healthy", message: "sentinel-skip"}` in sub-millisecond time.

**Telemetry log shape (SC-7):**

```ts
logger.info({
  ok: true,
  hostCount, vmCount, containerCount, driftCount,
  commitsha,   // note: no underscore, no camelCase ŌĆö operator-grep stability
}, "phase999.47-homelab-refresh");
```

On `ok: false` paths the same tag fires at `warn` level with `{ok: false, hostCount: 0, ..., commitsha: null, reason}`. When `consecutiveFailures >= 3`, an additional `error`-level line fires:

```ts
logger.error({
  ok: false,
  reason: failureReason,
  consecutiveFailures,
}, "phase999.47-homelab-fleet-alert");
```

This is the D-04c v1 alert deliverable. Discord webhook push deferred per CONTEXT.md.

**After Plan 03 ships refresh.sh and `clawcode reload` picks up the new check:**

```bash
journalctl -u clawcode -g phase999.47-homelab-refresh
# one line per tick
journalctl -u clawcode -g phase999.47-homelab-fleet-alert
# zero lines unless 3+ consecutive failures
```

**Registry update:** `CHECK_REGISTRY` 13 ŌåÆ 14 in `src/heartbeat/check-registry.ts`. Discovery test, runner boot-log test, and the Phase 999.8 drift-detector test (`check-registry.test.ts`) all bumped to 14 with `homelab-refresh` in the arrayContaining list.

### Task 3 ŌĆö `clawcode homelab` CLI (commit `410b39f`)

Two subcommands at `src/cli/commands/homelab.ts`:

```
clawcode homelab reindex [--repo-path <path>] [--strict] [--quiet]
clawcode homelab refresh [--repo-path <path>]
```

`reindex` enumerates the fleet via `readRegistry()` and fires the **existing Phase 49 `ingest-document` IPC** once per agent per INVENTORY.md / NETWORK.md / ACCESS.md (9 calls for a 3-agent fleet). Per-agent failures are logged and skipped (parallel-independence); `--strict` flips to exit-1 on any failure; `--quiet` suppresses the per-agent summary table. The heartbeat tick (Task 2) invokes this fire-and-forget after a successful refresh.

`refresh` is the operator escape hatch ŌĆö shells out directly to `<repoPath>/scripts/refresh.sh`. Exits with the script's exit code.

Both subcommands export pure runner functions (`runHomelabReindex`, `runHomelabRefresh`) so tests don't have to navigate `process.exit()`.

Final log line at completion:

```ts
logger.info({
  totalAgents, totalFiles, succeeded, failed, repoPath,
}, "phase999.47-homelab-reindex");
```

## Deviations from Plan

### [Rule 1 ŌĆö Bug] execa is not in lockfile despite threat-register claim

- **Found during:** Task 2.
- **Issue:** The plan's `<threat_model>` entry T-99947-SC asserts "uses existing `zod` (v4) + `execa` + `pino` + `commander` already in the lockfile." `grep '"execa"' package.json` returned zero hits, and `grep -rn 'from "execa"' src/` returned zero hits ŌĆö the codebase has never depended on execa.
- **Fix:** Used promisified `node:child_process.execFile` (Phase 91 sync-runner pattern; mirrors `src/heartbeat/checks/mcp-reconnect.ts`). Same behavioural contract (timeout-bounded, captures stdout/stderr/exitCode), no new dependency added.
- **Files modified:** `src/heartbeat/checks/homelab-refresh.ts`, `src/cli/commands/homelab.ts`.
- **Commits:** `164c966`, `410b39f`.

### [Rule 2 ŌĆö Missing critical] CheckContext has no `deps` field ŌĆö module-level injection instead

- **Found during:** Task 2.
- **Issue:** The plan's `<action>` block referred to "DI'd through `CheckContext.deps`" but `CheckContext` (in `src/heartbeat/types.ts`) has no `deps` field. Widening that surface to accommodate one check would have touched every existing check's test fixtures.
- **Fix:** Module-level test-injection seams: `__setExecaForTests`, `__setReindexRunnerForTests`, `__resetMutexForTests`, `__resetLastTickForTests`. Production code paths never call the setters; tests restore in `afterEach`. Same pattern applied to Task 3's CLI tests.
- **Files modified:** `src/heartbeat/checks/homelab-refresh.ts`, `src/cli/commands/homelab.ts`.
- **Commits:** `164c966`, `410b39f`.

### [Rule 3 ŌĆö Blocking] `discovery.test.ts` claim ("auto-discovery via src/heartbeat/discovery.ts") was stale

- **Found during:** Task 2 startup.
- **Issue:** The plan asserted "The new module is auto-discovered by `src/heartbeat/discovery.ts` because it lives in `src/heartbeat/checks/`." Phase 999.8 Plan 03 replaced the dynamic-import scan with a static `CHECK_REGISTRY` array ŌĆö checks must now be explicitly registered.
- **Fix:** Added `import homelabRefresh from "./checks/homelab-refresh.js"` + `homelabRefresh,` entry to `CHECK_REGISTRY`. Bumped count assertions in `discovery.test.ts`, `runner.test.ts`, and `check-registry.test.ts` (the Phase 999.8 drift detector).
- **Files modified:** `src/heartbeat/check-registry.ts` + 3 test files.
- **Commit:** `164c966`.

### [Rule 1 ŌĆö Bug] CLI refresh subcommand cannot import the CheckModule's execute()

- **Found during:** Task 3.
- **Issue:** Plan said `clawcode homelab refresh` should "import and invoke the homelab-refresh CheckModule's execute() directly." But execute() requires a `CheckContext` (sessionManager + registry) that only the daemon process holds ŌĆö the CLI runs out-of-process.
- **Fix:** CLI `refresh` shells out to `<repoPath>/scripts/refresh.sh` directly via `node:child_process.execFile`. Same script the heartbeat tick spawns; same exit code semantics. Cleaner separation ŌĆö daemon and CLI converge on the bash script, not on the JS-internal execute() function.
- **Files modified:** `src/cli/commands/homelab.ts`.
- **Commit:** `410b39f`.
