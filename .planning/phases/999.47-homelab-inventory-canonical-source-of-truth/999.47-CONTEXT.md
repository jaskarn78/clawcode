<domain>
## Phase Boundary

Build a **version-controlled, agent-readable inventory** at `/home/clawcode/homelab/` on the clawdy host capturing every host / VM / container / Cloudflare tunnel / Tailscale node / service / DNS entry in the homelab. Format is markdown + git (no UI). Credentials are 1Password `op://` references only — never literal secrets. Three canonical docs (INVENTORY / NETWORK / ACCESS) plus a quarantine doc (DRIFT.md), a retirement doc (RETIRED.md), and `scripts/refresh.sh` + `scripts/verify.sh` to keep Live State sections honest.

**In scope:**
- Three canonical markdown docs with stable-fact and live-state sections: `INVENTORY.md` (hosts/VMs/containers/services), `NETWORK.md` (Tailscale IPs, port maps, DNS, Cloudflare tunnels), `ACCESS.md` (SSH/VNC/RDP CLI snippets + 1Password item refs)
- `scripts/refresh.sh` — polls `tailscale status`, `virsh list` (Unraid), `docker ps` (Unraid), `op item list` for known vaults; regenerates Live State sections; commits when a diff exists
- `scripts/verify.sh` — sanity-check that every entry in INVENTORY is actually reachable (ping, port open, container running)
- `DRIFT.md` — quarantine for items found by refresh.sh that are NOT in INVENTORY.md; operator reviews + promotes to INVENTORY or retires
- `RETIRED.md` — items that have been decommissioned; preserve history, never delete
- Hourly croner-driven `refresh.sh` invocation
- One-line MEMORY.md pointer added to every agent's MEMORY.md: `- [Homelab inventory](/home/clawcode/homelab/INVENTORY.md) — canonical source of truth for hosts, VMs, containers, access`
- RAG-index INVENTORY / NETWORK / ACCESS into each agent's `memory_chunks` table via Phase 49 infra; refresh script triggers re-index after commit
- Bootstrap: hybrid auto-discovery scripts populate skeleton (names/IPs/op item refs/VM/container labels); operator fills semantic gaps (purpose / owner / access notes) via direct markdown edit

**Out of scope (explicit):**
- Building a UI for the inventory — markdown + git IS the UI
- Monitoring or alerting on host health (separate concern; partially handled by existing fleet-alert pattern)
- Cross-machine config sync (this is documentation, not configuration management)
- Pure operator-interview bootstrap (operator-fill-the-gaps after discovery is the chosen path; interactive Discord interview tool is a separate phase if ever needed)
- `homelab_lookup` MCP tool (RAG-index path covers this; new tool would be redundant)
- Auto-promotion of drift items into INVENTORY (operator-intent gate is non-negotiable for the canonical-source-of-truth invariant)
- Storing inventory inside the ClawCode workspace-coding repo or inside an agent workspace (standalone repo decision is locked)

<decisions>
## Implementation Decisions

### Bootstrap strategy

- **D-01:** **Hybrid: discovery scripts populate skeleton + operator fills semantic gaps via direct markdown edit.** Discovery scripts (`tailscale status`, `virsh list --all`, `docker ps -a`, `op item list --vault clawdbot`, etc.) write a draft INVENTORY/NETWORK/ACCESS that contain only machine-readable facts (host names, Tailscale IPs, VM names, container IDs, op-item identifiers). Operator then opens the files and fills in semantic fields ("Purpose:", "Owner:", "Access notes:", "Lifecycle:") via direct markdown edit. No interactive Discord interview tool at v1 — that's a follow-up phase if operator wants it.

### Storage location & repo boundary

- **D-02:** **Standalone git repo at `/home/clawcode/homelab/` on clawdy.** Independent of `~/.openclaw/workspace-coding` ClawCode source. Owned by `clawcode` service user (uid match with the systemd unit). Operator pushes/edits via SSH or a small `clawcode homelab edit <file>` CLI wrapper (decide in planning whether to bother with the wrapper or just SSH). Lives outside the ClawCode repo so refresh-driven cron commits don't bloat ClawCode git history, and the inventory survives ClawCode deploys (`/opt/clawcode/` upgrades don't touch `/home/clawcode/homelab/`).
- **D-02a:** **Refresh.sh commits Live State changes via cron** with author identity `clawcode-refresh <noreply@clawcode>` (separate from operator commits) so the audit trail distinguishes machine-driven vs human-driven edits.
- **D-02b:** **No remote push by default.** Repo is local-only at v1. Operator can `git remote add` later if they want offsite backup. Avoids accidentally publishing internal IPs/op-ref names to a remote.

### Agent access pattern

- **D-03:** **MEMORY.md pointer + RAG-index into `memory_chunks` (Phase 49 infra).** Two surfaces:
  1. **MEMORY.md pointer line** appended to every agent's `~/.clawcode/agents/<agent>/MEMORY.md` (or shared MEMORY surface — confirm in plan-phase): `- [Homelab inventory](/home/clawcode/homelab/INVENTORY.md) — canonical source of truth for hosts, VMs, containers, access`. Agents that auto-load MEMORY.md see it on every turn; they can `Read` the file on demand.
  2. **RAG-indexed via Phase 49 infra** — INVENTORY.md, NETWORK.md, ACCESS.md are ingested into each agent's `memory_chunks` + `vec_memory_chunks` + `memory_chunks_fts` on every refresh.sh commit. Chunks surface in pre-turn retrieval when relevant. No new MCP tool — existing `memory_lookup` / `search_documents` cover query.
- **D-03a:** **Index trigger lives in refresh.sh** — after the commit that follows any diff, refresh.sh fires a re-index path. Exact mechanism (per-agent IPC vs daemon-level rebuild) decided in plan-phase. Acceptance: changes show up in agent retrieval within one refresh cycle (≤ 1h).
- **D-03b:** **No homelab_lookup MCP tool.** Adding a fourth discoverability surface (beyond pointer + RAG + Read tool) would create drift if not all kept in sync. Defer to a future phase if pre-turn retrieval proves insufficient.

### Refresh cadence + drift policy

- **D-04:** **Hourly croner-driven refresh.** Use the existing `croner` dependency (already in the stack per CLAUDE.md). Cron job lives in the ClawCode daemon's existing scheduler (we already manage per-agent crons; the homelab refresh is a fleet-level cron, not per-agent — confirm in plan-phase whether it slots into the existing scheduler or runs as a system-level cron under the clawcode user).
- **D-04a:** **Drift policy = quarantine in DRIFT.md, operator-promote only.** When refresh.sh finds an item present in `tailscale status` / `virsh list` / `docker ps` / `op item list` that is NOT in INVENTORY.md, it appends an entry to `DRIFT.md` with timestamp + raw discovery payload + the source command that surfaced it. INVENTORY.md is NOT auto-modified. Operator reviews DRIFT.md when convenient, promotes entries (with semantic notes) into INVENTORY.md or moves them to RETIRED.md.
- **D-04b:** **Stale-down policy = mark, don't delete.** If an inventoried item is no longer present (VM destroyed, container gone, tunnel removed), refresh.sh updates the Live State section to `status: unreachable since <iso-timestamp>` rather than removing the entry. Operator decides if it goes to RETIRED.md.
- **D-04c:** **Failure mode = noisy commit, never silent.** If refresh.sh fails (network down, op:// auth broken, virsh unreachable), it writes a structured row to `DRIFT.md` under a `## Refresh Failures` section and exits non-zero. No commit on failure (don't pollute git history with empty diffs). Fleet-alert webhook fires if 3 consecutive refresh runs fail.

### Claude's Discretion

- Exact partition of which info goes in INVENTORY vs NETWORK vs ACCESS — planner can refine. Rough cut: INVENTORY = "what exists + what it does"; NETWORK = "addressing + topology"; ACCESS = "how to log in / SSH command snippets + op:// refs". Overlap is fine; cross-link with markdown anchors.
- Refresh.sh implementation language — bash vs Node.js vs TypeScript. Default to bash for the polling scripts (small, no deps, easy to audit); TypeScript for any logic that needs to talk to the ClawCode daemon IPC. Planner picks.
- Order in which categories are surveyed first (Hosts → VMs → Containers → Tunnels → DNS) — operator can re-order in plan-phase if they want a different priority.
- Test-fixture strategy for verify.sh — synthetic Tailscale device list + virsh dump-xml fixtures. Plan-phase picks the test surface.
