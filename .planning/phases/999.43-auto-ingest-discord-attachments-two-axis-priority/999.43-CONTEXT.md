## Phase Boundary

Hook every Discord attachment received by a flagged agent into the **now-live Phase 101 RAG engine**, applying two-axis priority weighting so high-signal client documents (Pon's tax return, brokerage statements) surface in retrieval above low-signal ephemeral uploads (form screenshots, web page captures). Closes the gap between manual `ingest_document` MCP tool (Phase 101) and daily Discord-attachment workflow.

**In scope (from ROADMAP §999.43 sub-scope 1-9):**
1. Discord attachment receipt hook in `src/discord/bridge.ts` — fires `attachmentReceived` after Phase 113 vision pre-pass
2. Heuristic classifier `src/documents/auto-ingest-classifier.ts` — pure function `(filename, mimeType, size, visionAnalysis?) → { eligible, contentClass, reason }`
3. Schema additions — `agents[*].autoIngestAttachments: bool` + `agents[*].ingestionPriority: "low"|"medium"|"high"` in `clawcode.yaml`; `agent_priority_weight` + `content_priority_weight` columns on documents table
4. Score-weighting in `search_documents` — multiplicative `base × agent_weight × content_weight × recency_boost`
5. Reaction emoji override — 🔴/🟡/🟢 on Discord attachment messages maps to `setDocPriority(doc_id, level)`
6. `clawcode_rag_set_priority` MCP tool — agent-callable self-classification (Phase 90 isolation respected)
7. CLI commands — `clawcode rag set-priority <doc-id> <level>` + `clawcode rag reclassify --agent <name> --rule <pattern>`
8. Provenance tagging — every ingested doc tagged `(agent_name, channel_id, message_id, user_id, ingested_at, source: "discord_attachment", auto_classified_class, override_class?)`
9. UAT — fin-acquisition with `autoIngestAttachments: true` + `ingestionPriority: high` auto-ingests a real client PDF; `search_documents` query "Ramy's brokerage balance" surfaces it above incidental form screenshots

**Out of scope (deferred per ROADMAP):**
- Binary/media files (video, audio, archives) — eligibility filter rejects
- Cross-agent document sharing — Phase 90 isolation lock unchanged
- Retention policy / storage caps — defer to follow-on phase
- LLM-based classifier (heuristic cascade only at v1; escalate if miscategorization >20%)
- Embeddings re-computation on priority changes — score-weighting is query-time only

## Implementation Decisions (all locked from operator 2026-05-08 ROADMAP enrichment)

### D-01 — Two-axis priority spec (LOCKED VERBATIM)

**Axis 1 — Per-agent base priority** (operator-set in `clawcode.yaml`):
```yaml
agents:
  - name: fin-acquisition
    autoIngestAttachments: true
    ingestionPriority: high
  - name: research
    autoIngestAttachments: true
    ingestionPriority: medium
  - name: personal
    autoIngestAttachments: true
    ingestionPriority: low
```
Multipliers: `high = 1.5`, `medium = 1.0`, `low = 0.7`.

**Axis 2 — Per-document content classification** (heuristic cascade — table verbatim from ROADMAP §999.43):

| Signal | Class |
|---|---|
| PDF, size > 100KB | HIGH |
| PDF, size < 100KB | LOW (likely receipt/form) |
| .docx / .xlsx / .csv | HIGH |
| .md / .txt | MEDIUM |
| Image, vision-pass detects text-heavy financial-doc content | HIGH |
| Image, vision-pass detects webpage/UI/form | LOW |
| Image, filename contains "Screenshot" | MEDIUM (LOW if also has form-keywords like "form"/"fill"/"checkout") |
| Image, filename contains operator-set client-name list | HIGH |
| Code files (.py / .js / .sql / .ts) | LOW |
| (default fallback) | MEDIUM |

Multipliers: `high = 1.5`, `medium = 1.0`, `low = 0.5`.

### D-02 — Final retrieval score formula (LOCKED VERBATIM)

```
score = base_similarity_or_bm25
      × agent_priority_weight    (1.5 / 1.0 / 0.7)
      × content_priority_weight  (1.5 / 1.0 / 0.5)
      × recency_boost            (1.3× if ingested within last 7 days, else 1.0×)
```

Maximum signal-to-noise spread (high-signal client-doc-from-priority-agent vs low-signal one-off-screenshot-from-low-priority-agent) ≈ **6×** at retrieval time.

### D-03 — Override surface (LOCKED, dual operator + agent control)

- **Operator-only reaction emoji** on attachment message: 🔴 HIGH, 🟡 MEDIUM, 🟢 LOW. Hooks Phase v1.x reaction-handling pipeline; audit-log entry per override.
- **Agent-callable MCP tool** `clawcode_rag_set_priority(doc_id, level)` — sandboxed via Phase 90 isolation (agent cannot escalate own doc beyond MEDIUM; only operator can set HIGH via emoji or CLI).
- **CLI override** `clawcode rag set-priority <doc-id> <high|medium|low>` — retroactive operator adjustment.
- **Bulk reclassify** `clawcode rag reclassify --agent fin-acquisition --rule "Screenshot*=low"` — systemic miscategorization fix.

### D-04 — Provenance tagging (LOCKED)

Every auto-ingested doc carries:
- `agent_name` (which agent triggered the ingest)
- `channel_id` (which Discord channel)
- `message_id` (which Discord message)
- `user_id` (Discord user who uploaded)
- `ingested_at` (ISO timestamp)
- `source: "discord_attachment"` (distinguishes from manual `ingest_document` calls)
- `auto_classified_class: "high"|"medium"|"low"` (heuristic result)
- `override_class?: "high"|"medium"|"low"` (operator/agent override if any)

Discord message deletion does NOT cascade-delete the doc (Phase 99 audit-trail pattern). Operator purges via CLI `clawcode rag delete <doc-id>`.

### D-05 — Phase 113 vision pre-pass IS the classifier feed for images

Image attachments already go through Phase 113 vision pre-pass at receipt time (Haiku 4.5). The classifier piggybacks on the existing analysis result — NO extra LLM call per image. Classifier reads the vision-pass output and applies the table from D-01 Axis 2.

### D-06 — Eligibility filter (locked from ROADMAP "Out of scope")

REJECT before classification:
- Video files (.mp4, .mov, .webm, .avi)
- Audio files (.mp3, .m4a, .wav, .ogg, .flac)
- Archives (.zip, .tar, .gz, .7z, .rar)

Everything else flows through the classifier. Default fallback = MEDIUM.

### D-07 — Recency boost is a query-time computation (LOCKED)

The 1.3× recency boost is applied at retrieval time by `search_documents`, computed from `ingested_at` timestamp vs `now()`. **Not stored as a column.** Priority changes apply immediately without re-embedding.

### D-08 — MCP tool sandbox (LOCKED per ROADMAP)

`clawcode_rag_set_priority` is agent-callable BUT:
- Agents can only adjust docs they themselves ingested (Phase 90 isolation lock)
- Agents CANNOT escalate own doc beyond MEDIUM (operator-only HIGH via emoji or CLI)
- Every set_priority call writes an audit-log entry with `(agent, doc_id, old_level, new_level, reason?, ts)`

### D-09 — Default config — `autoIngestAttachments: false` fleet-wide

Backward-compatible: existing agents without the flag preserve current behavior (manual ingest only). Operator opts in per-agent via `clawcode.yaml` edit + `clawcode reload` (config-watcher hot-reload supported per Phase 22).

## Specific Ideas

- **Discord attachment flow (end-to-end):**
  1. User uploads file to a Discord channel bound to an agent with `autoIngestAttachments: true`
  2. Phase 113 attachment download → temp file
  3. Phase 113 vision pre-pass (images only)
  4. `attachmentReceived` event fires
  5. `auto-ingest-classifier.ts` returns `{eligible: true, contentClass: "high", reason: "PDF size 247KB"}`
  6. Path 999.43 dispatches → daemon's `ingest-document` IPC handler with `{agent_priority_weight, content_priority_weight, provenance}`
  7. Phase 101 engine ingests + writes to `vec_document_chunks` + cross-ingests into `memory_chunks`
  8. `phase999.43-autoingest` structured log line fires
  9. Reaction-emoji override hook listens 60s for operator 🔴/🟡/🟢 on the source message → updates priority columns

- **Test fixtures (synthetic — no real Pon data needed):**
  - `pon-test-tax-return.pdf` (247KB synthetic) → expect HIGH content_class
  - `stripe-receipt.pdf` (12KB synthetic) → expect LOW (size < 100KB)
  - `client-statement.xlsx` → expect HIGH (.xlsx mapping)
  - `screenshot-form-fill.png` with filename "Screenshot 2026-05-16 form fill.png" → expect LOW (Screenshot + "form" keyword)
  - `random-meme.png` → expect MEDIUM (default fallback)

- **Performance:** auto-ingest should NOT block the agent's turn. Fire-and-forget pattern with `.then().catch(err => logger.warn(...))` like Phase 101's daemon-tick reindex CLI invocation.
