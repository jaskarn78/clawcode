## Incidents

### Incident 1 — `autoIngestAttachments` hot-reload didn't propagate

**Symptom:** After editing `clawcode.yaml` to add `autoIngestAttachments: true` for fin-acquisition and running `clawcode reload`, the IPC handler still returned `"autoIngestAttachments disabled for agent"`.

**Root cause:** Plan 01 T03 added `autoIngestAttachments` to `RELOADABLE_FIELDS` per the spec, but the resolver's in-memory snapshot for the running agent did NOT refresh on reload. The journal logged `config hot-reloaded` for fin-acquisition but the runtime resolved-config object the IPC handler reads via `getAgentConfig(agent)` continued to return the pre-reload value (false/undefined).

**Workaround:** Full `systemctl restart clawcode`. After restart, the resolver picked up the yaml and the IPC handler resolved correctly.

**Follow-up bug:** **Phase 999.43-fu candidate** — `RELOADABLE_FIELDS` registration is necessary but not sufficient for hot-reload to actually take effect at the per-agent config-read site. The reload code path needs to actually swap the in-memory ResolvedAgentConfig for the affected agent. Worth a small Quick Task or follow-up phase to harden the reload path for any agent-level boolean/enum field.

**Impact today:** None — operator just runs `systemctl restart` (we did, 3792248 PID). Future operators flipping `autoIngestAttachments` per-agent will hit the same surprise until the bug is fixed.

### Incident 2 — Path-resolver disambiguation (not a bug, just diagnostic noise)

**Symptom:** Initial `sqlite3 .tables` query against `/home/clawcode/.clawcode/agents/finmentum/memory/fin-acquisition/memories.db` showed NO `documents` table, suggesting Plan 01 T02's migration failed.

**Root cause:** `getAgentMemoryDbPath(config.memoryPath)` returns `<memoryPath>/memory/memories.db`. For fin-acquisition with `memoryPath: /home/clawcode/.clawcode/agents/finmentum/memory/fin-acquisition`, the actual DB is at `.../fin-acquisition/memory/memories.db` (double-`memory`). My diagnostic query was on the wrong file — the legacy/orphan `.../fin-acquisition/memories.db`.

**Outcome:** Once I queried the correct double-`memory` path, the `documents` table + all `vec_document_chunks` machinery were present as Plan 01 T02 designed. No bug.

**Note for future operators:** When diagnosing per-agent memory state, the canonical DB path for fin-acquisition is `<memoryPath>/memory/memories.db`, NOT `<memoryPath>/memories.db`. The legacy `.../memories.db` files at the parent level are pre-Phase-75 orphans.
