## Accomplishments

- **T01 (DocumentStore + IPC):** `search()` signature gains optional `agentWeight` (default 1.0 — back-compat). Both `searchAll`/`searchBySource` SQL statements now LEFT JOIN `documents` to surface `content_priority_weight`, `ingested_at`, `source_kind` per chunk. Post-fetch ranking pass applies the D-02 multiplicative formula `similarity × agentWeight × contentWeight × recencyBoost` (recency 1.3× if ingested ≤ 7 days, else 1.0×) and re-sorts DESC before slicing to caller-requested limit. Over-fetch by 3× covers the case where a high-similarity row would be displaced by a lower-similarity / higher-multiplier row. `daemon.ts case "search-documents"` resolves the LIVE `ingestionPriority` from the agent config (hot-reload honored) and threads it into the store; IPC payload gains additive `weighted_score` + `recency_boost_applied` fields. 4 unit tests verify the multiplicative spread, legacy compat (no `agentWeight` arg), recency cliff at 7 days, and pre-999.43 placeholder neutrality.

- **T02 (memory-retrieval):** New exported pure helper `applyDocumentPriorityWeight<T>(candidates, deps)` on `memory-retrieval.ts`. Detects `document:` prefix via `path.startsWith()`, extracts slug, calls injected `getDocumentRow(slug)`, applies the D-02 formula to those candidates only; non-doc candidates pass through with `weightedFused === fusedScore`. `RetrieveArgs.documentPriority?` is the opt-in plumbing — when omitted, the legacy Phase 90 + Phase 101 pipeline runs unchanged (back-compat for tests + bootstrap paths). When present, the helper runs between time-window filtering and the optional reranker; candidates are re-sorted DESC by `weightedFused` before the rerank gate. **`DocumentStore.getDocumentRowBySlug(slug)`** bridges the memory_chunks `path='document:<slug>'` ⇆ `documents.source=<filepath>` asymmetry via `computeDocSlug` equality on a bounded source scan. **`SessionManager.getMemoryRetrieverForAgent` wires `documentPriority` INSIDE the returned closure** — both `docStore` and `ingestionPriority` are re-resolved per turn so `clawcode reload` propagates without re-wiring the retriever. `phase999.43-weight` structured log fires once per call (when applied count > 0). 4 unit tests verify multiplicative weighting on doc candidates, missing-row fallback, agentWeight threading, and telemetry log shape.

## Decisions Made

- **Plan helper signature changed from `{tag, source, rrfScore}` to `{path, fusedScore}`** — `Hydrated` (the actual candidate shape in memory-retrieval.ts at lines 229-238) has `path/heading/body/fusedScore/scoreWeight/source`; it has NO `tag` or `rrfScore` field. The plan's helper signature would not have compiled. Adapted to match the real shape. Output field renamed `weightedRrf` → `weightedFused` for clarity. Documented as Rule 3 deviation in the T02 commit message.
- **Slug→source bridge via `getDocumentRowBySlug` linear scan** — `documents.source` is the full file path (e.g., `/tmp/foo.pdf`), but cross-ingested `memory_chunks.path` is `document:<slug>` where slug is `computeDocSlug(basename(path))`. The mapping is many-to-one (different paths can share a slug after lowercase + collapse) but the cross-ingest path uses the same `computeDocSlug` so `getDocumentRowBySlug` is the deterministic inverse for any chunk written by Plan 02. Alternative considered: add a slug column with index — REJECTED as a schema change. The plan explicitly required no schema migration in Plan 03.
- **`DocumentSearchResult.weightedScore + recencyBoostApplied` typed OPTIONAL** — additive over the existing IPC payload at `src/mcp/server.ts:1220-1234`. Plan 02 SUMMARY's "no IPC breaking changes" posture is preserved; legacy MCP clients ignore unknown fields and see identical `similarity` semantics.
- **Over-fetch by 3× then in-memory re-rank** — sqlite-vec's `MATCH k=?` requires a fixed integer at SQL time, so ORDER BY weighted score cannot happen in SQL. Over-fetch by `limit × 3` (clamped to 60 = MAX_SEARCH_LIMIT × 3) covers the case where a row at SQL-rank 8 has a high enough multiplier to displace SQL-rank 1. Cost: at most 60 row hydrations for a limit=20 query — acceptable.
- **`applyDocumentPriorityWeight` is a PURE function** — no DB, no I/O. Caller (SessionManager) injects the `getDocumentRow` closure that knows how to call `DocumentStore.getDocumentRowBySlug`. Memory-retrieval has zero coupling to DocumentStore; only an interface contract. This keeps the helper testable with synthetic Hydrated arrays + a stub `getDocumentRow` factory (Test 1-3 pattern).
- **`documentPriority` constructed INSIDE the SessionManager closure** — not captured at retriever construction time. Both `docStore = self.getDocumentStore(agentName)` and `livePriority = self.configs.get(agentName)?.ingestionPriority` are re-resolved on every turn. This honors the Plan 01 SUMMARY hot-reload promise: `clawcode reload` that flips an agent's `ingestionPriority` is visible on the NEXT TURN without restarting the daemon.
- **Reranker gate now consumes `prioritized` instead of `windowed`** — Phase 101 Plan 04 reranker LOGIC is unchanged (the reranker still operates on its `topNToRerank` window and returns `finalTopK`). Only the INPUT ORDER shifts when document candidates are present. No double-rerank, no double-weight, no policy change to the reranker's `(query, passage)` scoring.
- **D-01 multipliers LITERAL in source at both call sites** — `daemon.ts case "search-documents"` and `session-manager.ts getMemoryRetrieverForAgent` each declare `{high: 1.5, medium: 1.0, low: 0.7}` inline (not imported from store.ts). Rationale: operator grep on `high: 1.5` / `low: 0.7` should hit the call sites directly (verbatim D-01 LOCKED table), matching the precedent set by `src/manager/auto-ingest-handler.ts` in Plan 02 T01 (decision documented in Plan 02 SUMMARY "Decisions Made").
