<objective>
Distinguish HTTP 404 from malformed-body errors in the plugin install pipeline. Today, plugins listed on ClawHub without a fetchable manifest (e.g. `hivemind`) surface as `**hivemind** manifest is invalid: clawhub plugin manifest: 404 Not Found.` — the "manifest is invalid" wording is misleading because the body is empty, not malformed.

This plan adds:
- `ClawhubManifestNotFoundError` (new error class in `clawhub-client.ts`) thrown on 404
- `manifest-unavailable` outcome variant in `PluginInstallOutcome`
- `mapFetchErrorToOutcome` branch routing the new error to the new variant
- `renderPluginInstallOutcome` Discord copy that says the registry can't serve the manifest, retry later or pick another plugin

Pure UX-resilience work per RESEARCH §Pitfall 5. The fallback URL construction at `daemon.ts:1045-1047` is already correct (live probes confirmed no URL shape resolves; the registry simply hasn't published manifests for some listed plugins).
