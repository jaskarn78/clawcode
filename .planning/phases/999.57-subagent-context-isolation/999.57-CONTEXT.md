# Phase 999.57: subagent-context-isolation - Context

**Gathered:** 2026-05-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Stop subagent sessions from inheriting source-agent lifecycle configs that
surface stale or unrelated content during the subagent's short-lived task
execution. Specifically: override `heartbeat`, `memoryAutoLoad`,
`memoryRetrievalTopK`, `memoryScannerEnabled`, `memoryFlushIntervalMs`, and
inbox watching in the subagent's `ResolvedAgentConfig` so heartbeats don't
fire, memory retrieval doesn't surface cross-task content, and the inbox
chokidar watcher doesn't pick up unrelated pending messages.

**In scope:**
- Override the inheritance-leak fields when building `subagentConfig` in
  `src/discord/subagent-thread-spawner.ts:637-648`.
- Regression tests pinning each override.
- Documentation update so future maintainers don't accidentally remove the
  overrides thinking they're dead code.

**Out of scope (already shipped — commit `07bd5b0`):**
- Auto-relay-on-first-stream-end gate (the immediate-damage fix from earlier
  tonight). See `subagent-thread-spawner.ts:683` — when `delegateTo` is set,
  `autoRelay` defaults to false; subagent must call `subagent_complete`.
- `subagent_complete` instruction injected into the delegated subagent's
  threadContext.

**Out of scope (deferred to a later phase if it becomes a real problem):**
- Adding a per-spawn `inheritMemory: true` flag for tasks that legitimately
  need parent-agent memory context (e.g., "follow up on yesterday's research").
  No production demand observed yet; ship the safer default first.
- A general "subagent-aware gate" inside `HeartbeatRunner` /
  `InboxSource` (gating on sessionName regex). The cleaner contract is to
  override the *config* per-subagent so the runner/watcher don't need to
  know about subagent semantics. Defer unless we discover a path where
  config-override alone is insufficient.

</domain>

<decisions>
## Implementation Decisions

### Inheritance overrides (the actual fix)

- **D-01:** Subagent config explicitly overrides `heartbeat: { enabled: false, ...rest }` at the spawn site. Heartbeat fires nothing in subagent sessions. The existing `HeartbeatRunner` already gates on `config.heartbeat.enabled === false` (runner.ts:271), so no runner-side change needed.
- **D-02:** Subagent config sets `memoryAutoLoad: false`. The subagent's initial prompt is the `task` string + threadContext + thread title — no MEMORY.md block injected. This eliminates the Run 1 vector where memory retrieval surfaced HeyGen-tagged content from the source agent.
- **D-03:** Subagent config sets `memoryScannerEnabled: false` and a no-op `memoryFlushIntervalMs` (use the schema-default 15min — the subagent is short-lived and unlikely to flush). The scanner can't ingest new memories in a transient session; suppressing it avoids spurious `mid-session` MemoryEntries tagged with the subagent's task that would later pollute the source agent's memory store.
- **D-04:** Subagent config sets `memoryRetrievalTopK: 0` — disables hybrid RRF retrieval entirely. The subagent doesn't get cross-task memory context. Pair with `memoryRetrievalTokenBudget: 0` so even if D-04's TopK gate is bypassed somewhere downstream, the token budget caps zero injection.
- **D-05:** Subagent config sets `inboxEnabled: false` (or whatever the equivalent gate field is — researcher to confirm exact name and path). The chokidar inbox watcher doesn't fire on subagent JSONL drops.

### Scope of override — "delegated only" vs "all subagents"

- **D-06:** Apply ALL overrides D-01..D-05 to BOTH `delegateTo`-set spawns AND `delegateTo`-unset (`-sub-`) spawns. Rationale: even non-delegated subagents are short-lived task-scoped sessions; the inheritance leaks affect both. The 999.57 BACKLOG observed both failures with `delegateTo: research`, but the architectural fix is broader. Tests pin both shapes.

### Override structure — single-source-of-truth factory

- **D-07:** Extract a `buildSubagentOverrides(parentConfig, sourceConfig, isDelegated)` factory function in `src/discord/subagent-config-overrides.ts` that returns the override object. Keeps the inheritance-override list single-sourced so future audits don't have to grep across files. The spawner imports it and spreads it last so it wins over `subagentSourceConfig`. Mirrors the pattern at `subagent-thread-spawner.ts:634-648` (the inline overrides for `channels`, `schedules`, `webhook`, etc., plus the 999.57 additions).

### Backwards compatibility

- **D-08:** No yaml-side opt-out for tonight. If we discover an agent that legitimately relies on subagent memory retrieval, we add a per-spawn `inheritMemory: true` flag in a follow-up. Production fleet today has zero evidence of intentional cross-task memory reliance.

### Tests pin the contract

- **D-09:** Tests added to `subagent-thread-spawner.test.ts`:
  - DEL-15: subagent config has `heartbeat.enabled === false` (both delegated and non-delegated)
  - DEL-16: subagent config has `memoryAutoLoad === false` and `memoryRetrievalTopK === 0`
  - DEL-17: subagent config has `memoryScannerEnabled === false`
  - DEL-18: inbox-enabled equivalent gate is false (exact assertion depends on research finding)
  - DEL-19: parent (non-subagent) agent's config stays unchanged after spawn — no leakage upward
- **D-10:** Wave 0 (Plan 04 pattern from 999.54): if `buildSubagentOverrides` is a new file, confirm the export shape compiles before writing tests.

### Live verification (operator-gated)

- **D-11:** After deploy + Ramy-check + restart, repeat the original failing prompt: ask admin-clawdy in Discord to spawn a delegated research subagent on "look up cool features openclaw has". Confirm:
  - The subagent's first turn does NOT surface stale memory content
  - No heartbeat tick fires during the subagent's session (grep journalctl for the subagent's session name + `heartbeat`)
  - Auto-relay does NOT fire on first-stream-end (already fixed in commit 07bd5b0; re-confirm)
  - Subagent produces real research, calls `subagent_complete`, relay fires the actual deliverable

</decisions>

<specifics>
## Specific Ideas

- **Don't introduce a new top-level config field for "subagent inheritance policy".** The current inline-override pattern is the right shape; adding a yaml knob would expose internals operators don't need to think about.
- **Heartbeat config still needs to be a valid `ResolvedAgentConfig.heartbeat` shape** (line 347-373). Use `intervalSeconds: 0`, `checkTimeoutSeconds: 0`, `contextFill: { warningThreshold: 1, criticalThreshold: 1 }` plus `enabled: false`. The runner short-circuits on `enabled: false` before touching the other fields, but the type requires them.
- **Test the negative case explicitly** — a subagent spawned with the source's heartbeat enabled MUST NOT see heartbeat fires. If we don't pin the negative, a future refactor that swaps `enabled: false` for `enabled: undefined` could silently regress.

</specifics>

<deferred>
## Deferred Ideas

- **Per-spawn `inheritMemory: true` opt-in flag** — surface in `spawn_subagent_thread` MCP schema. Future work if a real use case emerges.
- **Subagent-aware gating inside `HeartbeatRunner` / `InboxSource`** — defense-in-depth, not needed for the current failure pattern.
- **Configurable `memoryRetrievalExcludeTags` defaults for subagents** — softer than `topK: 0` (allows some retrieval but filters out session-summary / mid-session tags). Defer unless `topK: 0` proves too aggressive.
- **Inbox watcher cleanup on subagent session stop** — chokidar leak audit. Separate from this phase.
- **Cross-agent inbox identity drift** (Phase 999.36 sub-bug C precedent) — adjacent to this phase but separately scoped.

</deferred>
