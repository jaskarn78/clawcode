# Phase 68.1: Close isTrustedChannel Provenance Wiring Gap - Context

**Gathered:** 2026-04-18
**Status:** Ready for planning
**Mode:** Gap-closure phase (surfaced by v1.9 milestone audit)

<domain>
## Phase Boundary

Close the single cross-phase wiring gap identified by the v1.9 milestone audit: thread `isTrustedChannel` from `DiscordBridge` through `CaptureInput` to `ConversationStore.recordTurn` so the production capture path records turns with the correct trust provenance (currently always `false`).

This is a surgical cross-phase fix that affects three requirements (CONV-01, SEC-01, RETR-02) that individually passed phase-level verification but fail in cross-phase integration. Not a new feature — restoring correct behavior of existing requirements.

</domain>

<decisions>
## Implementation Decisions

### Claude's Discretion
The fix surface is small and well-specified by the integration checker:

1. `src/discord/capture.ts` — add `isTrustedChannel?: boolean` field to `CaptureInput` type
2. `src/discord/capture.ts::captureDiscordExchange` — pass `input.isTrustedChannel` through to both `recordTurn` calls (user turn and assistant turn)
3. `src/discord/bridge.ts` line 633 — at the `captureDiscordExchange` call site, pass `isTrustedChannel: true` (since the message has already passed `checkChannelAccess` at line 441 — only ACL-allowed channels reach this code path)
4. Add a regression test to `src/discord/__tests__/capture.test.ts` that verifies a turn captured with `isTrustedChannel: true` lands in the ConversationStore with `is_trusted_channel = 1`

Key implementation constraints:
- Do NOT weaken the default trust filter in `ConversationStore.searchTurns` (the security model depends on it)
- Do NOT touch the existing Phase 64/65/68 code beyond the wiring changes listed above
- Keep the change surgical — 1-2 tasks max

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `checkChannelAccess` in `src/discord/bridge.ts` line 441 — already computes the boolean channel trust status. Early-returns if `allowed === false`, so by construction only trusted messages reach the capture call site
- `CaptureInput` type in `src/discord/capture.ts` — current shape missing `isTrustedChannel`
- `RecordTurnInput` in `src/memory/conversation-types.ts` — already has `isTrustedChannel?: boolean` (optional field, defaults to false in the store's coercion)
- `ConversationStore.recordTurn` already coerces `isTrustedChannel === true ? 1 : 0` at the SQL boundary

### Established Patterns
- DiscordBridge has a clean separation: `checkChannelAccess` is the ACL gate; everything past line 442 is trusted
- `captureDiscordExchange` uses a try/catch wrapper so failures never block Discord message delivery (pattern from Phase 65 — preserve it)
- Test harness in `src/discord/__tests__/capture.test.ts` already has stubbed ConversationStore fixtures

### Integration Points
- The fix is a one-way thread-through — no reverse coupling
- `ConversationStore.searchTurns` default trust filter (`AND t.is_trusted_channel = 1`) will start returning results as soon as production turns have the correct trust value
- No schema migration needed (the column exists from Phase 64)

</code_context>

<specifics>
## Specific Ideas

- The fix is exactly as described in the milestone audit; planner should not re-investigate
- Passing `isTrustedChannel: true` at the bridge call site is the right choice: simpler than threading the `allowed` variable (which in `checkChannelAccess` already gates the entire path) and clearer in intent
- The test should seed one turn with `isTrustedChannel: true` and one with `isTrustedChannel: false` (or omitted), verify the DB row, then search via FTS5 with default trust filter and confirm only the trusted turn is returned

</specifics>

<deferred>
## Deferred Ideas

- Threading the actual channel-level trust status (e.g., distinguishing DM vs guild vs thread) — out of scope for v1.9; v2 concern if needed
- Exposing a user-level `includeUntrustedChannels` flag in the MCP tool — deferred; current default is correct
- Back-filling old untrusted turns that should have been trusted — not applicable; old turns are irrelevant once the fix lands because new turns will be correctly marked

</deferred>
