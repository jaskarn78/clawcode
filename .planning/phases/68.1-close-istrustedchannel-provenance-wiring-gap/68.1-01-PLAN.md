# Plan 68.1-01: Close isTrustedChannel Provenance Wiring Gap

## Objective

Thread `isTrustedChannel` from `DiscordBridge` through `CaptureInput` to `ConversationStore.recordTurn` so production-captured turns carry the correct trust provenance (currently always stored as `false`, breaking the default FTS5 trust filter).

## Task 1: Wire isTrustedChannel end-to-end

### Action

**Step 1: Extend CaptureInput type in `src/discord/capture.ts`**

Add a new optional field to the `CaptureInput` type:

```typescript
readonly isTrustedChannel?: boolean;
```

**Step 2: Pass the field through both recordTurn calls in `captureDiscordExchange`**

Inside the function body, both `recordTurn` calls (user turn and assistant turn) must include:

```typescript
isTrustedChannel: input.isTrustedChannel,
```

Place it alongside the existing provenance fields (channelId, discordUserId, etc.).

**Step 3: Pass isTrustedChannel: true at the bridge call site**

In `src/discord/bridge.ts`, find the `captureDiscordExchange` call (around line 633 per the milestone audit). Add to the call's input object:

```typescript
isTrustedChannel: true,
```

This is correct because the message has already passed `checkChannelAccess` (line 441); the early-return at line 442 ensures only ACL-allowed messages reach this code path, so all captured turns are by-construction from trusted channels.

In `src/discord/__tests__/capture.test.ts`, add a new test:
