Add 4 `describe(...)` blocks (one per new directive), each with 3 assertions: key-present, enabled-by-default, canonical-phrase-in-text. ### Pattern 1: Directive Entry Shape (frozen record)
**What:** Each entry is `Object.freeze({ enabled: boolean, text: string })` keyed by a kebab-case directive name in the `DEFAULT_SYSTEM_PROMPT_DIRECTIVES` frozen record.
**When to use:** Every new directive added.

### Anti-Patterns to Avoid
- **Don't add a new schema or test file:** the existing pattern is purpose-built. New file = drift risk.
- **Don't introduce a sort step:** `resolveSystemPromptDirectives` (loader.ts:653) already sorts alphabetically by key for prompt-cache hash stability. Just add the entries; sorting happens at resolver-time.
- **Don't normalize/lowercase the canonical phrase:** static-grep tests use `.toContain(...)` against the exact string. Capitalization, punctuation, and apostrophe-style (`'` vs `'`) all matter. Use straight ASCII apostrophes (existing code uses `'`).
- **Don't add a per-agent override fixture test for each new directive:** `REG-OVERRIDE-PARTIAL` (line 223-240) already covers the override merge path generically against `file-sharing`. Adding 4 redundant copies adds zero coverage.

## Common Pitfalls

### Pitfall 1: Membership-Test Drift
**What goes wrong:** A developer adds a new entry to `DEFAULT_SYSTEM_PROMPT_DIRECTIVES` but forgets to update both `REG-DEFAULTS-PRESENT` AND `REG-V25-BACKCOMPAT`. Tests fail with `expected ["a", "b", "c"] to equal ["a", "b", "c", "d"]`.
**Why it happens:** Two tests assert the same membership invariant in different files (`schema-system-prompt-directives.test.ts:47` and `:181`). Easy to update one, miss the other.
**How to avoid:** Update both expectations in the same edit. Run `grep -n "cross-agent-routing" src/config/__tests__/` to surface every place the sorted-array literal appears.
**Warning signs:** First test passes ("REG-DEFAULTS-PRESENT"), second fails ("REG-V25-BACKCOMPAT").

### Pitfall 2: Wording That Doesn't Translate to LLM Behavior Change
**What goes wrong:** Directive text uses vague guidance ("be careful with X") instead of concrete instruction ("ALWAYS run `web_search` BEFORE answering"). LLM ignores it.
**Why it happens:** Natural-language wording feels polite; LLM behavior change requires verb-first imperative + ALL-CAPS contrast for emphasis.
**How to avoid:** Mirror the existing 7 directives' style — they use:
- Capitalized imperatives (ALWAYS, NEVER, DO NOT, MUST)
- Concrete verbs (`upload`, `invoke`, `surface`, `Skip`)
- Specific tool/method names (`memory_lookup`, `clawcode_share_file`, `web_search`)
- Numbered patterns when multi-step ("Pattern: 1. ... 2. ...")
**Warning signs:** Wording starts with "consider", "perhaps", "be mindful". These are anti-patterns for LLM directives.

### Pitfall 3: Conflict with Existing Directives
**What goes wrong:** New directive contradicts a shipped one. Example: DERIV says "creating new files is in-scope work product" while `file-sharing` says "ALWAYS upload via Discord" — could read as contradiction ("upload, but also create files locally").
**Why it happens:** Each directive was authored in isolation; they all land in the same stable prefix.
**How to avoid:** DERIV's text scopes itself: "creating new files... are all in-scope WORK PRODUCT" — work product is artifact-creation. `file-sharing` scopes itself to "files the user wants to access" — distribution. They're complementary: DERIV authorizes creation, file-sharing routes the result. Make sure the wording surfaces this clearly. The CONTEXT.md draft does this correctly.
**Warning signs:** Two directives use the same trigger phrase ("when you produce a file"). Reword one.

### Pitfall 4: Canonical Phrase Not Unique
**What goes wrong:** The canonical phrase pinned for a new directive accidentally appears in a neighbor's text. Static-grep `.toContain(...)` returns true for the wrong reason — the test passes even if the new directive is missing.
**Why it happens:** Common words ("file", "tool", "use") appear everywhere.
**How to avoid:** Pick a phrase containing 3-5 words unique to this directive's purpose. Verify uniqueness:
```bash
grep -c "are all in-scope work product" src/config/schema.ts  # MUST return 1
```
The 4 chosen canonical phrases (per CONTEXT.md) are all unique enough — verified against the 7 existing entries. None overlap.
**Warning signs:** A canonical phrase appears in multiple `text` fields when checked with `grep -c`.

### Pitfall 5: Token Bloat Across Directives
**What goes wrong:** Each new directive seems "small enough"; collectively they bloat the stable prefix beyond the cache-stability budget.
**Why it happens:** The 7 existing directives already total ~1,800 tokens (long-output-to-file alone is ~280 tokens). Adding 4 more without discipline adds another 400-500 tokens.
**How to avoid:** Hold each new directive to ~100 tokens (~400 chars). The CONTEXT.md draft estimates ~80-110 tokens each — within budget. See §Token Budget below for the actual measurement.
**Warning signs:** Any single directive's `text` exceeds 600 chars in v1. Trim.

### Pitfall 6: FRESH Over-Triggering
**What goes wrong:** Wording like "for any factual lookup, run `web_search`" causes the agent to web-search EVERY query, including ones it can answer from training (e.g., "what's the capital of France").
**Why it happens:** The directive lands in stable prefix and primes every turn. Vague triggers get over-applied.
**How to avoid:** Anchor on time-sensitive CATEGORIES, not on "factual lookup":
- prices / quotes
- tax rules / regulations / laws
- financial filings / current events / recent news
- "anything dated within ~6 months of today"
The CONTEXT.md draft (D-FR-03) already has the right anchoring.
**Warning signs:** Agent emits gratuitous `web_search` calls for queries with stable training-data answers.

### Pitfall 7: TRUST Drift to Authorization Override
**What goes wrong:** "Skip all CYA language" reads to the LLM as "skip all caution"; agent now executes destructive shell commands without confirmation.
**Why it happens:** "CYA" is itself jargon; an LLM that doesn't fully grok the term may generalize.
**How to avoid:** Anchor TRUST text concretely on PHRASING patterns, not on AUTHORIZATION:
- The CONTEXT.md draft lists the exact disclaimers to drop ("this is not malware", "this is for legitimate purposes", etc.)
- The directive scopes the override to *prefacing/disclaimer language*, not to *authorization checks*
- Existing safety rails (security.md no-hardcoded-secrets, validate-inputs, parameterized-queries) remain
**Warning signs:** Post-deploy: agent skips a destructive-action confirmation that it would have asked for pre-deploy. (Validate via post-deploy smoke test — see §Validation Architecture.)

### Pitfall 8: TABLE Over-Avoiding Tables
**What goes wrong:** Agent stops using tables even when 5+ columns × 5+ rows of comparable values genuinely IS the right format. Output becomes a giant bulleted mess.
**Why it happens:** "Prefer bullets" reads as "always bullets".
**How to avoid:** Directive text MUST acknowledge the legitimate case: "Use markdown tables only when the data is genuinely tabular and dense (4+ columns × 4+ rows of comparable values); the webhook-wrap fence renders those as monospace code blocks as a safety net." CONTEXT.md draft has this line correctly.
**Warning signs:** Operator sees nested-bullet attempts to render what should be a 6×6 table.

1. **Does the FRESH directive's "anything dated within ~6 months of today" phrasing trigger the LLM to compute relative dates at runtime?**
   - What we know: The agent has Bash tool access (per D-FR-02 it can run `date`). LLMs can usually reason about relative time with a known anchor.
   - What's unclear: Without an injected today-date, the LLM may guess "6 months ago" relative to its training cutoff, which is wrong.
   - Recommendation: Accept the ambiguity for v1. Operator-observe agent behavior post-deploy. If agents drift on dates, revisit dynamic date injection (deferred idea in CONTEXT.md). Mitigant: the directive's last sentence tells the agent to run `date` if it needs to know today.

   - What's unclear: This reorders the rendered block — first deploy = one cache miss per agent (acceptable per Phase 94 D-04 cache-stability trade-off).
