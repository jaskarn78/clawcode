# v1.9 Persistent Conversation Memory — Milestone Audit

## Executive Summary

All 5 phases (64-68) passed their individual VERIFICATION.md goal-backward checks. Plan-level tests are green: 1019/1023 cross-phase tests pass (the 4 failures are pre-existing `session-memory-warmup` timing flakes under parallel load, unrelated to v1.9).

However, the `gsd-integration-checker` identified **one cross-phase wiring gap** that affects three requirements (CONV-01, SEC-01, RETR-02) in production even though individual-phase tests did not catch it: **`isTrustedChannel` provenance is never threaded from `DiscordBridge` through `CaptureInput` to `ConversationStore.recordTurn`**, so every captured conversation turn is stored with `is_trusted_channel = 0`. This breaks the default FTS5 search path in Phase 68 (which filters `AND t.is_trusted_channel = 1`) because no turn ever has that value set to 1 via the production capture path.

The fix is a ~3-line change in `src/discord/capture.ts` + `src/discord/bridge.ts` (add `isTrustedChannel` to `CaptureInput`, set to `true` at the bridge call site since only ACL-allowed messages reach capture).

## Cross-Phase Integration Findings

**Completed flows (2 of 3):**
- Flow 2: Agent Resume → ConversationBrief Injection (SESS-02, SESS-03) — fully wired end-to-end
- Flow 3: On-demand memory_lookup → searchByScope (RETR-01, RETR-03) — fully wired end-to-end
- Orphaned exports: 0 (all 14 phase exports are consumed downstream)
- `calculateRelevanceScore` reused across phases (not duplicated) as required by CONTEXT.md

**Partial flow (1 of 3):**
- Flow 1: Capture → Store → Summarize — captures happen correctly, turns are stored, summaries are generated. The gap is in the `isTrustedChannel` provenance field.

**Specific integration gap:**
- **Location:** `src/discord/bridge.ts` line 633 — `captureDiscordExchange` call does not pass `isTrustedChannel`
- **Type gap:** `src/discord/capture.ts` — `CaptureInput` does not have an `isTrustedChannel` field
- **Downstream effect:** `RecordTurnInput.isTrustedChannel` defaults to `undefined` → coerced to `0` in `ConversationStore.recordTurn`
- **Production impact:** Every captured turn has `is_trusted_channel = 0`. `ConversationStore.searchTurns` default trust filter (`AND t.is_trusted_channel = 1`) returns zero rows in production, even though unit tests that set the field directly work correctly
- **Affected REQ-IDs:** CONV-01 (provenance incomplete), SEC-01 (trust field incorrect), RETR-02 (default FTS5 search returns empty in prod)
- **Fix:** Add `isTrustedChannel?: boolean` to `CaptureInput`; at the bridge call site (after `checkChannelAccess` succeeds), pass `isTrustedChannel: true` since only allowed messages reach capture. Alternative: change the default trust-filter policy in `searchTurns` — but that weakens the security model. The wire-up fix is correct.
